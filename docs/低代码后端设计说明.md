# 官网内容低代码后端设计说明

本文档用于定义官网内容低代码能力的后端实现方案，重点描述模块边界、分层职责、领域对象、数据库设计建议、服务链路、事务与缓存策略、安全与审计要求，以及分阶段实施方式。本文档是 [低代码重构方案](低代码重构方案.md) 的后端展开稿，不替代 [架构设计说明](架构设计说明.md)、[数据库设计说明](数据库设计说明.md) 和 [接口文档](接口文档.md)。

后端实现必须同时遵守 [后端开发规则](../rules/backend.md)、[安全开发规则](../rules/security.md)、[测试规则](../rules/testing.md) 和 [代码 Review 规则](../rules/review.md)。

---

## 一、文档定位

### 1. 目标

本说明聚焦以下问题：

* 官网展示层低代码能力在后端如何落地。
* 新增模块如何与当前 `site`、`product`、`casecenter`、`media`、`system` 等模块协作。
* 页面草稿、预览、发布、回滚、缓存失效和审计应如何实现。
* 页面 Schema、组件模板、数据绑定和发布快照的后端边界应如何定义。

### 2. 适用范围

本说明适用于以下后端能力：

* 页面定义管理。
* 页面草稿和版本管理。
* 组件模板元数据管理。
* 页面发布快照生成。
* 页面渲染数据装配。
* 页面级缓存和关联实体变更后的缓存联动。

### 3. 非目标

本说明当前不覆盖以下内容：

* 前端拖拽交互细节和编辑器具体实现。
* 完整的多租户、工作流审批、细粒度 RBAC 平台能力。
* 任意自定义表达式、任意 SQL 数据源和用户自定义脚本执行。

---

## 二、设计目标与约束

### 1. 设计目标

* 让官网展示页面从固定接口组合，演进到页面 Schema 驱动。
* 让后台可以管理页面布局、组件属性和公开数据绑定。
* 让线上页面只读取已发布快照，避免草稿污染正式流量。
* 复用现有认证、审计、媒体、缓存、事务和模块分层基线。
* 保持对展示型页面高可配置，对核心业务实体保持强建模。

### 2. 关键约束

* 当前项目仍为单体 Spring Boot 应用，不拆微服务。
* 后台权限模型仍默认只有 `Administrator`。
* Portal 缓存仍以 Redis 或等价共享缓存为生产默认实现。
* 页面低代码不应削弱现有可见性、逻辑删除、媒体引用、审计和缓存一致性规则。
* 页面系统不直接承载产品、案例、线索等核心业务实体建模。

### 3. 核心原则

* 页面结构低代码化，业务实体模块化。
* 编辑态与发布态强隔离。
* 模块间只通过 Service 或 Application Service 交互。
* 所有页面公开输出必须具备稳定结构和缓存策略。
* 高风险动作必须有审计和事务提交后副作用控制。

## 三、后端总体架构

### 1. 新增模块

建议在现有工程中新增 `modules/pagebuilder` 模块，并新增对应的应用层服务：

```text
src/main/java/com/company/officialwebsite
├── application
│   ├── admin
│   │   └── PageBuilderApplicationService.java
│   └── portal
│       └── PageRenderApplicationService.java
├── modules
│   ├── pagebuilder
│   │   ├── controller
│   │   ├── service
│   │   ├── mapper
│   │   ├── entity
│   │   ├── dto
│   │   ├── vo
│   │   ├── converter
│   │   ├── enums
│   │   ├── support
│   │   └── model
│   ├── site
│   ├── product
│   ├── casecenter
│   ├── media
│   ├── lead
│   └── system
└── infrastructure
    ├── cache
    ├── storage
    ├── security
    └── audit
```

### 2. 模块协作关系

* `pagebuilder`
  负责页面生命周期、组件模板、Schema 校验、版本管理和发布快照。
* `site`
  继续负责站点配置、导航、首页静态内容等原子内容实体。
* `product`
  继续负责产品、行业方案等实体和公开查询。
* `casecenter`
  继续负责案例实体和公开查询。
* `media`
  继续负责媒体上传、引用维护和媒体校验。
* `system`
  继续负责管理员身份、审计日志和系统级安全能力。

### 3. 核心链路

低代码后端需要支撑五条核心链路：

* 页面定义链路。
* 页面草稿编辑链路。
* 页面预览链路。
* 页面发布与回滚链路。
* Portal 页面读取与缓存链路。

## 四、分层职责设计

### 1. Controller 层

职责：

* 接收 Admin 和 Portal 请求。
* 执行 DTO 注解校验。
* 调用 Application Service 或 Module Service。
* 返回统一响应。

约束：

* 不直接访问 Mapper。
* 不直接拼装跨模块页面数据。
* 不直接操作 Redis、审计和媒体引用。

建议控制器：

* `AdminPageDefinitionController`
* `AdminPageDraftController`
* `AdminPagePublishController`
* `AdminComponentTemplateController`
* `PortalPageController`

### 2. Application Service 层

#### `application.admin.PageBuilderApplicationService`

职责：

* 编排页面定义、草稿保存、预览生成、发布和回滚。
* 组织跨模块公开数据源元信息。
* 统一处理页面搭建器后台返回模型。

#### `application.portal.PageRenderApplicationService`

职责：

* 读取发布快照。
* 解析页面 Schema。
* 按绑定规则装配公开数据。
* 生成 Portal 稳定页面响应。

### 3. Module Service 层

建议服务拆分：

* `PageDefinitionService`
* `PageDraftService`
* `PageVersionService`
* `PagePublishService`
* `PageSnapshotService`
* `ComponentTemplateService`
* `PageSchemaValidationService`
* `PageBindingResolutionService`
* `PageDependencyService`

职责边界：

* `PageDefinitionService`
  管理页面元信息和页面身份约束。
* `PageDraftService`
  管理当前草稿读写。
* `PageVersionService`
  管理历史版本和版本编号。
* `PagePublishService`
  管理发布、回滚、事务和 afterCommit 副作用。
* `PageSnapshotService`
  负责快照的持久化和读取。
* `ComponentTemplateService`
  管理组件模板元数据。
* `PageSchemaValidationService`
  负责组件、字段、链接、媒体和绑定配置校验。
* `PageBindingResolutionService`
  负责按绑定源获取公开数据。
* `PageDependencyService`
  负责维护页面与业务实体之间的依赖关系，用于缓存联动。

### 4. Mapper 层

Mapper 只负责数据访问，不参与业务决策。

建议 Mapper：

* `PageDefinitionMapper`
* `PageDraftMapper`
* `PageVersionMapper`
* `PagePublishSnapshotMapper`
* `ComponentTemplateMapper`
* `PageDependencyMapper`

复杂查询如果超出 MyBatis Plus 基础能力，可用 XML，但必须显式列字段，禁止 `select *`。

## 五、领域模型设计

### 1. 页面定义 `PageDefinition`

用途：

* 描述页面身份、路由和页面类型。
* 作为页面草稿、版本、快照和依赖关系的上游主实体。

建议字段：

* `id`
* `pageKey`
* `name`
* `routePath`
* `pageType`
* `status`
* `visible`
* `sortOrder`
* `version`
* `deletedMarker`
* `createdAt`
* `updatedAt`
* `createdBy`
* `updatedBy`

建议状态：

* `ENABLED`
* `DISABLED`

页面定义级别的 `visible` 代表页面是否可在 Portal 暴露入口，不替代页面内部组件显隐。

### 2. 页面草稿 `PageDraft`

用途：

* 保存页面最新可编辑内容。
* 与历史版本解耦，避免每次编辑都产生一个完整版本。

建议字段：

* `id`
* `pageId`
* `schemaJson`
* `schemaHash`
* `editorSessionRemark`
* `version`
* `updatedAt`
* `updatedBy`

设计理由：

* 对于低代码场景，草稿是高频写对象。
* 单独设计草稿表可以减少历史版本膨胀和发布链路耦合。

### 3. 页面版本 `PageVersion`

用途：

* 保存可追踪的历史版本。
* 作为发布和回滚的审计锚点。

建议字段：

* `id`
* `pageId`
* `versionNo`
* `sourceType`
* `schemaJson`
* `schemaHash`
* `changeSummary`
* `createdAt`
* `createdBy`

建议 `sourceType`：

* `MANUAL_SAVE`
* `PUBLISH_BASE`
* `ROLLBACK_BASE`

### 4. 发布快照 `PagePublishSnapshot`

用途：

* 保存 Portal 读取的正式页面结果。
* 与草稿、历史版本隔离。

建议字段：

* `id`
* `pageId`
* `versionId`
* `snapshotJson`
* `snapshotHash`
* `publishStatus`
* `publishedAt`
* `publishedBy`

建议 `publishStatus`：

* `ACTIVE`
* `SUPERSEDED`

### 5. 组件模板 `ComponentTemplate`

用途：

* 描述系统允许的组件类型。
* 控制字段协议、默认值、绑定能力和编辑限制。

建议字段：

* `id`
* `componentCode`
* `name`
* `category`
* `schemaDefinitionJson`
* `defaultPropsJson`
* `bindingCapabilityJson`
* `status`
* `sortOrder`

建议状态：

* `ACTIVE`
* `INACTIVE`

### 6. 页面依赖 `PageDependency`

用途：

* 描述页面与具体业务实体之间的引用关系。
* 为实体变更触发页面缓存失效提供依据。

建议字段：

* `id`
* `pageId`
* `snapshotId`
* `componentInstanceId`
* `dependencyType`
* `targetModule`
* `targetEntityType`
* `targetEntityId`
* `createdAt`

适用场景：

* 页面绑定了具体产品 ID。
* 页面绑定了具体案例 ID。
* 页面引用了特定媒体 ID。

## 六、数据库设计建议

### 1. 新增表清单

建议新增以下表：

* `cms_page_definition`
* `cms_page_draft`
* `cms_page_version`
* `cms_page_publish_snapshot`
* `cms_component_template`
* `cms_page_dependency`

### 2. 表设计原则

* 命名遵循现有 `cms_` 前缀规则。
* 逻辑删除按项目统一使用 `deleted_marker`。
* 高并发更新对象应具备 `version` 字段。
* JSON 字段仅承载页面结构、组件配置和白名单绑定配置，不承载强业务主数据。

### 3. `cms_page_definition`

核心字段建议：

* `id BIGINT`
* `page_key VARCHAR(64)`
* `name VARCHAR(128)`
* `route_path VARCHAR(255)`
* `page_type VARCHAR(32)`
* `status VARCHAR(32)`
* `visible TINYINT`
* `sort_order INT`
* `version INT`
* `deleted_marker BIGINT`
* `created_at DATETIME`
* `updated_at DATETIME`
* `created_by BIGINT`
* `updated_by BIGINT`

索引建议：

* 唯一索引：`(page_key, deleted_marker)`
* 唯一索引：`(route_path, deleted_marker)`
* 普通索引：`(status, deleted_marker)`

### 4. `cms_page_draft`

核心字段建议：

* `id BIGINT`
* `page_id BIGINT`
* `schema_json LONGTEXT`
* `schema_hash VARCHAR(64)`
* `editor_session_remark VARCHAR(255)`
* `version INT`
* `updated_at DATETIME`
* `updated_by BIGINT`

索引建议：

* 唯一索引：`page_id`

### 5. `cms_page_version`

核心字段建议：

* `id BIGINT`
* `page_id BIGINT`
* `version_no INT`
* `source_type VARCHAR(32)`
* `schema_json LONGTEXT`
* `schema_hash VARCHAR(64)`
* `change_summary VARCHAR(255)`
* `created_at DATETIME`
* `created_by BIGINT`

索引建议：

* 唯一索引：`(page_id, version_no)`
* 普通索引：`(page_id, created_at)`

### 6. `cms_page_publish_snapshot`

核心字段建议：

* `id BIGINT`
* `page_id BIGINT`
* `version_id BIGINT`
* `snapshot_json LONGTEXT`
* `snapshot_hash VARCHAR(64)`
* `publish_status VARCHAR(32)`
* `published_at DATETIME`
* `published_by BIGINT`

索引建议：

* 普通索引：`(page_id, publish_status, published_at)`

约束建议：

* 同一页面同一时刻只能有一个 `ACTIVE` 快照。

### 7. `cms_component_template`

核心字段建议：

* `id BIGINT`
* `component_code VARCHAR(64)`
* `name VARCHAR(128)`
* `category VARCHAR(64)`
* `schema_definition_json LONGTEXT`
* `default_props_json LONGTEXT`
* `binding_capability_json LONGTEXT`
* `status VARCHAR(32)`
* `sort_order INT`
* `created_at DATETIME`
* `updated_at DATETIME`

索引建议：

* 唯一索引：`component_code`
* 普通索引：`(status, sort_order)`

### 8. `cms_page_dependency`

核心字段建议：

* `id BIGINT`
* `page_id BIGINT`
* `snapshot_id BIGINT`
* `component_instance_id VARCHAR(64)`
* `dependency_type VARCHAR(32)`
* `target_module VARCHAR(64)`
* `target_entity_type VARCHAR(64)`
* `target_entity_id BIGINT`
* `created_at DATETIME`

索引建议：

* 普通索引：`(page_id, snapshot_id)`
* 普通索引：`(target_module, target_entity_type, target_entity_id)`

### 9. JSON 字段边界

建议存入 JSON 的内容：

* 页面布局结构。
* 组件属性。
* 组件样式令牌。
* 绑定源配置。

不建议只存入 JSON 的内容：

* 产品、案例、行业方案等实体主数据。
* 权限配置。
* 媒体元数据。
* 审计信息。

## 八、Schema 后端模型设计

### 1. 推荐内部模型

后端不建议全程用 `Map<String, Object>` 处理 Schema。

建议定义内部模型：

* `PageSchemaModel`
* `LayoutModel`
* `SectionModel`
* `ComponentNodeModel`
* `BindingModel`
* `StyleTokenModel`
* `SeoModel`

### 2. 目的

* 便于服务层做结构化校验。
* 便于版本差异分析。
* 便于生成依赖关系和发布快照。
* 降低 Controller 和 Service 中遍历原始 JSON 的复杂度。

### 3. 校验策略

Schema 校验应分两层：

* DTO 层基础校验。
* Service 层业务校验。

Service 层至少校验：

* 页面包含的组件数量是否超限。
* 组件类型是否注册且启用。
* 组件字段是否符合模板定义。
* 媒体 ID 是否存在且可用。
* 链接协议是否合法。
* 绑定源是否为白名单。
* 绑定参数是否超出允许范围。
* 富文本是否经过白名单清洗。

## 九、组件模板后端设计

### 1. 模板职责

组件模板是后端物料中心的配置协议，而不是前端组件代码本体。后端只管理：

* 模板元信息。
* 字段协议。
* 默认属性。
* 绑定能力。
* 上下线状态。

### 2. 模板字段协议建议

每个字段建议包含：

* `fieldKey`
* `label`
* `fieldType`
* `required`
* `maxLength`
* `defaultValue`
* `enumOptions`
* `supportBinding`
* `helpText`

### 3. 模板变更约束

模板元信息变更存在兼容性风险，应遵循：

* 已上线组件字段不应随意删除。
* 字段重命名必须提供兼容迁移策略。
* 下线组件前必须识别受影响页面。

### 4. 是否允许后台编辑模板

当前阶段建议：

* 模板注册和字段协议优先由代码或初始化脚本维护。
* 后台只开放模板查询和有限状态管理。

原因：

* 可以降低物料协议被误操作破坏的风险。
* 更适合当前项目的稳定性要求。

## 十、数据绑定与依赖解析设计

### 1. 绑定类型

后端建议只支持三类绑定：

* `STATIC`
  纯静态配置内容。
* `ENTITY`
  绑定单个实体或实体列表。
* `AGGREGATE`
  绑定由应用层封装的聚合数据源。

### 2. 绑定源白名单

绑定源必须走明确白名单，不允许前端提交任意表名或任意模块名。

建议首批绑定源：

* `site_config`
* `navigation_menu`
* `home_metric_card`
* `product`
* `industry_solution`
* `case`
* `timeline_event`
* `value_card`
* `promise_tag`
* `contact_info`

### 3. 绑定解析方式

解析职责放在 `PageBindingResolutionService`，由它根据 `bindingType + bindingSource + bindingConfig` 调用对应模块只读查询能力。

不推荐方案：

* 在 Controller 中直接按字符串分支查询。
* 在 Schema 中暴露 SQL。
* 通过反射任意调用模块方法。

### 4. 依赖关系抽取

发布时应从已校验 Schema 中抽取依赖：

* 组件引用的媒体 ID。
* 组件绑定的具体产品 ID、案例 ID 等。
* 组件绑定的页面 key 或导航目标。

依赖抽取结果写入 `cms_page_dependency`，供后续缓存联动使用。

## 十一、Admin 侧接口与用例设计

### 1. 页面定义管理

建议接口：

* `GET /admin/api/page-builder/pages`
* `POST /admin/api/page-builder/pages`
* `PUT /admin/api/page-builder/pages/{pageId}`
* `GET /admin/api/page-builder/pages/{pageId}`

后端职责：

* 校验 `page_key`、`route_path` 唯一性。
* 校验页面类型是否合法。
* 校验逻辑删除和禁用状态。

### 2. 页面草稿管理

建议接口：

* `GET /admin/api/page-builder/pages/{pageId}/draft`
* `PUT /admin/api/page-builder/pages/{pageId}/draft`

后端职责：

* 反序列化 Schema。
* 执行组件协议校验。
* 清洗富文本。
* 校验媒体引用。
* 按乐观锁更新草稿。

### 3. 预览

建议接口：

* `POST /admin/api/page-builder/pages/{pageId}/preview`
* `GET /admin/api/page-builder/previews/{previewToken}`

建议实现：

* 生成短期有效预览 token。
* 预览内容可从临时缓存或临时表中读取。
* 预览不写正式发布快照。

### 4. 发布与回滚

建议接口：

* `POST /admin/api/page-builder/pages/{pageId}/publish`
* `POST /admin/api/page-builder/pages/{pageId}/rollback`
* `GET /admin/api/page-builder/pages/{pageId}/versions`
* `GET /admin/api/page-builder/pages/{pageId}/snapshots`

后端职责：

* 发布前做完整校验。
* 生成历史版本记录。
* 生成新快照。
* 更新旧快照状态。
* 写入依赖关系。
* 注册 afterCommit 缓存失效。
* 写入审计日志。

---

## 十二、Portal 侧接口与读取设计

### 1. 页面读取接口

建议接口：

* `GET /portal/api/pages/{pageKey}`

响应内容建议包含：

* 页面基础信息。
* 已发布 Schema。
* 绑定后的公开数据。
* SEO 信息。

### 2. 读取流程

建议流程：

```text
PortalPageController
  -> PageRenderApplicationService
  -> read page cache
  -> cache miss -> read active snapshot
  -> resolve bindings
  -> filter invisible or deleted entities
  -> build portal page VO
  -> write cache with TTL
  -> return
```

### 3. 过滤要求

Portal 渲染时必须确保：

* 只读取 `ACTIVE` 发布快照。
* 页面定义本身需可见且未删除。
* 绑定出来的实体必须满足 `visible = 1 AND deleted_marker = 0`。
* 已下线组件模板不影响已发布快照读取，但新编辑应禁止再使用。

---

## 十三、事务设计

### 1. 事务边界

以下操作必须在 Service 层声明事务：

* 创建页面定义。
* 保存页面草稿。
* 发布页面。
* 回滚页面。
* 下线或删除页面定义。

### 2. 发布事务内容

发布事务内至少完成：

* 读取并锁定页面定义。
* 读取当前草稿。
* 校验草稿与版本约束。
* 生成历史版本。
* 生成发布快照。
* 更新旧快照状态。
* 重建页面依赖。
* 写审计日志。
* 注册 afterCommit 缓存失效。

### 3. 事务外操作

以下操作不应阻塞数据库事务太久：

* 延迟二次删除调度。
* 非核心统计上报。
* 预览缓存清理。

这些动作应在事务提交后执行。

---

## 十四、缓存设计

### 1. 缓存对象

建议缓存以下对象：

* 页面正式输出：`official:portal:page:{pageKey}`
* 页面元信息：`official:portal:page-meta:{pageKey}`
* 预览输出：`official:admin:page-preview:{previewToken}`

### 2. TTL 建议

建议：

* 正式页面缓存使用中等 TTL。
* 预览缓存使用较短 TTL。
* 不允许永久缓存。

具体 TTL 数值在正式实施时进入配置文件，不写死在业务代码中。

### 3. 失效策略

页面发布、回滚、禁用、删除后必须：

* 事务提交后立即删除对应页面缓存。
* 调度延迟二次删除。

关联实体发生变更时也应清理受影响页面缓存，例如：

* 产品隐藏。
* 案例逻辑删除。
* 联系方式更新。

### 4. 关联实体变更的缓存联动

联动方式建议：

* 业务模块写操作成功后，通过应用层或事件机制调用页面依赖查询。
* 查询 `cms_page_dependency` 找到受影响页面。
* afterCommit 批量删除相关页面缓存。

这样可以避免页面模块直接侵入各业务模块 Mapper。

---

## 十五、安全与审计设计

### 1. 权限控制

* 所有后台页面搭建写接口必须要求管理员登录。
* 页面发布、回滚、删除、模板上下线属于高风险动作，必须仅限 `Administrator`。
* Portal 页面读取接口允许匿名，但只能读取发布态内容。

### 2. 输入安全

需要重点防范：

* 富文本 XSS。
* 非法外链协议。
* 越权绑定未公开数据。
* 非法媒体 ID 引用。
* 过大的页面 Schema 和异常嵌套结构。

### 3. 审计动作

必须审计：

* 页面创建。
* 页面草稿保存。
* 页面发布。
* 页面回滚。
* 页面删除或禁用。
* 组件模板状态变更。

建议审计快照：

* 页面基础信息变更前后。
* 草稿 Schema 摘要或摘要 hash。
* 发布版本 ID 和快照 ID。

### 4. 日志要求

* 日志中不打印完整大体积 Schema。
* 日志中不打印完整预览 token。
* 富文本异常或链接校验失败应记录必要上下文，但避免原文全量落日志。

---

## 十六、媒体与富文本协同设计

### 1. 媒体引用校验

页面 Schema 中涉及媒体字段时，保存和发布阶段都应：

* 校验媒体资源存在。
* 校验媒体状态允许被引用。
* 维护页面对媒体的依赖关系。

### 2. 富文本处理

富文本建议策略：

* 后台提交前端富文本内容。
* 服务层做白名单清洗。
* 若富文本中引用媒体，要求同时提交媒体 ID 清单。
* 页面依赖以媒体 ID 清单为准，不仅依赖 HTML 解析。

### 3. 媒体解绑

当页面草稿更新或发布快照替换后：

* 若旧快照不再引用某些媒体，页面依赖应同步重建。
* 媒体绑定状态由媒体模块统一判断，不由页面模块直接改底层媒体状态。

---

## 十七、预览设计

### 1. 预览目标

预览应满足：

* 可查看未发布草稿效果。
* 不污染正式发布快照。
* 不通过 Portal 正式页面缓存返回。

### 2. 预览实现建议

推荐方式：

* 后台保存草稿后，调用预览接口生成 `previewToken`。
* 预览内容写入短期缓存。
* 预览读取时通过 token 拉取对应预览数据。

预览缓存内容建议包括：

* 页面草稿结构。
* 绑定后的预览数据。
* 过期时间。

### 3. 安全要求

* token 必须短期有效。
* token 访问范围应限制为后台登录态或受控来源。
* 过期后必须不可读。

---

## 十八、回滚设计

### 1. 回滚原则

回滚不应直接修改历史快照，而应：

* 选择一个历史快照或历史版本。
* 复制其内容作为新的发布结果。
* 生成新的快照记录和审计记录。

### 2. 回滚流程

```text
管理员发起回滚
  -> 校验目标版本或快照存在
  -> 复制目标内容为新发布基础
  -> 生成新的历史版本记录
  -> 生成新的 ACTIVE 快照
  -> 标记旧 ACTIVE 快照为 SUPERSEDED
  -> 重建依赖
  -> afterCommit 清理页面缓存
  -> 写审计日志
```

### 3. 回滚限制

建议限制：

* 禁止回滚到已损坏或已不兼容的模板结构。
* 若目标快照引用的组件模板已被永久移除，应在回滚前做兼容校验。

---

## 十九、实施步骤建议

### 1. 第一阶段

落地范围：

* 建立数据库表和 Entity/Mapper。
* 建立页面定义、草稿、发布快照服务。
* 建立最小 Portal 页面读取接口。
* 完成首页页面低代码后端基线。

### 2. 第二阶段

落地范围：

* 建立组件模板管理和 Schema 校验能力。
* 建立预览和回滚。
* 建立页面依赖与实体缓存联动。

### 3. 第三阶段

落地范围：

* 将产品、案例等公开列表型展示接入绑定源。
* 建立更多模板和更完整的审计信息。

### 4. 第四阶段

可评估能力：

* 多语言快照。
* 多主题页面。
* 页面模板复制。
* 页面分组和站点级模板继承。

---

## 二十、测试设计要求

### 1. 单元测试

应覆盖：

* 页面状态校验。
* Schema 校验。
* 绑定源校验。
* 回滚流程的合法性校验。
* 依赖抽取逻辑。

### 2. Web 层测试

应覆盖：

* 未登录访问后台接口。
* 非管理员访问发布和回滚接口。
* 非法页面 ID。
* 非法组件模板。
* 非法绑定源。

### 3. 集成测试

应覆盖：

* 保存草稿成功。
* 发布生成版本、快照和依赖。
* 发布后 afterCommit 缓存失效。
* 回滚后 afterCommit 缓存失效。
* 产品或案例变更触发页面缓存联动。
* 页面删除或禁用后 Portal 不可见。

### 4. Portal API 测试

应覆盖：

* 只读取 ACTIVE 快照。
* 不返回草稿内容。
* 隐藏或删除实体不出现在页面结果中。
* 缓存命中和缓存失效行为正确。

---

## 二十一、待确认项

正式实施前建议进一步确认：

* 页面草稿是否需要支持协同编辑或编辑锁。
* 预览 token 是否仅后台可见，还是可分享给受控外部评审方。
* 组件模板协议是代码注册还是后台半动态维护。
* Schema 是否最终直接透给前端渲染器，还是由后端转成更稳定的页面 VO。
* 页面依赖是否需要细分到“列表查询条件级”而不仅是“实体 ID 级”。

---

## 二十二、结论

官网内容低代码的后端设计，核心不是新增几个页面配置表，而是建立一套完整的页面生命周期体系：

* 页面定义负责身份和路由。
* 页面草稿负责高频编辑。
* 页面版本负责历史追踪。
* 发布快照负责 Portal 稳定读取。
* 组件模板负责协议约束。
* 页面依赖负责缓存联动和实体变更感知。

在该体系下，现有强业务模块继续负责产品、案例、媒体、线索和系统支撑能力，低代码模块只负责“如何把公开内容组合成页面”。这样既能满足拖拽搭建、实时预览、发布回滚的需求，也能保持现有项目在安全、事务、审计和缓存一致性上的质量基线。
