# 官网内容低代码重构方案

本文档用于描述官网展示层从传统“固定页面 + 固定接口”模式，演进到“页面搭建 Schema + 组件物料库 + 渲染引擎 + 发布快照”模式的详细重构方案。本文档是项目级设计稿，不替代 [需求说明](需求说明.md)、[架构设计说明](架构设计说明.md)、[数据库设计说明](数据库设计说明.md) 和 [接口文档](接口文档.md)。

---

## 一、文档定位

### 1. 目标

本方案回答以下问题：

* 官网内容如果支持后台拖拽布局、内容编辑、预览、发布、回滚，应如何重构现有后端与前端。
* 哪些能力属于“官网内容低代码”，哪些能力仍应保持强业务模块实现。
* 如何在不破坏现有安全、审计、缓存和模块边界的前提下演进。

### 2. 适用范围

本方案适用于当前官网项目中的展示型页面和展示型内容区块，重点覆盖：

* 首页
* 企业实力
* AI 战略与核心能力
* 产品矩阵与行业方案的展示层
* 标杆案例展示层
* 关于我们
* 联系我们展示页

### 3. 不负责内容

本方案不直接定义以下细节：

* 单个接口的字段级契约，相关内容进入 [接口文档](接口文档.md)。
* 单表 DDL 的最终字段与索引，相关内容进入 [数据库设计说明](数据库设计说明.md)。
* 最终前端技术选型和组件库选型，仅给出能力约束和架构要求。

## 二、背景与问题定义

### 1. 当前实现模式

当前官网后端已经具备较完整的传统内容管理能力：

* Admin 侧可以维护站点配置、导航、首页内容、产品、案例、线索、媒体和系统管理功能。
* Portal 侧通过稳定 API 向前端返回公开内容。
* 后端已经具备认证鉴权、审计日志、Redis 缓存、媒体资源管理和数据库迁移基线。

当前问题不在于“内容是否能配”，而在于“页面结构和展示方式是否仍大量固化在前端代码里”。

### 2. 当前模式的主要限制

如果继续沿用传统前后端模式，会遇到以下问题：

* 页面结构调整依赖前端发版，运营无法自主调整区块顺序和布局。
* 同一类展示内容在多个页面中复用时，前端和后端容易重复开发。
* 页面展示能力分散在多个固定接口和固定组件里，后续扩展成本不断增加。
* 预览、发布、回滚缺乏统一机制，线上页面变更难以治理。
* 页面级缓存只能围绕固定 API 设计，不利于统一页面输出与细粒度失效。

### 3. 重构目标

本次重构目标不是构建一个面向任意业务系统的全功能低代码平台，而是构建一个“官网内容搭建器”：

* 后台可通过拖拽与配置完成页面区块布局调整。
* 后台可配置组件属性、绑定展示数据、控制显示规则。
* 页面有草稿、预览、发布、回滚机制。
* Portal 只读取发布快照，不读取编辑态草稿。
* 强业务实体继续保留现有模块建模，不被页面 JSON 吞并。

## 三、范围边界与非目标

### 1. 纳入低代码范围

以下能力纳入页面搭建域：

* 页面区块编排：顺序、分组、布局、显示与隐藏。
* 组件属性编辑：标题、副标题、按钮、图标、背景图、标签、样式参数。
* 展示数据绑定：绑定产品、案例、行业方案、导航、指标卡等公开内容。
* 草稿、预览、发布、回滚。
* 页面级缓存与页面快照读取。

### 2. 暂不纳入低代码范围

以下能力不建议在当前阶段低代码化：

* 后台账号、登录、权限和角色管理。
* 线索提交流程、导出、状态跟进和敏感数据查看。
* 审计日志管理。
* 媒体资源库和文件上传安全校验。
* 产品、案例、行业方案、线索等核心业务实体的字段建模与状态流转。
* 任意 SQL 数据源、任意脚本、自定义后端逻辑、自定义审批流。

### 3. 非目标

本方案明确不以以下能力为当前目标：

* 无限自由画布式页面编辑。
* 面向任意表单、任意流程、任意权限模型的通用平台。
* 支持用户自定义 JavaScript 执行。
* 支持用户输入任意 SQL 或模板表达式。

## 四、总体思路

### 1. 从固定页面转向 Schema 驱动

现有模式下，页面由前端代码决定结构，后端提供固定接口。重构后的目标模式如下：

```text
后台编辑器
  -> 编辑页面 Schema
  -> 保存草稿版本
  -> 预览
  -> 发布生成快照

Portal 渲染器
  -> 读取已发布快照
  -> 解析页面结构
  -> 绑定公开数据
  -> 渲染正式页面
```

页面不再由单个前端页面文件完全决定，而由以下三类对象共同决定：

* 组件模板：定义能搭哪些内容积木。
* 页面 Schema：定义一个页面如何由这些积木拼起来。
* 发布快照：定义线上稳定读取的最终内容版本。

### 2. 核心原则

* 页面结构低代码化，业务实体不低代码化。
* 展示配置可配，安全边界不可放宽。
* 预览和发布分离，Portal 只读发布态。
* 事务、缓存、审计继续沿用现有项目规则。
* 先做受限低代码，再决定是否扩展为更强平台能力。

## 五、目标架构

### 1. 逻辑架构

建议在现有单体架构中新增独立的页面搭建域：

```text
src/main/java/com/company/officialwebsite
├── application
│   ├── admin
│   │   └── PageBuilderApplicationService
│   └── portal
│       └── PageRenderApplicationService
├── modules
│   ├── site
│   ├── product
│   ├── casecenter
│   ├── media
│   ├── lead
│   ├── system
│   └── pagebuilder
│       ├── controller
│       ├── service
│       ├── mapper
│       ├── entity
│       ├── dto
│       ├── vo
│       ├── converter
│       └── support
└── infrastructure
    ├── cache
    ├── storage
    ├── security
    └── audit
```

### 2. 分层职责

* `modules/pagebuilder/controller`
  负责后台搭建器 API 与 Portal 页面读取 API 的接口适配。
* `application/admin/PageBuilderApplicationService`
  负责页面草稿、组件配置、预览和发布编排。
* `application/portal/PageRenderApplicationService`
  负责组装页面快照与公开业务数据，返回稳定 Portal 页面模型。
* `modules/pagebuilder/service`
  负责版本管理、组件校验、状态流转、发布快照生成、审计记录和事务提交后缓存失效。
* `modules/pagebuilder/mapper`
  负责页面定义、版本、快照、绑定关系的数据访问。

### 3. 与现有模块的关系

* `site`、`product`、`casecenter`、`media`、`system` 等模块继续维护自身实体、查询和业务规则。
* `pagebuilder` 不直接接管这些模块的数据建模，只消费它们暴露出来的只读查询能力。
* 跨模块页面聚合由 `application` 层完成，不允许 `pagebuilder` 直接跨模块访问 Mapper。

## 六、核心领域模型

### 1. PageDefinition

页面定义，描述一个页面的身份和基础路由信息。

建议字段：

* `id`
* `page_key`
* `name`
* `route_path`
* `page_type`
* `status`
* `visible`
* `deleted_marker`
* `created_at`
* `updated_at`

用途：

* 标识首页、关于我们、产品页等页面实体。
* 提供稳定的页面主键和路由绑定。

### 2. PageVersion

页面版本，保存草稿或待发布内容。

建议字段：

* `id`
* `page_id`
* `version_no`
* `version_status`
* `schema_json`
* `schema_hash`
* `remark`
* `created_by`
* `created_at`

版本状态建议：

* `DRAFT`
* `PUBLISHED`
* `ROLLED_BACK`
* `ARCHIVED`

### 3. PublishSnapshot

发布快照，供 Portal 稳定读取。

建议字段：

* `id`
* `page_id`
* `version_id`
* `snapshot_json`
* `snapshot_hash`
* `published_by`
* `published_at`

设计目的：

* Portal 永远读取发布快照。
* 后台草稿修改不影响线上页面。
* 回滚以发布快照为单位完成。

### 4. ComponentTemplate

组件模板，定义搭建器可使用的标准物料。

建议字段：

* `id`
* `component_code`
* `name`
* `category`
* `schema_definition_json`
* `default_props_json`
* `binding_capability_json`
* `status`

### 5. ComponentInstance

组件实例，指页面中的一个具体组件节点。组件实例可以仅保存在 `schema_json` 中，也可以按需拆表。

当前阶段建议：

* 页面结构和组件实例以 `schema_json` 为主。
* 只有当后续需要复杂查询、跨版本追踪、组件级审计时，再考虑拆分为独立表。

### 6. DataBinding

组件与业务数据之间的绑定定义。

建议字段：

* `id`
* `page_id`
* `component_instance_id`
* `binding_type`
* `binding_source`
* `binding_config_json`

绑定类型建议：

* `STATIC`
* `ENTITY`
* `AGGREGATE`

## 七、页面 Schema 设计

### 1. 设计目标

Schema 要满足以下要求：

* 可表达页面的区块顺序与布局。
* 可表达组件的静态属性。
* 可表达组件的数据绑定来源。
* 可做服务端校验和前端渲染。
* 避免引入任意脚本和任意表达式执行。

### 2. 推荐结构

```json
{
  "pageKey": "home",
  "name": "首页",
  "layout": {
    "type": "vertical",
    "responsive": {
      "mobileColumns": 1,
      "desktopColumns": 12
    }
  },
  "sections": [
    {
      "id": "hero_1",
      "component": "HeroBanner",
      "props": {
        "title": "让组织拥有持续进化的数字智能能力",
        "subtitle": "数据驱动增长，AI赋能决策",
        "backgroundMediaId": 1001,
        "primaryButtonText": "了解解决方案",
        "primaryButtonLink": "/solutions"
      },
      "binding": {
        "mode": "static"
      },
      "style": {
        "theme": "brand-dark",
        "paddingTop": 64,
        "paddingBottom": 64
      },
      "visible": true
    },
    {
      "id": "product_grid_1",
      "component": "ProductGrid",
      "props": {
        "title": "产品矩阵",
        "columns": 4
      },
      "binding": {
        "mode": "entity",
        "source": "product",
        "query": {
          "visible": true,
          "limit": 8
        }
      },
      "visible": true
    }
  ],
  "seo": {
    "title": "首页",
    "keywords": ["数字化", "AI", "官网"],
    "description": "官网首页"
  }
}
```

### 3. 字段设计原则

* `component` 必须来自后台已注册的组件模板，不允许前端任意字符串透传。
* `props` 只允许模板定义过的字段。
* `binding` 只允许白名单来源和白名单参数。
* `style` 只允许有限样式参数，不支持自定义 CSS 文本。
* `visible` 仅控制页面内显示，不替代实体可见性规则。

### 4. 不支持能力

为避免安全和维护风险，Schema 当前阶段不支持：

* 自定义脚本执行。
* 任意 HTML 注入。
* 任意 SQL 数据源。
* 任意条件表达式和复杂模板语言。
* 自定义 CSS 文本和自定义外部脚本地址。

## 八、组件物料体系

### 1. 组件分类

建议将官网组件分为以下类别：

* 基础结构组件：Section、Container、Grid、Tabs、Accordion。
* 展示组件：HeroBanner、TextBlock、ImageCard、MetricCards、LogoWall、Timeline、TagGroup。
* 业务展示组件：ProductGrid、IndustrySolutionGrid、CaseGrid、NavigationBar、ContactInfoBlock。
* 富文本组件：RichTextBlock。
* 表单组件：ContactLeadForm。
* SEO 组件：MetaConfigBlock。

### 2. 组件模板元数据

每个组件模板建议提供以下元信息：

* 组件编码
* 组件名称
* 组件分类
* 可编辑字段定义
* 默认属性
* 支持的数据绑定类型
* 是否允许多实例
* 是否允许嵌套子组件

### 3. 组件字段定义建议

字段定义建议包括：

* `fieldKey`
* `label`
* `type`
* `required`
* `maxLength`
* `enumOptions`
* `mediaType`
* `defaultValue`
* `helpText`

字段类型建议：

* `TEXT`
* `TEXTAREA`
* `NUMBER`
* `BOOLEAN`
* `ENUM`
* `MEDIA`
* `LINK`
* `TAG_LIST`
* `RICH_TEXT`
* `COLOR_TOKEN`

### 4. 当前阶段推荐首批物料

首批组件建议只覆盖现有官网高复用区块：

* HeroBanner

## 九、前端重构方案

### 1. 总体拆分

前端建议拆成两套运行模式，但共用同一套组件协议：

* `Portal Renderer`
  渲染线上正式页面。
* `Editor Renderer`
  渲染后台编辑器中的可编辑页面。

### 2. 核心前端能力

前端需要补齐以下基础设施：

* 组件注册中心
* Schema 解析器
* 拖拽排序器
* 栅格布局引擎
* 属性面板生成器
* 数据绑定配置面板
* 预览模式
* 页面差异对比能力
* 发布前校验提示

### 3. 组件协议建议

建议每个前端组件统一实现以下能力：

* `componentCode`
* `render(props, data, mode)`
* `getMeta()`
* `getDefaultProps()`
* `getEditableFields()`
* `validateProps(props)`

其中 `mode` 建议至少区分：

* `portal`
* `editor`
* `preview`

### 4. 拖拽编辑策略

官网场景更适合“受限拖拽”，不建议一开始实现无限自由画布。

建议首阶段支持：

* 区块上下排序
* 栅格列宽调整
* 组件新增、删除、复制
* 组件显隐
* 区块级样式切换

暂不支持：

* 任意像素级自由定位
* 任意层级容器嵌套
* 用户自定义脚本
* 任意 CSS 注入

### 5. 预览模式

预览模式需要满足：

* 后台未发布草稿可以被预览。
* Portal 正式流量不可访问草稿。
* 预览链接可以使用临时 token 或后台会话态访问。
* 预览页面需明确标记当前版本和发布时间状态。

## 十、后端重构方案

### 1. 新增模块建议

建议新增 `modules/pagebuilder`，内部结构如下：

```text
modules/pagebuilder
├── controller
│   ├── AdminPageController.java
│   ├── AdminPageVersionController.java
│   ├── AdminComponentTemplateController.java
│   └── PortalPageController.java
├── service
│   ├── PageDefinitionService.java
│   ├── PageVersionService.java
│   ├── PagePublishService.java
│   ├── ComponentTemplateService.java
│   └── PageSchemaValidationService.java
├── mapper
├── entity
├── dto
├── vo
├── converter
└── support
```

### 2. 应用层建议

建议新增：

* `application.admin.PageBuilderApplicationService`
* `application.portal.PageRenderApplicationService`

职责：

* 后台应用层编排页面保存、预览、发布、回滚。
* Portal 应用层从快照出发，组装页面渲染所需公开数据。

### 3. 关键服务职责

* `PageSchemaValidationService`
  负责校验组件类型、字段、媒体引用、链接协议、数据绑定参数。
* `PageVersionService`
  负责草稿保存、版本生成和版本查询。
* `PagePublishService`
  负责生成发布快照、写审计日志、注册事务提交后缓存失效。
* `ComponentTemplateService`
  负责组件模板注册和元数据读取。

### 4. 与现有业务模块的依赖方式

`pagebuilder` 不应直接访问 `product`、`casecenter`、`site` 等模块的 Mapper。

推荐方式：

* 由对应模块暴露只读查询 Service。
* Portal 聚合走 `application.portal` 层。
* 公共查询结果统一转换为页面渲染所需的数据结构。

## 十一、数据模型与数据库设计建议

### 1. 新增表建议

建议新增以下表：

* `cms_page_definition`
* `cms_page_version`
* `cms_component_template`
* `cms_page_publish_snapshot`
* `cms_component_binding`

### 2. `cms_page_definition`

用途：

* 存放页面元信息。

核心字段建议：

* `id BIGINT`
* `page_key VARCHAR`
* `name VARCHAR`
* `route_path VARCHAR`
* `page_type VARCHAR`
* `status VARCHAR`
* `visible TINYINT`
* `sort_order INT`
* `deleted_marker BIGINT`
* `version INT`
* `created_at DATETIME`
* `updated_at DATETIME`
* `created_by BIGINT`
* `updated_by BIGINT`

索引建议：

* 唯一索引：`(page_key, deleted_marker)`
* 唯一索引：`(route_path, deleted_marker)`

### 3. `cms_page_version`

用途：

* 存放页面草稿与历史版本。

核心字段建议：

* `id BIGINT`
* `page_id BIGINT`
* `version_no INT`
* `version_status VARCHAR`
* `schema_json JSON or LONGTEXT`
* `schema_hash VARCHAR`
* `remark VARCHAR`
* `created_at DATETIME`
* `created_by BIGINT`

索引建议：

* 普通索引：`(page_id, version_no)`
* 普通索引：`(page_id, version_status)`

### 4. `cms_component_template`

用途：

* 存放组件模板和字段协议。

核心字段建议：

* `id BIGINT`
* `component_code VARCHAR`
* `name VARCHAR`
* `category VARCHAR`
* `schema_definition_json JSON or LONGTEXT`
* `default_props_json JSON or LONGTEXT`
* `binding_capability_json JSON or LONGTEXT`
* `status VARCHAR`
* `sort_order INT`

索引建议：

* 唯一索引：`component_code`

### 5. `cms_page_publish_snapshot`

用途：

* 存放线上发布快照。

核心字段建议：

* `id BIGINT`
* `page_id BIGINT`
* `version_id BIGINT`
* `snapshot_json JSON or LONGTEXT`
* `snapshot_hash VARCHAR`
* `published_at DATETIME`
* `published_by BIGINT`

索引建议：

* 普通索引：`(page_id, published_at)`

### 6. `cms_component_binding`

用途：

* 存放组件绑定关系和绑定配置。

核心字段建议：

* `id BIGINT`
* `page_id BIGINT`
* `component_instance_id VARCHAR`
* `binding_type VARCHAR`
* `binding_source VARCHAR`
* `binding_config_json JSON or LONGTEXT`
* `created_at DATETIME`
* `updated_at DATETIME`

### 7. JSON 与结构化字段的边界

应遵循以下原则：

* 页面结构、区块顺序、局部样式、静态文案适合存 JSON。
* 需要检索、排序、唯一约束、权限控制、统计分析的业务数据，不应只存 JSON。
* 产品、案例、行业方案、线索、媒体、审计仍由现有强业务表维护。

---

## 十二、API 设计草案

### 1. Admin API

建议新增以下后台接口：

* `GET /admin/api/page-builder/pages`
  获取页面列表。
* `POST /admin/api/page-builder/pages`
  创建页面定义。
* `GET /admin/api/page-builder/pages/{pageId}`
  获取页面元信息。
* `GET /admin/api/page-builder/pages/{pageId}/draft`
  获取当前草稿。
* `PUT /admin/api/page-builder/pages/{pageId}/draft`
  保存当前草稿。
* `POST /admin/api/page-builder/pages/{pageId}/preview`
  生成预览信息。
* `POST /admin/api/page-builder/pages/{pageId}/publish`
  发布页面。
* `POST /admin/api/page-builder/pages/{pageId}/rollback`
  回滚到指定发布快照。
* `GET /admin/api/page-builder/pages/{pageId}/versions`
  获取版本历史。
* `GET /admin/api/page-builder/component-templates`
  获取组件模板列表。

所有写接口必须满足：

* 仅允许 `Administrator`。
* 记录审计日志。
* 执行 Schema 校验。
* 发布后走事务提交后缓存失效。

### 2. Portal API

建议新增以下前台接口：

* `GET /portal/api/pages/{pageKey}`
  获取页面发布快照或渲染模型。

可选扩展：

* `GET /portal/api/pages/{pageKey}/meta`
  仅获取页面 SEO 和基础信息。

### 3. Portal 输出策略

Portal 输出建议有两种实现方向，二选一：

* 直接返回页面 Schema 与数据绑定结果，由前端渲染器解释。
* 服务端先聚合成稳定页面模型，再由前端渲染器消费。

当前阶段更推荐第一种中的“受控 Schema 输出”，原因是：

* 页面扩展性更强。
* 更适合统一渲染器。
* 页面缓存天然以快照为单位。

前提是：

* 输出结构必须稳定。
* 不暴露后台草稿和管理字段。
* 不暴露隐藏、删除或未发布内容。

## 十三、发布、回滚与缓存方案

### 1. 状态流转

建议页面版本状态流转如下：

```text
DRAFT
  -> PUBLISHED
  -> ARCHIVED

PUBLISHED
  -> ROLLED_BACK
```

说明：

* 编辑器保存内容时写入草稿版本。
* 发布时根据草稿生成独立快照。
* 回滚时以某个历史快照重新生成新的发布结果，而不是直接修改旧快照记录。

### 2. 发布流程

建议发布流程如下：

```text
管理员提交发布
  -> 校验页面状态
  -> 校验 Schema 合法性
  -> 校验组件引用的媒体和绑定源
  -> 生成发布快照
  -> 写审计日志
  -> 注册 afterCommit 缓存失效
  -> 提交事务
  -> 删除页面缓存
  -> 调度延迟二次删除
```

### 3. 回滚流程

建议回滚流程如下：

```text
管理员选择历史快照
  -> 校验目标快照存在
  -> 复制快照内容生成新发布记录
  -> 写审计日志
  -> 注册 afterCommit 缓存失效
  -> 提交事务
```

这样可以保证：

* 回滚也是显式操作，有可审计记录。
* 历史快照不可被直接修改。
* Portal 永远读取“最新有效发布快照”。

### 4. 缓存设计

缓存 key 建议：

* `official:portal:page:{pageKey}`
* `official:portal:page-meta:{pageKey}`

要求：

* 只缓存已发布快照。
* 必须设置 TTL。
* 发布和回滚都必须 afterCommit 失效。
* 保留延迟二次删除兜底。

### 5. 页面缓存与实体缓存的关系

页面快照缓存不替代现有业务实体缓存。

建议原则：

* 页面级输出优先缓存页面快照结果。
* 如果某些组件仍依赖热点实体查询，可以保留实体缓存，但需明确失效顺序。
* 当产品、案例等公开数据变更时，应识别受影响页面并清理关联页面缓存。

## 十四、安全、审计与治理要求

### 1. 权限要求

* 所有 `/admin/api/page-builder/**` 写接口必须要求登录并限制 `Administrator`。
* 预览、发布、回滚均属于高风险操作，必须单独记录审计日志。
* Portal 不允许读取草稿和预览态内容。

### 2. Schema 安全校验

页面 Schema 至少应校验以下内容：

* 组件类型必须存在于模板白名单。
* 组件字段必须符合字段定义、长度和枚举限制。
* 链接必须限制协议和域名白名单。
* 富文本必须做白名单清洗。
* 媒体 ID 必须校验存在性和访问范围。
* 数据绑定源必须在白名单内，不能接受任意表名或任意 SQL。

### 3. 审计要求

以下动作必须审计：

* 创建页面
* 修改草稿
* 删除页面
* 预览页面
* 发布页面
* 回滚页面
* 组件模板上下线

审计字段应至少包含：

* 操作者 ID
* 操作者名称
* 模块
* 动作
* 页面 ID
* 目标版本 ID
* 结果
* IP
* User-Agent
* traceId

### 4. 日志与敏感信息

页面搭建域虽然不是线索域，但仍应注意：

* 日志中不直接打印完整 Schema 内容中的敏感字段。
* 不记录后台会话令牌和完整预览 token。
* 审计快照中如包含外链、富文本、媒体引用，应避免产生过长或不可检索日志。

## 十五、前后端迁移路线

### 1. 阶段一：基础搭建域落地

目标：

* 建立 `pagebuilder` 模块。
* 建立页面定义、版本、发布快照、组件模板能力。
* 建立最小可用的后台编辑器和 Portal 页面渲染器。

范围：

* 先只支持首页。
* 先只支持静态组件和少量实体绑定。

完成判定：

* 首页可通过后台拖拽区块排序和编辑内容。
* 首页具备草稿、预览、发布、回滚。
* 发布后 Portal 页面更新符合现有缓存规则。

### 2. 阶段二：展示型页面迁移

目标：

* 迁移企业实力、关于我们、联系我们展示页。
* 补齐时间轴、标签组、Logo 墙、价值观卡片等物料。

完成判定：

* 展示型页面主要结构均可后台配置。
* 前端不再为这些页面写死区块顺序。

### 3. 阶段三：业务展示页迁移

目标：

* 将产品列表页、行业方案页、案例列表页的展示层接入搭建器。
* 产品和案例实体管理仍留在原业务模块。

完成判定：

* 页面可配置布局。
* 组件可绑定产品和案例的公开只读数据。
* 不影响实体新增、编辑、隐藏、删除的原有业务规则。

### 4. 阶段四：增强能力评估

可评估能力：

* 页面模板复制
* 多语言
* 多主题
* 组件市场
* 页面级 AB 实验

前提：

* 阶段一到三已稳定。
* 安全、缓存和审计策略已在生产验证过。

## 十七、风险与待确认项

### 1. 主要风险

* Schema 设计过度通用，导致项目从官网演进失控为平台产品。
* 前端只做拖拽界面，没有形成统一组件协议，后续维护复杂度失控。
* 页面快照与实体数据缓存关系不清，导致线上展示不一致。
* 让强业务数据过度 JSON 化，导致查询、统计和权限控制变差。
* 未限制富文本、外链和绑定源，带来 XSS 或越权风险。

### 2. 当前建议

* 先做受限低代码，不做全通用平台。
* 先迁移展示型页面，不先改线索、鉴权和系统管理。
* 先用白名单绑定源，不开放任意数据访问。
* 先以页面快照为发布单位，不让 Portal 读草稿。

### 3. 待确认项

以下内容在正式实施前建议补充明确：

* 前端编辑器技术方案和现有前端工程结构。
* 预览 token 的生命周期和访问策略。
* 页面 Schema 是否直接由 Portal 渲染，还是后端转换为稳定页面模型。
* 产品、案例等实体变更时，如何维护“受影响页面”的关联关系。
* 是否需要页面审核流或双人发布机制。

---

## 十八、结论

本项目如果要实现“官网内容拖拽搭建、布局调整、内容实时配置”的低代码能力，合理路线不是推翻现有业务模块，而是在现有单体架构上新增“页面搭建域”，以页面 Schema、组件模板、发布快照和统一渲染协议承载展示层重构。

该方案的关键不是拖拽本身，而是以下四点：

* 页面结构从代码抽离为 Schema。
* 组件能力抽象为受控物料库。
* 发布与预览治理独立于编辑态草稿。
* Portal 输出和缓存围绕发布快照而不是编辑状态运行。

在此基础上，现有 `site`、`product`、`casecenter`、`lead`、`media`、`system` 等模块仍保留其强业务职责，低代码化只覆盖官网展示层，既能提高运营效率，又能控制系统复杂度与安全风险。
