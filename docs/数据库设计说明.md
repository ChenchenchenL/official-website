# 数据库设计说明

本文档定义官网后台系统与 Portal 前台接口的数据库设计规则。本文档**不提前固化具体业务表结构**，不列出每张表的完整字段清单；后续具体表结构应在实现对应模块时，通过迁移脚本、Entity、Mapper、接口文档和 Review 逐步确定。

数据库设计必须遵守 [后端开发规则](../rules/backend.md)、[安全开发规则](../rules/security.md)、[测试规则](../rules/testing.md) 和 [代码 Review 规则](../rules/review.md)。

---

## 一、文档定位与适用范围

### 1. 文档定位

* **只定义规则**：本文只约束数据库命名、通用字段、逻辑删除、索引、关联、媒体、线索、审计、缓存支撑和迁移脚本规则。
* **不定义死表结构**：本文不提前固定 `cms_*`、`media_*`、`lead_*`、`sys_*` 等具体表的字段明细。
* **实现时再细化**：实际表结构必须随业务模块实现同步补充迁移脚本，并接受 Review。
* **规则优先**：如后续表设计与本文规则冲突，必须先修改本文或说明例外原因，不能在迁移脚本中绕过规则。

### 2. 适用范围

适用于以下数据：

* 官网内容数据：站点配置、导航、首页、产品、行业案例、新闻资讯等。
* 媒体资源数据：图片、图标、视频、PDF、衍生版本、媒体引用关系、媒体处理任务。
* 客户线索数据：咨询表单、跟进状态、联系方式、处理备注。
* 系统支撑数据：后台用户、角色、审计日志、缓存失效记录、任务执行记录。

## 二、命名规则

### 1. 表命名

* 表名统一使用小写下划线。
* 表名必须按模块加前缀，便于识别归属：
    * `cms_`：官网内容管理数据。
    * `media_`：媒体资源和媒体引用数据。
    * `lead_`：客户咨询线索数据。
    * `sys_`：后台账号、角色、审计、系统支撑数据。
* 禁止使用含糊表名，例如 `data`、`info`、`record`、`list`。
* 表名应表达业务实体，不应表达页面组件或前端样式。

### 2. 字段命名

* 字段名统一使用小写下划线。
* 主键字段统一命名为 `id`。
* 外部关联字段统一使用 `{entity}_id` 形式，例如 `media_id`、`category_id`。
* 状态字段必须语义明确，例如 `visible`、`status`、`deleted_marker`。
* 时间字段必须带 `_at` 后缀，例如 `created_at`、`updated_at`、`handled_at`。
* 禁止使用无语义缩写，例如 `ct`、`ut`、`st`。

### 3. 索引命名

* 主键索引：`pk_{table}`。
* 普通索引：`idx_{table}_{columns}`。
* 唯一索引：`uk_{table}_{columns}`。
* 索引名应能看出索引用途，避免无意义编号。

## 三、通用字段规则

### 1. 主键规则

* 所有业务表主键统一使用 `BIGINT` 类型。
* ID 生成策略必须在项目初始化时统一，禁止不同模块混用多种策略。
* 不得使用业务字段作为主键，例如路径、编码、手机号、邮箱。

### 2. 时间字段

常规业务表应包含：

* `created_at`：创建时间。
* `updated_at`：更新时间。

要求：

* 时间字段类型建议使用 `DATETIME`。
* 后端统一使用 `LocalDateTime` 映射。
* 时间写入由后端统一填充，不依赖前端传入。

### 3. 操作者字段

后台可修改的数据表应包含：

* `created_by`：创建人 ID。
* `updated_by`：更新人 ID。

要求：

* 系统自动任务写入时，应使用系统用户 ID 或约定的系统操作者标识。
* Portal 匿名提交数据可允许 `created_by` 为空，但必须记录来源信息。

### 4. 乐观锁字段

高频编辑或可能并发覆盖的数据表应包含：

* `version`：乐观锁版本。

要求：

* 更新时必须校验版本，避免后提交覆盖先提交。
* 后台内容编辑、系统配置、媒体元数据、线索处理备注等建议使用乐观锁。

### 5. 排序字段

前台需要稳定排序的数据表应包含：

* `sort_order`：排序值，数值越小越靠前。

要求：

* Portal 查询必须使用稳定排序规则。
* 同一排序值下应补充主键或创建时间作为二级排序，避免分页抖动。

## 四、内容可见性与逻辑删除规则

### 1. 可见性字段

内容型数据默认使用：

* `visible`：是否前台显示，`1` 表示显示，`0` 表示隐藏。

要求：

* Portal 查询必须只返回 `visible = 1` 的内容。
* 后台列表可查询显示和隐藏内容。
* 隐藏不等于删除，隐藏内容仍可继续编辑和恢复显示。

### 2. 逻辑删除字段

涉及逻辑删除的数据表统一使用：

* `deleted_marker BIGINT NOT NULL DEFAULT 0`

语义：

* `deleted_marker = 0`：活跃数据。
* `deleted_marker != 0`：已逻辑删除数据。
* 删除时将 `deleted_marker` 写入本行主键 ID 或其他全局唯一删除标记。

要求：

* Portal 查询必须仅查询 `deleted_marker = 0` 的活跃数据。
* 后台默认列表也应仅查询 `deleted_marker = 0` 的活跃数据。
* 只有审计、回收站或特殊管理能力才能查询已删除数据。
* 禁止使用布尔型 `is_deleted` 作为唯一约束兜底字段。

### 3. 内容生命周期边界

当前阶段只默认支持：

* 显示。
* 隐藏。
* 逻辑删除。

以下能力不作为当前默认数据库设计前提：

* 草稿。
* 审批。
* 定时发布。
* 定时下线。
* 多版本发布流。

如后续引入上述能力，必须先更新 PRD、架构说明、接口文档、测试规则和本文档。

## 五、唯一约束与索引规则

### 1. 唯一约束规则

有唯一业务键的数据表必须使用数据库唯一索引兜底，不能只依赖 Service 校验。

涉及逻辑删除且存在唯一业务键时，必须使用联合唯一索引：

```text
(business_key, deleted_marker)
```

示例：

```text
(path, deleted_marker)
(code, deleted_marker)
(name, deleted_marker)
```

原因：

* 活跃数据中业务键必须唯一。
* 逻辑删除后的历史数据不应阻止重新创建相同业务键。
* 布尔删除字段无法支持多条历史删除数据并存。

### 2. 普通索引规则

以下字段通常需要建立索引：

* Portal 高频过滤字段，例如 `visible`、`deleted_marker`。
* 分类、父级、业务归属字段，例如 `category_id`、`parent_id`、`industry_id`。
* 排序字段，例如 `sort_order`、展示时间字段。
* 线索筛选字段，例如处理状态、提交时间、跟进人。
* 审计筛选字段，例如操作者、模块、动作、对象 ID、操作时间。
* 媒体筛选字段，例如状态、上传人、上传时间、文件哈希。

### 3. 索引设计原则

* 高频查询先按实际接口查询条件设计索引，不为不存在的查询提前建大量索引。
* 组合索引字段顺序必须匹配常用查询条件和排序方式。
* 避免过多低选择性单列索引。
* 大字段不得进入索引。
* 每个新增索引必须能说明服务的查询场景。

## 六、关联关系与外键策略

### 1. 关联字段

业务表之间通过 ID 字段建立逻辑关联，例如：

* 内容表引用媒体资源 ID。
* 子表引用父表 ID。
* 审计日志引用操作者 ID。
* 线索引用跟进人 ID。

要求：

* 关联字段必须建立普通索引。
* Service 层必须校验关联对象是否存在且可用。
* Portal 查询不得因为关联数据缺失而直接抛出系统异常，应按业务场景降级或返回明确错误。

### 2. 外键策略

当前阶段默认不强制数据库物理外键。

原因：

* 内容管理系统需要逻辑删除、历史保留和审计追溯。
* 媒体资源存在绑定、解绑和延迟清理流程。
* 物理外键可能阻塞数据修复、逻辑删除和历史数据保留。

要求：

* 不使用物理外键不等于不管理一致性。
* 必须通过索引、Service 校验、事务和测试保证业务一致性。

## 七、媒体资源数据规则

### 1. 媒体元数据

媒体资源必须记录可支撑安全校验、渲染和清理的元数据，包括但不限于：

* 原始文件名。
* 存储路径。
* 访问地址。
* 文件大小。
* MIME 类型。
* 文件扩展名。
* 图片宽高或视频基础信息。
* 文件哈希。
* 上传人。
* 上传时间。
* 资源状态。

### 2. 媒体状态

媒体资源至少支持以下状态：

* `TEMPORARY`：已上传但未被有效业务内容引用。
* `BOUND`：至少被一个有效业务内容引用。
* `UNBOUND`：曾经被引用，但当前不再被有效业务内容引用，等待清理。
* `DELETED`：已删除或已归档，不再对外使用。

要求：

* 业务保存成功后，引用到的媒体从 `TEMPORARY` 转为 `BOUND`。
* 业务内容更新、替换媒体或逻辑删除后，不再被任何有效内容引用的媒体应转为 `UNBOUND`。
* 定时清理任务只清理过期的 `TEMPORARY` 和 `UNBOUND` 资源。

### 3. 媒体衍生版本

图片压缩、缩略图、WebP 等衍生文件不得阻塞上传请求线程。

要求：

* 衍生版本应记录与原始媒体的关联关系。
* 衍生版本生成失败必须可追踪。
* 原始文件应保留，除非后续业务明确允许删除原图。

---

## 八、富文本媒体引用规则

富文本正文中的图片、PDF 等媒体资源必须通过媒体引用清单维护绑定关系。

要求：

* 富文本保存接口必须提交媒体 ID 清单或媒体引用清单。
* 后端以媒体清单维护媒体引用关系。
* HTML 正文只用于展示和安全清洗，不作为唯一的媒体绑定依据。
* 禁止只依赖正则或 DOM 解析 HTML 来判断媒体是否绑定。
* 保存富文本时应记录媒体清单快照或可追踪引用关系，便于排查和重建。

---

## 九、客户线索数据规则

### 1. 敏感字段规则

客户线索中的以下数据属于敏感数据：

* 姓名。
* 手机号。
* 邮箱。
* 公司名称。
* 职位。
* 留言内容。

要求：

* 列表默认脱敏展示手机号和邮箱。
* 查看单条完整联系方式必须记录审计日志。
* 仅管理员可查看单条完整联系方式并更新处理状态和备注。
* 线索导出默认仅允许管理员操作。

### 2. 检索字段规则

如需要按手机号、邮箱检索，应优先使用哈希字段或等价方案，避免直接使用明文敏感字段做模糊查询。

### 3. 处理状态规则

线索处理状态应使用稳定枚举或字典值，不得使用随意字符串。

当前最小状态包括：

* 待处理。
* 跟进中。
* 已完成。
* 无效。

---

## 十、审计日志数据规则

### 1. 必须审计的行为

以下行为必须写审计日志：

* 登录成功、登录失败、登出。
* 内容显示、隐藏、逻辑删除。
* 系统级删除。
* 系统配置修改。
* 账号启停、密码重置。
* 查看线索完整联系方式。
* 线索导出。
* 媒体删除。
* 手工刷新缓存。

### 2. 审计字段要求

审计日志必须能支撑追溯，至少记录：

* 操作者 ID。
* 操作者名称。
* IP。
* User-Agent。
* 操作模块。
* 操作动作。
* 操作对象 ID。
* 操作结果。
* 操作时间。
* `traceId`。

涉及内容修改、系统配置修改、权限变更等操作，应记录变更前后核心字段快照。

### 3. 审计日志保护

* 审计日志不得被后台普通业务流程修改或删除。
* 管理员也不应直接删除审计记录。
* 如存在日志归档或清理需求，必须先定义保留周期和归档策略。

## 十一、缓存失效支撑规则

数据库可按需要记录缓存失效或任务执行日志，但不要求所有缓存操作都入库。

当前使用 `portal_cache_invalidation_retry` 持久化 Portal 缓存删除失败与延迟二删调度失败的补偿任务；其保存待删除 key、重试次数、下次重试时间、最近失败原因与执行状态。任务可由任一实例恢复执行，成功后保留执行记录供审计追溯。

需要记录的场景：

* 缓存失效失败。
* 延迟二次删除调度失败。
* 手工刷新缓存。
* 关键 Portal 缓存重建失败。

缓存相关记录应包含：

* 缓存 key 或模块标识。
* 触发业务模块。
* 触发对象 ID。
* 操作者或系统任务标识。
* 执行结果。
* 错误信息摘要。
* 触发时间。

---

## 十二、迁移脚本规则

### 1. 基本要求

* 所有表结构变更必须通过迁移脚本管理。
* 迁移脚本必须可在空库执行。
* 涉及历史数据的迁移必须说明兼容策略和回滚策略。
* 不得手工在线上数据库直接改表后不补脚本。

### 2. 脚本内容要求

迁移脚本必须显式声明：

* 字段类型。
* 非空约束。
* 默认值。
* 主键。
* 普通索引。
* 唯一索引。
* 字段注释或表注释，按团队规范执行。

### 3. 高风险变更

以下变更必须重点 Review：

* 删除字段。
* 修改字段类型。
* 修改字段语义。
* 新增唯一约束。
* 大表新增索引。
* 数据回填。
* 清理历史数据。

---

## 十三、禁止事项

* 禁止提前在本文档中固化尚未实现模块的完整表结构。
* 禁止使用布尔 `is_deleted` 解决逻辑删除唯一性问题。
* 禁止 Portal 查询默认返回隐藏或已逻辑删除数据。
* 禁止接口直接返回数据库 Entity。
* 禁止将富文本 HTML 作为媒体绑定关系的唯一来源。
* 禁止明文存储密码、Token、验证码和密钥。
* 禁止日志或审计快照中保存未脱敏的敏感凭证。
* 禁止没有查询场景就盲目创建大量索引。
* 禁止无迁移脚本修改数据库结构。

---

## 十四、数据库设计评审清单

设计或修改数据库结构前必须检查：

* 是否符合表命名和字段命名规则。
* 是否需要 `visible`、`deleted_marker`、`version`、`sort_order` 等通用字段。
* 涉及逻辑删除和唯一业务键时，是否使用 `(business_key, deleted_marker)` 联合唯一索引。
* Portal 查询是否只读取显示且未删除的数据。
* 关联字段是否建立索引并由 Service 校验一致性。
* 媒体资源是否维护引用关系和解绑清理规则。
* 富文本媒体是否通过媒体 ID 清单或引用清单维护绑定。
* 线索敏感字段是否支持脱敏、审计和受控导出。
* 审计日志是否能追踪关键操作。
* 缓存失效失败是否可追踪。
* 迁移脚本是否可重复审查，并覆盖默认值、索引和约束。

## 十五、可视化编辑器扩展表 (V37)

本节说明 V37 迁移脚本新增的 7 张表的字段策略、逻辑删除规则、审计字段和关键约束。

### 1. `cms_editor_lock`（独占编辑锁）

适用资源类型：`PAGE`、`PRODUCT`、`CASE`、`INDUSTRY_SOLUTION`。

| 字段 | 类型 | 策略说明 |
| --- | --- | --- |
| `resource_type` + `resource_id` | VARCHAR/BIGINT | 唯一约束通过 `(resource_type, resource_id, deleted_marker)` 联合唯一索引兜底，保证同一资源同一时刻只有一把活跃锁。 |
| `lock_token_hash` | VARCHAR(64) | 只存储 Lock Token 的 SHA-256 哈希摘要，明文 Token 只在创建锁时返回一次，不持久化。 |
| `owner_display_name` | VARCHAR(64) | 脱敏展示名，冲突响应时返回给其他管理员，不得返回原始账号 ID 或邮箱。 |
| `expires_at` + `last_heartbeat_at` | DATETIME | 心跳续期更新 `last_heartbeat_at` 和 `expires_at`；过期检测通过 `idx_cms_editor_lock_expire(deleted_marker, expires_at)` 索引支撑。 |
| `version` | INT | 乐观锁，续期/释放操作必须携带版本号，防止并发抢占产生两把有效锁。 |
| `deleted_marker` | BIGINT | 释放锁时写入本行主键 ID，使联合唯一索引允许历史释放记录并存，活跃锁固定为 `0`。 |
| `created_at` / `updated_at` | DATETIME | 由后端统一填充，`updated_at` 使用 `ON UPDATE CURRENT_TIMESTAMP`。 |

审计要求：强制解锁操作不写此表，写入 `sys_audit_log`，包含解锁人、原因、被解锁资源和 `traceId`。

### 2. 草稿表（`cms_product_draft` / `cms_case_draft` / `cms_industry_solution_draft`）

三张草稿表结构对称，字段策略一致。

| 字段 | 类型 | 策略说明 |
| --- | --- | --- |
| `{entity}_id` | BIGINT | 关联主实体 ID；通过 `({entity}_id, deleted_marker)` 联合唯一索引保证每个实体只有一条活跃草稿。 |
| `draft_json` | LONGTEXT | 草稿完整 JSON，包含 `draft`、基础字段、媒体、富文本、SEO、关联实体等；Portal 不得读取此字段。 |
| `draft_hash` | VARCHAR(64) | 草稿 JSON 的 SHA-256 哈希，用于预览 Token 绑定和发布前一致性校验。 |
| `editor_session_remark` | VARCHAR(255) | 编辑会话备注，可选，不参与业务校验，仅用于审计追溯。 |
| `version` | INT | 乐观锁版本；保存草稿、创建预览、发布、回滚时必须携带并校验，冲突返回 `DETAIL_VERSION_CONFLICT(10012)`。 |
| `created_by` / `updated_by` | VARCHAR(64) | 操作人账号，由 Service 层从 Security Context 填充，不由前端传入。 |
| `deleted_marker` | BIGINT | 逻辑删除，删除时写入本行主键 ID；Portal 和编辑器均只查询 `deleted_marker = 0` 的活跃草稿。 |

### 3. 版本表（`cms_product_version` / `cms_case_version` / `cms_industry_solution_version`）

三张版本表结构对称，字段策略一致。

| 字段 | 类型 | 策略说明 |
| --- | --- | --- |
| `version_no` | INT | 单调递增版本序号；通过 `({entity}_id, version_no, deleted_marker)` 联合唯一索引兜底，禁止覆盖历史版本。 |
| `snapshot_json` | LONGTEXT | 发布时刻的不可变快照 JSON；Portal 详情接口只读取已发布、可见、未逻辑删除的最新快照。 |
| `snapshot_hash` | VARCHAR(64) | 快照 SHA-256 哈希，用于预览/引用一致性校验和去重。 |
| `change_summary` | VARCHAR(255) | 发布或回滚时填写的变更说明，必填，写入审计快照。 |
| `publisher` | VARCHAR(64) | 发布人账号，由 Service 层从 Security Context 填充。 |
| `rollback_source_version_id` | BIGINT NULL | 回滚操作时填写来源版本 ID；正常发布时为 `NULL`。回滚不覆盖历史版本，而是生成新版本并记录来源，便于追溯回滚链路。 |
| `published_at` | DATETIME | 发布时间，由 Service 填充，不依赖前端传入。 |
| `deleted_marker` | BIGINT | 逻辑删除；版本历史原则上不删除，仅在测试清理或特殊管理场景下逻辑删除。 |

审计要求：发布、回滚、下线操作均写入 `sys_audit_log`，记录操作人、资源 ID、版本号、变更说明和 `traceId`。缓存失效在事务提交后（`afterCommit`）触发，失败时记录可重试任务，由 TTL 兜底。

### 4. `cms_page_draft_history`（页面草稿历史修订快照表）

用于实现 P2-1 草稿自动保存与有限数量历史快照追溯。

| 字段 | 类型 | 策略说明 |
| --- | --- | --- |
| `page_id` | BIGINT | 关联页面 ID；通过索引 `idx_page_revision(page_id, revision_no)` 支撑版本排序与快照查找。 |
| `draft_id` | BIGINT | 关联主草稿 ID。 |
| `revision_no` | INT | 单页面内单调递增的历史修订版本序号 (1, 2, 3...)。 |
| `schema_json` | LONGTEXT | 当前修订的完整 Schema 配置 (JSON)；列表摘要查询时默认过滤此字段以提升网络传输效率。 |
| `schema_hash` | VARCHAR(64) | 当前配置 SHA-256 哈希，与 `schema_json` 序列化结果严格匹配。 |
| `editor_session_remark` | VARCHAR(255) | 编辑会话备注（如“自动保存”、“恢复至修订版本 #5”）。 |
| `created_by` | VARCHAR(64) | 操作人账号，由 Service 统一填充。 |
| `created_at` | DATETIME | 修订创建时间。 |

容量策略：单个页面默认仅保留最多 20 条最新修订。新修订写入后，服务端按版本正序物理擦除最老超出 20 条的历史记录，保证数据规模有界且高效。
