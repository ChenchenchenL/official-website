# 架构设计说明

本文档描述官网后台系统的整体架构、分层职责、核心数据流和关键一致性机制。本文档不重复数据库字段细节，数据库物理设计以 [数据库设计说明](数据库设计说明.md) 为准；接口字段、路径和实现位置以 [接口文档](接口文档.md) 为准；开发约束以 `rules/` 目录下规则为准。

---

## 一、架构目标与边界

### 1. 架构目标

* **支撑全站动态配置**：站点配置、导航、首页、产品、案例、新闻、媒体和线索均可由后台维护。
* **保障 Portal 读取稳定**：前台接口只返回显示、未删除、可公开展示的数据。
* **控制复杂度**：当前阶段采用单体 Spring Boot 应用，可多实例部署，不拆分微服务。
* **保障一致性**：写操作在数据库事务内完成业务数据、审计记录和媒体引用维护；数据库事务提交后再触发缓存失效、异步处理和外部副作用。
* **避免过度设计**：当前阶段不引入完整 RBAC 权限矩阵、审批流、消息队列和独立媒体处理服务。

### 2. 系统边界

系统分为两类接口：

* **Admin 后台接口**：用于后台内容管理、媒体管理、线索跟进、后台账号、审计日志和系统配置。
* **Portal 前台接口**：用于官网前台读取公开内容和提交客户咨询表单。

当前后端服务同时承载 Admin API 与 Portal API，但必须在包结构、接口路径、权限控制和缓存策略上保持清晰边界。

## 二、总体部署架构

### 1. 部署形态

当前阶段采用“单体应用，可多实例部署”的架构：

```text
Browser / Admin UI / Portal UI
        |
        v
Load Balancer / Reverse Proxy
        |
        v
Spring Boot Official Website Backend  x N
        |
        |-- MySQL
        |-- Redis or equivalent shared cache
        |-- File storage / object storage
```

### 2. 组件职责

* **Spring Boot 后端**：提供 Admin API、Portal API、认证授权、业务逻辑、媒体元数据管理、审计日志、应用内任务。
* **MySQL**：保存业务数据、媒体元数据、媒体引用关系、线索、用户、审计日志。
* **Redis 或等价共享缓存**：保存 Portal 高频读取数据缓存，支撑多实例缓存一致性。
* **文件存储或对象存储**：保存图片、图标、视频、PDF 等实际文件。
* **应用内任务**：处理图片衍生版本生成、临时/已解绑媒体清理、缓存延迟二次删除等任务。

### 3. 多实例要求

生产环境如部署多个后端实例，不允许依赖本地 JVM 缓存保存 Portal 关键内容。Portal 缓存必须使用 Redis 或等价共享缓存。本地缓存仅允许用于开发、测试或明确的单实例部署。

## 三、后端分层架构

后端包结构以 [后端开发规则](../rules/backend.md) 为准：

```text
src/main/java/com/company/officialwebsite
├── OfficialWebsiteApplication.java
├── common
├── application
├── modules
└── infrastructure
```

### 1. Controller 层

* 接收 HTTP 请求、执行基础注解校验、调用 Service 或 Application Service、返回统一响应。
* Admin Controller 路径统一为 `/admin/api/**`。
* Portal Controller 路径统一为 `/portal/api/**`。
* 不直接调用 Mapper，不写数据库查询，不承载有状态业务规则。

### 2. Application Service 层

* 负责跨模块编排、Portal 聚合查询、只读 VO 组装和复杂用例流程。
* 用于避免模块 Service 之间互相调用形成循环依赖。
* 典型场景包括首页聚合接口、产品详情聚合、行业案例聚合、后台保存内容时的媒体引用协调。

### 3. Module Service 层

* 负责模块内业务规则、事务边界、唯一性校验、显示/隐藏/逻辑删除规则、审计记录和事务提交后动作注册。
* 写操作事务必须声明在 Service 层。
* 缓存失效、外部通知、延迟任务调度等副作用必须在事务提交后执行。

### 4. Mapper 层

* 负责 MyBatis Plus 数据访问。
* 简单 CRUD 使用 MyBatis Plus 能力，复杂查询可使用 XML。
* 不承载业务判断，不拼接未校验的用户输入。

### 5. Infrastructure 层

* 承载缓存、文件存储、安全适配、审计支撑、异步任务等基础设施能力。
* 业务模块通过接口依赖基础设施，不直接散落调用 Redis、对象存储或安全上下文。

## 四、核心业务数据流

### 1. Admin 内容写入链路

```text
Admin Controller
  -> Request DTO validation
  -> Application Service or Module Service
  -> Business validation
  -> Database transaction
  -> Audit log write
  -> Media reference update if needed
  -> Register after-commit cache invalidation
  -> Commit
  -> Execute after-commit actions
  -> Return response
```

关键规则：

* 写操作必须校验角色权限。
* 内容型数据使用显示、隐藏、逻辑删除控制 Portal 可见性。
* 逻辑删除表使用 `deleted_marker`，活跃数据为 `0`。
* 内容变更后必须在事务提交后失效对应 Portal 缓存。

### 2. Portal 读取链路

```text
Portal Controller
  -> Application Service
  -> Read shared cache
  -> Cache hit: return VO
  -> Cache miss: query visible active data
  -> Build stable Portal VO
  -> Write cache with TTL
  -> Return response
```

关键规则：

* Portal 查询必须仅查询 `visible = 1 AND deleted_marker = 0` 的可展示数据。
* Portal 不返回后台 Entity，只返回稳定 VO。
* Portal 缓存必须设置 TTL，禁止永久缓存。

### 3. 客户线索链路

```text
Portal form submit
  -> Basic validation and anti-abuse checks
  -> Save lead consultation
  -> Admin list view with masking
  -> Administrator views single full contact if needed
  -> Audit full contact view
  -> Update status and remark
```

权限边界：

* **管理员**：负责后台全部管理操作，包括内容维护、线索处理、系统配置、账号管理、批量导出和其他高风险操作。
* **前台访客**：仅能匿名访问 Portal 公开内容和提交客户咨询表单，不能进入后台。

### 4. 媒体上传与引用链路

```text
Upload file
  -> Validate file type, size and content
  -> Store original file
  -> Save media_asset as TEMPORARY
  -> Return media ID and URL
  -> Business content save with media reference list
  -> Update media_reference
  -> Mark referenced assets as BOUND
  -> Mark no-longer-referenced assets as UNBOUND
  -> Cleanup task removes expired TEMPORARY / UNBOUND assets
```

关键规则：

* 图片压缩、缩略图、WebP 转换等高耗时处理走应用内异步任务。
* 富文本保存必须提交媒体 ID 清单或引用清单。
* 后端不得只依赖解析 HTML 来判断富文本媒体绑定关系。
* 业务内容更新、替换媒体或逻辑删除时，必须维护媒体引用关系并处理解绑。

## 五、缓存与一致性设计

### 1. 缓存对象

Portal 高频读取的数据可以缓存：

* 站点全局配置。
* 导航菜单。
* 首页聚合数据。
* 产品分类与产品摘要。
* 推荐案例、推荐新闻等公开内容。

客户线索完整信息、后台用户权限等敏感数据默认不作为 Portal 缓存对象。

### 2. 缓存失效策略

内容变更采用：

```text
数据库事务提交成功
  -> 立即删除对应缓存
  -> 调度延迟二次删除
  -> 缓存 TTL 兜底
```

要求：

* 缓存删除必须通过 `afterCommit`、事务事件监听器或等价机制执行。
* 延迟二次删除用于降低并发读旧值后回填缓存的风险。
* 所有 Portal 缓存必须设置 TTL。
* 缓存失效失败不回滚已提交业务数据，但必须记录错误日志，并提供重试或人工刷新能力。

### 3. 缓存键设计

缓存 key 使用统一前缀：

```text
official:portal:{module}:{identifier}
```

示例：

```text
official:portal:site-config
official:portal:navigation
official:portal:home
official:portal:product:category-list
```

## 六、媒体资源处理设计

### 1. 媒体状态

媒体资源至少包含以下状态：

* **TEMPORARY**：已上传但未被有效业务内容引用。
* **BOUND**：至少被一个有效业务内容引用。
* **UNBOUND**：曾经被引用，但当前不再被有效业务内容引用，等待清理。
* **DELETED**：已删除或已归档，不再对外使用。

### 2. 媒体引用关系

系统必须维护业务对象与媒体资源之间的引用关系。直接字段引用和富文本引用都应进入统一引用关系。

富文本媒体引用由接口提交的媒体 ID 清单或引用清单决定。HTML 正文只用于展示内容和安全清洗，不作为唯一的媒体绑定依据。

### 3. 异步处理任务

当前阶段使用应用内任务处理：

* 图片缩略图生成。
* WebP 或压缩版本生成。
* 过期 TEMPORARY 资源清理。
* 过期 UNBOUND 资源清理。
* 缓存延迟二次删除。

后续如上传量、媒体体积或任务可靠性要求明显提高，可演进为消息队列或独立任务服务。

## 七、认证授权与审计设计

### 1. 权限框架

当前架构不强制绑定具体认证授权框架，可选 Spring Security、Sa-Token 或自研适配。无论选型如何，实现必须满足：

* Admin 接口默认需要认证。
* Portal 公开读取接口可匿名访问。
* Portal 表单提交接口需要限流、防刷和输入校验。
* 后台仅保留 `Administrator` 管理员角色。

### 2. 权限职责

* **管理员**：负责全部后台管理操作，包括内容维护、线索处理、系统配置、缓存手工刷新、账号维护、导出和系统级删除等高风险操作。
* **前台访客**：仅能访问 Portal 公开接口，不参与后台权限体系。

### 3. 审计日志

以下操作必须记录审计日志：

* 登录成功、登录失败、登出。
* 内容显示、隐藏、逻辑删除。
* 线索完整联系方式查看、线索导出。
* 账号启停、密码重置。
* 系统配置修改、缓存手工刷新。
* 媒体删除和媒体清理任务异常。

审计日志不得被后台普通业务流程修改或删除。管理员也不应直接删除审计记录。

## 八、异常、日志与可观测性

### 1. 统一异常处理

* 业务异常返回稳定错误码和可理解消息。
* 系统异常由全局异常处理器捕获，不向前端暴露 SQL、堆栈、服务器路径和类名。
* 参数校验失败需要返回字段级错误信息。

### 2. 请求追踪

* 每个请求生成或透传 `traceId`。
* 接口响应和日志中应包含同一个 `traceId`。
* 登录失败、权限拒绝、上传失败、缓存失效失败、媒体任务失败等异常必须可通过日志定位。

### 3. 敏感信息保护

日志不得输出：

* 密码、Token、Cookie、Authorization Header。
* 完整手机号、完整邮箱。
* 对象存储密钥、数据库密码、Webhook 地址。

## 九、测试与演进约束

### 1. 测试重点

架构实现必须优先覆盖：

* Admin 认证与管理员授权。
* Portal 只返回显示、未删除内容。
* `deleted_marker` 与唯一索引规则。
* 事务提交后的缓存失效、延迟二次删除和 TTL。
* 媒体绑定、解绑、富文本媒体引用清单。
* 线索脱敏、管理员查看单条完整联系方式审计、导出限制。

### 2. 演进约束

以下能力如需引入，必须先更新 PRD、架构说明、数据库设计和相关 rules：

* 草稿、审批、定时发布、定时下线。
* 完整 RBAC 权限矩阵。
* 消息队列。
* 独立媒体处理服务。
* 多租户或多站点能力。
* 搜索引擎或全文检索。

## 十、待确认事项

当前基础包名为 `com.company.officialwebsite`。如需更换，必须同步更新源码目录、规则文档、架构说明和接口文档示例。

当前构建方式为 Maven，当前版本基线为 Spring Boot 3.3.6、Java 17、MyBatis Plus 3.5.9。

* MySQL 版本、Redis 版本及部署参数。
* 文件存储使用本地挂载、对象存储还是 CDN 回源。
* Portal 缓存 TTL 默认值和延迟二次删除延迟时间。
* 认证授权框架及登录态方案，例如 Spring Security、Sa-Token、Session、JWT 或其他 Token 机制。
* 是否对接企业微信、钉钉、邮件、CRM 或在线客服。
