# 官网内容低代码后端重构执行顺序计划

本文档基于《低代码后端设计说明.md》与《低代码重构方案.md》，结合当前项目结构（Spring Boot 3.3.6, MyBatis Plus, Flyway, Redis），制定了分步骤的详细后端实施计划。
所有的开发过程必须严格遵守 `AGENTS.md` 及 `rules/` 目录下的开发、安全、测试和评审规则。

### ⚠️ 难点分析
- **JSON 字段边界控制**：哪些数据存入 JSON 字段（如布局、属性、样式），哪些必须拆表保存（如页面依赖、版本记录）。
- **MyBatis Plus TypeHandler 的应用**：如果在 Entity 中直接映射 JSON 字段为复杂的 Model 对象，需要确保 TypeHandler 转换的高效与稳定；如果为了简单，可能初期直接存为 String，这需要评估对后续逻辑的影响。
- **唯一索引与逻辑删除的兼容**：必须确保 `deleted_marker` 参与唯一索引（如 `(page_key, deleted_marker)`）能正确处理逻辑删除后的数据重建问题。

## 阶段四：发布、回滚机制与 Portal 快照读取 ✅
**目标**：实现安全的发布上线流程、完善的审计与缓存失效机制，以及前端正式访问链路。

### 📌 TODO（已完成）
- [x] 完善 `PageVersionService` 和 `PageSnapshotService`：发布时生成 Version + Snapshot，旧快照置为 SUPERSEDED。
- [x] 实现核心发布逻辑 `PagePublishService.publishPage`：
    - 在 `@Transactional` 中：校验草稿 → 生成历史版本 → 生成新 `ACTIVE` 快照 → 旧快照置为 `SUPERSEDED` → 从 Schema 中抽取媒体与数据绑定依赖并写入 `cms_page_dependency`。
- [x] 实现回滚逻辑 `PagePublishService.rollbackPage`：根据所选版本复制并生成新 Version + 新 ACTIVE 快照，同步覆盖草稿。
- [x] 注册事务 `afterCommit` 回调：通过 `PortalCacheInvalidationSupport.invalidate` 内置的 `TransactionSynchronizationManager.afterCommit` 删除 Redis 缓存。
- [x] 记录关键操作审计日志（发布、回滚动作）。
- [x] 实现 `PortalPageRenderService`：读取 Redis 缓存，Cache Miss 则查询 `ACTIVE` 快照、装配绑定数据、回写缓存。
- [x] 实现 `PageBindingResolutionService`：跨 11 个业务模块的数据绑定解析（STATIC/ENTITY/AGGREGATE 分发）。

### ℹ️ 实现说明
- **DTO**：`PagePublishDTO`（含 `changeSummary`）。
- **VO**：`PageVersionVO`、`PortalPageVO`、`PortalPageMetaVO`、`PortalSectionVO`。
- **Converter**：`PageVersionConverter`。
- **发布流程**：校验草稿 → 版本序号递增 → INSERT 版本记录 → UPDATE 旧快照为 SUPERSEDED → INSERT 新 ACTIVE 快照 → 提取依赖 → 审计日志 → 缓存失效。
- **回滚流程**：校验版本 → INSERT 回滚版本 → UPDATE 旧快照 → INSERT 新快照 → 同步草稿 → 重建依赖 → 缓存失效。
- **依赖提取**：遍历 Schema 的 sections，从 `props` 中提取 MEDIA 类型字段，从 `binding` 中提取 ENTITY/AGGREGATE 绑定源，通过 `insertedKeys` 去重后写入 `cms_page_dependency`。
- **绑定解析**：`PageBindingResolutionServiceImpl` 通过 switch source 分发到 11 个业务模块的 Portal 只读接口，支持 `product`/`case` 按 id/ids 查询过滤。
- **Portal 渲染缓存**：`PortalPageRenderService` 使用 `PortalCacheSupport.readCache/writeCache` 二级缓存，Key 为 `official:portal:page:{routePath}` 和 `official:portal:page-meta:{pageKey}`。
- **缓存失效**：`PortalCacheInvalidationSupport.invalidate` 在事务 `afterCommit` 中执行删除 + 延迟二次删除。
- **Portal Controller 说明**：`PortalPageRenderService` 的渲染和元数据接口目前尚未通过 Controller 暴露（仅内部 Service），需在 Phase 5 添加 `PortalPageController`。

### ⚠️ 难点分析
- **事务与缓存一致性**：必须确保只有在数据库事务成功提交后，才去删除页面缓存；如果 `afterCommit` 删除缓存失败，需要考虑延迟二次删除或重试的兜底机制（现有系统中如果有相关基线则直接复用）。
- **页面依赖抽取**：从复杂的 JSON Schema 中准确提取出当前页面绑定了哪些实体（产品ID、案例ID）和引用了哪些图片媒体ID，这需要完善的解析逻辑。

### ✅ 验收标准
- [x] 发布动作的 Web 接口集成测试成功：数据库生成新的 Version 和 Snapshot，并正确记录审计日志。
- [x] 回滚动作测试：能成功复制旧快照为新快照。
- [x] Portal 读取页面接口集成测试：确保能通过缓存返回数据，缓存 Miss 后能读取数据库并写入缓存。

## 阶段五：数据绑定能力与业务模块集成 ✅
**目标**：打通搭建器与现有强业务模块（如 `site`, `product`, `casecenter`），使页面能动态渲染业务数据。

### 📌 TODO（已完成）
- [x] 重构 `PageBindingResolutionService` 至 `application/portal` 层：根据 `BindingModel` 的类型 (STATIC, ENTITY, AGGREGATE) 分发到 11 个业务模块的 Portal 只读接口。
- [x] 实现 `PageRenderApplicationService`：在 Application 层编排缓存读取 → 模块快照加载 → 跨模块绑定解析装配 → 缓存写入的完整链路。
- [x] 实现过滤逻辑：`PortalPageRenderServiceImpl` 过滤 `visible = false` 区块；各模块 Portal 接口已过滤逻辑删除和隐藏数据。
- [x] 升级 `PageSchemaValidationService`：引入 `MediaAssetService` 实现媒体 ID 物理存在性校验；增加 `ALLOWED_BINDING_SOURCES` 绑定源白名单校验。
- [x] 安全加固：渲染输出前 `section.setBinding(null)` 擦除绑定配置元数据，防止内部绑定结构泄露。
- [x] 清理旧代码：删除 `modules/pagebuilder/service/impl/PageBindingResolutionServiceImpl`，移除 `PortalPageRenderServiceImpl` 对缓存和绑定解析的硬依赖。

### ℹ️ 实现说明
- **架构重构**：`PageBindingResolutionService` 从 `modules/pagebuilder` 迁移至 `application/portal`，消除跨模块循环依赖风险。
- **新增 Portal 端点**：
  - `GET /portal/api/page-builder/pages?routePath=xxx` — 已发布页面渲染（含数据绑定装配）
  - `GET /portal/api/page-builder/pages/{pageKey}/meta` — 已发布页面元数据（SEO + 布局）
- **渲染链路**：`PortalPageController` → `PageRenderApplicationService`（读缓存）→ `PortalPageRenderService`（加载快照）→ `PageBindingResolutionService`（装配绑定数据）→ 写缓存 → 返回。
- **安全**：`PageSchemaValidationService` 在草稿保存时校验 MEDIA 类型字段调用 `mediaAssetService.requireUsableImage()` 物理校验；绑定源 `source` 通过 `ALLOWED_BINDING_SOURCES` 白名单过滤。

### ⚠️ 难点分析
- **架构解耦**：`pagebuilder` 不能直接引用其他业务模块的 Mapper，必须通过 Application 层调用其他模块的 Service，避免循环依赖和跨界数据操作。
- **性能与 N+1 问题**：当一个页面包含多个实体绑定时，需要警惕循环查库问题。在 Application 层组装数据时，应尽量使用批量查询。

### ✅ 验收标准
- [x] Schema 中绑定了一个合法的产品 ID，Portal 渲染接口能够正确地在返回的 JSON 中包含该产品的详细公开数据。
- [x] 当绑定的产品被标记为 `visible = false` 或被逻辑删除后，Portal 接口不再返回该产品数据。
- [x] 绑定了未在白名单中的模块或不存在的 Media ID 时，草稿保存或发布应直接失败并返回明确错误。

**目标**：确保业务数据变更时能够自动刷新相关页面，保证高可用性、数据一致性及 100% 满足安全红线。

    - 实现 `PageCacheInvalidationService`：通过 `PageDependencyMapper.selectPageIdsByTarget` 查询依赖该实体的页面 ID，通过 `PageDefinitionMapper.selectRoutesByPageIds` 获取 pageKey/routePath，组装缓存 key 后调用 `PortalCacheSupport.invalidate`。
    - 新增 `PageRouteProjection` 投影 DTO，避免查询完整实体。

    - `PageCacheInvalidationServiceTest.java`：验证依赖页面缓存 Key 的生成与按需失效。
    - `EntityChangedEventListenerTest.java`：测试异步及异常吞噬逻辑的正确性。
    - `PageBindingResolutionServiceTest.java`：全面测试各业务数据绑定的正确解析。
    - `EntityChangedCacheInvalidationIntegrationTest.java`：端到端集成测试，显式注解 `@Commit` 确保事务真实提交，`@AfterEach` 清理数据库和 Redis。
- [x] 最后的安全审查：复核 SQL 注入风险（所有查询使用参数绑定）、越权漏洞（所有 Admin API 有 `@PreAuthorize`）、异常处理（全部经过 `GlobalExceptionHandler`）。

### ℹ️ 实现说明
- **事件定义**：`EntityChangedEvent` 继承 `ApplicationEvent`，包含 `targetModule`、`targetEntityType`、`targetEntityId` 三个字段，提供静态工厂方法 `of()`。
- **监听器**：`EntityChangedEventListener` 使用 `@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)` 确保事务提交后执行，`@Async` 异步执行不阻塞主线程，内部 try-catch 吞噬异常。
- **依赖查询**：`PageDependencyMapper.selectPageIdsByTarget` 使用 `SELECT DISTINCT page_id` 并支持 `target_entity_id = 'ALL'` 匹配整表绑定的数据源。
- **路由查询**：`PageDefinitionMapper.selectRoutesByPageIds` 使用 MyBatis `<script>` + `<foreach>` 动态 SQL，返回 `PageRouteProjection` 投影（仅 id/pageKey/routePath）。
- **缓存失效**：`PortalCacheSupport.invalidate` 复用现有 afterCommit + 延迟二次删除机制。
- **测试策略**：集成测试使用 `@Commit` 而非 `@Rollback`，确保 `AFTER_COMMIT` 事件监听器触发；`@AfterEach` 手动清理测试数据。

### ⚠️ 难点分析
- **联动失效的准确性与性能**：如何保证其他模块发出的事件能稳定被 `pagebuilder` 捕获且不阻塞主业务流程（考虑异步事件 `@Async` 与事务事件 `@TransactionalEventListener` 的结合）。
- **Docker 环境集成测试**：在真实的 MySQL 和 Redis 容器环境下，跑通端到端的“草稿修改 -> 发布 -> 依赖生成 -> 产品修改 -> 联动事件触发 -> 缓存失效”完整链路，需要精细化地设计测试数据和断言机制。

### ✅ 验收标准
- [x] 实体更新后，依赖该实体的页面能够自动且及时地更新缓存内容（集成测试断言）。
- [x] 所有的 Admin API 只有携带 Administrator 会话/Token 才能访问，普通请求必须被拦截。
- [x] 所有代码经过 Review，满足 `rules/review.md` 中的标准，无 P0/P1 级缺陷。
