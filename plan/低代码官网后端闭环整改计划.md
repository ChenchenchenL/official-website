# 低代码官网后端闭环整改计划

本文档记录对 `pagebuilder` 低代码官网后端接口与实现的闭环审查结果，以及上线前必须完成的整改项。它基于 [重构接口文档](../docs/重构接口文档.md)、[接口文档](../docs/接口文档.md)、[低代码后端重构执行顺序](低代码后端重构执行顺序.md) 和当前实现代码整理。

当前结论：页面定义、草稿、模板、发布、回滚、Portal 渲染与实体变更缓存联动的主链路已经具备；但预览安全、页面生命周期缓存失效和正式接口契约一致性仍存在阻塞问题。未完成本计划中的阻塞项前，不得将低代码页面作为生产 Portal 页面入口。

---

## 一、整改范围与原则

### 1. 整改目标

* 关闭未发布草稿和绑定配置泄露风险。
* 保证页面隐藏、删除、路由变更后 Portal 不继续命中旧缓存。
* 建立唯一、准确、可供前后端联调的 API 契约。
* 补齐发布并发、预览真实性和版本治理等生产使用缺口。

### 2. 执行原则

* 安全和公开内容可见性优先于编辑器便利性。
* 所有缓存失效必须在事务提交后执行，并保留延迟二次删除和 TTL 兜底。
* 所有 Admin 写操作必须要求 `Administrator` 角色并记录审计。
* 正式接口契约以 `docs/接口文档.md` 为准；`docs/重构接口文档.md` 仅可作为重构过程记录，不能与正式契约冲突。
* 每项整改必须同步更新接口文档、错误码说明、自动化测试和必要的架构说明。

## 二、阶段零：阻塞问题整改

本阶段完成前，禁止进行生产页面切流和匿名预览能力对外开放。

### P0-1 预览草稿泄露与预览契约不完整

**现状问题：**

* `GET /portal/api/page-builder/pages/preview` 位于匿名 Portal 路径下。
* 预览响应返回原始草稿 Schema，包含内部 `binding` 配置。
* 当前预览不执行正式页面的数据绑定、可见性过滤和绑定元数据清除。
* 完整预览 Token 被写入应用日志。
* Token 仅依赖随机 UUID 与 Redis TTL，没有操作者绑定、撤销能力或响应禁止缓存约束。

**整改任务：**

- [x] 将预览渲染改为复用 `PageRenderApplicationService` 的绑定解析和公开内容过滤能力。
- [x] 预览返回 `PortalPageVO` 或等价的“已装配渲染模型”，不得返回原始 `PageSchemaModel`。
- [x] 输出前清除全部 `binding` 元数据，不得暴露绑定源、过滤条件、实体 ID 列表或后台内部字段。
- [x] 预览 Token 日志仅记录哈希或前 6 位掩码，禁止记录完整 token。
- [x] 明确预览授权模型并实施其一：
  - [x] 仅管理员会话可访问预览，使用 Admin 路径或后台代理访问。
  - [x] 如必须支持外部分享，Token 必须只保存哈希、绑定 `pageId + draftVersion/schemaHash + createdBy`、支持撤销、短期有效且响应设置 `Cache-Control: no-store`。
- [x] 在预览 Token 使用次数、过期、撤销和权限不匹配时返回稳定业务错误码。
- [x] 预览查询不得进入正式 Portal 页面缓存。

**建议接口契约：**

* `POST /admin/api/page-builder/drafts/{pageId}/previews`
  创建当前草稿的受控预览，返回预览标识、到期时间和预览地址或受控访问参数。
* `GET /portal/api/page-builder/previews/{previewToken}`
  仅当外部分享明确需要时保留；返回已装配的预览渲染模型，响应必须 `no-store`。
* `DELETE /admin/api/page-builder/previews/{previewId}`
  可选但推荐，用于管理员主动撤销外部预览。

**验收标准：**

- [x] 未携带有效授权的请求无法读取草稿预览。
- [x] 预览响应中不存在 `binding`、内部模块名、查询配置或完整敏感实体数据。
- [x] 隐藏、逻辑删除或不可公开的绑定实体不出现在预览结果中。
- [x] 应用日志、审计日志和异常信息不包含完整预览 Token。
- [x] 预览过期、撤销和权限不匹配均有自动化测试。

### P0-2 页面生命周期操作未完整失效 Portal 缓存

**现状问题：**

* 发布与回滚会失效页面渲染和元数据缓存。
* 页面更新、隐藏、路由变更和删除没有明确的 afterCommit 缓存失效。
* 路由变更必须同时清理旧路由和新路由缓存，否则旧 URL 在 TTL 内仍可能返回内容。

**整改任务：**

- [x] 在 `PageDefinitionService` 中注入统一的页面缓存失效支撑，不允许 Controller 直接删除 Redis。
- [x] 页面名称、可见性、排序、状态、路由更新后，在 afterCommit 删除对应缓存。
- [x] 路由变更时，同时删除旧 `official:portal:page:{oldRoutePath}`、新 `official:portal:page:{newRoutePath}` 和 `official:portal:page-meta:{pageKey}`。
- [x] 页面隐藏、停用、逻辑删除时，删除当前路由和页面元数据缓存。
- [x] 继续使用现有延迟二次删除策略；缓存异常只记录日志，不回滚已经提交的数据库事务。
- [x] 为页面状态增加显式启用/停用用例，或将 `status` 纳入更新 DTO 并明确定义状态转换规则。

**建议接口契约：**

* `POST /admin/api/page-builder/pages/{id}/enable`
* `POST /admin/api/page-builder/pages/{id}/disable`

若不新增接口，则 `PUT /admin/api/page-builder/pages/{id}` 必须显式包含 `status`、版本号和状态转换校验规则。

**验收标准：**

- [x] 页面隐藏后，原路由与元数据接口不再返回缓存中的历史页面。
- [x] 页面删除后，原路由与元数据接口立即返回页面不存在。
- [x] 页面路由变更后，旧路由不可访问，新路由返回最新发布页面。
- [x] 页面启用/停用后，Portal 可见性与缓存表现一致。
- [x] 覆盖 afterCommit、事务回滚、延迟二次删除和多实例 Redis 场景的测试。

### P0-3 正式接口文档与实际实现冲突

**现状问题：**

* [接口文档](../docs/接口文档.md) 与 [重构接口文档](../docs/重构接口文档.md) 对发布、回滚和版本列表的路径、参数、分页、响应字段和错误码存在冲突。
* 当前实现与正式文档也不一致，例如回滚使用查询参数而非路径参数，版本列表未分页，发布仅返回 `PageVersionVO`。

**整改任务：**

- [x] 选择唯一契约来源：`docs/接口文档.md` 为正式来源。
- [x] 逐项以实现为准修订文档，或按正式文档调整 Controller、DTO、VO 和 Service；不得保留不一致状态。
- [x] 将 `docs/重构接口文档.md` 改为“实施记录/设计说明”，并在文首声明正式契约以 `docs/接口文档.md` 为准；或移除重复 API 明细。
- [x] 所有低代码接口补齐：路径、HTTP 方法、鉴权、CSRF 要求、请求字段、响应字段、错误码、缓存与审计说明、Controller 路径和方法名。
- [x] 修复错误码引用不一致，例如历史版本不存在的业务码必须统一。
- [x] 增加 API 契约测试，确保路径、请求校验、错误码和响应结构与正式文档一致。

**验收标准：**

- [x] 两份文档不再对同一接口提供互相冲突的契约。
- [x] 自动化 API 测试覆盖发布、回滚、版本列表和预览接口的真实路由与参数。
- [x] 前端可仅依赖 `docs/接口文档.md` 完成联调，不需要从代码反推接口行为。

## 三、阶段一：上线前重要缺口

本阶段应在首个低代码页面切入生产前完成。

### P1-1 发布与回滚并发控制

**问题：** 发布不携带预期草稿版本或 `schemaHash`，审阅完成后可能发布到其他管理员后来保存的草稿；回滚也未校验当前页面状态。

**整改任务：**

- [x] 发布请求增加 `draftVersion` 或 `schemaHash`，并在事务内与当前草稿比对。
- [x] 回滚请求增加 `expectedPageVersion` 或当前 ACTIVE 快照 ID，防止并发发布被无意覆盖。
- [x] 并发冲突统一返回 `COMMON_STATE_CONFLICT`，响应包含最新版本摘要或刷新提示。
- [x] 为重复发布/重复回滚定义幂等语义，至少保证不会生成不可解释的重复 ACTIVE 快照。

**验收标准：**

- [x] 两个管理员并发编辑、发布、回滚时，不会发布非预期 Schema。
- [x] 并发冲突和重复提交均有 API 与集成测试。

### P1-2 草稿恢复与编辑冲突处理

**问题：** 当前只有覆盖草稿，不支持放弃未发布修改、恢复到当前已发布版本或查看草稿修订历史。

**整改任务：**

- [x] 提供“草稿重置为当前 ACTIVE 快照”能力。
- [x] 明确保存冲突的前端恢复流程：返回最新草稿版本、hash 和更新时间。
- [x] 评估是否需要草稿自动保存历史；如不实现，文档中明确当前仅保存单一可编辑草稿。

**建议接口：**

* `POST /admin/api/page-builder/drafts/{pageId}/reset-to-published`

### P1-3 版本查询与回滚前审阅

**问题：** 当前版本列表返回完整 Schema 且未分页，不利于长期管理，也不能明确区分版本摘要、版本详情和快照状态。

**整改任务：**

- [x] 版本列表改为分页摘要返回，默认不返回完整 `schemaJson`。
- [x] 新增单版本详情接口，用于回滚前查看完整 Schema。
- [x] 可选增加两个版本的差异对比接口或由前端基于详情计算 diff。
- [x] 返回版本来源、创建人、创建时间、变更说明、hash 和关联发布快照状态。

**建议接口：**

* `GET /admin/api/page-builder/pages/{pageId}/versions?pageNo=1&pageSize=20`
* `GET /admin/api/page-builder/pages/{pageId}/versions/{versionId}`

### P1-4 页面状态、路由与导航治理

**问题：** 页面存在 `status`、`visible`、`routePath` 和 `sortOrder`，但启停管理、路由改名迁移、页面与导航菜单关系未形成完整治理约定。

**整改任务：**

- [x] 明确 `status`、`visible` 的语义和合法状态转换。
- [x] 为页面启用/停用提供接口或纳入页面更新接口。
- [x] 明确路由变更是否需要保留旧路由重定向；如需要，设计路由别名表或由网关处理。
- [x] 明确页面定义是否自动生成导航项；若否，前端导航必须继续以 `NavigationMenu` 为唯一来源。
- [x] 如页面需被搜索引擎发现，补充 Portal 路由清单或 sitemap 生成策略。

### P1-5 组件模板治理边界

**问题：** 模板接口仅提供已启用模板读取，当前不清楚模板是代码/迁移脚本管理，还是允许管理员动态修改。

**整改任务：**

- [x] 明确当前阶段组件模板为“代码与迁移脚本托管”，后台只读，不开放运营编辑。
- [x] 在模板变更流程中加入受影响页面扫描、兼容校验和迁移策略。
- [x] 模板下线前校验是否被 ACTIVE 快照或草稿引用，禁止破坏线上渲染。
- [x] 将上述策略补入架构设计和正式接口文档。

## 四、阶段二：运营与可观测性增强

以下项目不阻塞首个页面上线，但应在低代码页面规模扩大前排期完成。

- [ ] 发布、回滚、预览创建和预览访问接入指标：成功数、失败数、耗时、缓存失效失败数。
- [ ] 对绑定解析异常建立告警和可追踪日志，日志包含页面 ID、区块 ID、组件编码和 traceId，不记录完整 Schema 或敏感数据。
- [x] 提供管理员可用的页面依赖诊断查询，展示已发布快照引用的媒体、产品、案例和聚合数据源。
- [ ] 为长页面和多绑定页面做性能压测，重点控制绑定解析 N+1 查询和缓存重建风暴。
- [ ] 建立首页试点的灰度切换与一键回退方案，保留传统页面读取链路直到验收完成。

---

## 五、实施顺序

1. 完成 P0-1：先封堵匿名草稿泄露与完整 Token 日志问题。
2. 完成 P0-2：补齐页面更新、路由变更、隐藏和删除的 afterCommit 缓存失效。
3. 完成 P0-3：统一代码、正式接口文档和重构记录的契约。
4. 完成 P1-1：为发布和回滚增加并发控制与重复提交语义。
5. 完成 P1-2、P1-3：补齐草稿恢复和历史版本审阅能力。
6. 完成 P1-4、P1-5：明确页面状态/导航/模板治理边界。
7. 完成阶段二后，再扩大页面迁移范围。

---

## 六、总体验收门禁

只有同时满足以下条件，低代码官网后端才可判定为生产闭环：

- [x] 草稿、预览、发布、回滚、Portal 渲染路径均有明确且一致的 API 契约。
- [x] 未发布草稿、绑定配置和后台内部信息不会通过匿名接口泄露。
- [x] 页面定义与绑定实体任一变更后，相关 Portal 缓存均在提交后失效。
- [x] Portal 只返回启用、可见、未逻辑删除且存在 ACTIVE 快照的页面和公开绑定数据。
- [x] 发布、回滚和草稿保存具备并发冲突控制、审计记录和自动化测试。
- [x] 正式接口文档、错误码说明、架构说明和测试用例与真实实现同步。
- [x] Docker 环境完成端到端验证：草稿保存 -> 受控预览 -> 发布 -> Portal 渲染 -> 实体变更 -> 缓存失效 -> 回滚。

**验证记录（2026-07-22）：** 使用 Compose MySQL 8.4 与 Redis 7.4 运行 `EntityChangedCacheInvalidationIntegrationTest`。测试从空库执行 V1 至 V39 迁移，并完成草稿保存、管理员预览、双版本发布、回滚、Portal 渲染；随后在独立事务提交 `EntityChangedEvent(product/Product/1)`，异步 AFTER_COMMIT 监听器删除共享 Redis 中的页面渲染和 SEO 缓存 Key。应用容器镜像下载受 Docker Hub/CloudFront EOF 影响未完成，但 Maven 工具容器已在相同 Compose 网络、MySQL 和 Redis 上成功启动 Spring Boot 并完成上述验证。
