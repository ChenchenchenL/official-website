# 低代码官网后续实施计划

本文档用于承接低代码官网后端主链路闭环后的后续工作。正式 API 契约仍以 [接口文档](../docs/接口文档.md) 为准；已有安全、缓存和发布闭环整改记录见 [低代码官网后端闭环整改计划](低代码官网后端闭环整改计划.md)。

## 一、目标与边界

### 1. 目标

将当前“页面配置可保存、可预览、可发布、可按路由渲染”的后端能力，扩展为可供可视化编辑器稳定长期使用的配置平台，并补齐生产运行保障。

### 2. 后端职责

* 保存、校验、版本化页面 Schema。
* 按模板和绑定白名单返回页面渲染模型。
* 保证草稿、预览、发布、回滚、缓存失效、审计和权限控制的一致性。
* 为前端编辑器提供稳定的组件元数据、草稿、版本和诊断接口。

### 3. 非后端职责

* 画布拖拽、吸附线、缩放、选中态、属性面板、撤销重做交互和最终 CSS 渲染由前端实现。
* 网关流量切换、CDN、监控平台、告警平台和正式生产编排由运维/部署方案承接。
* 后端不解析或执行前端任意表达式、脚本、SQL 或查询条件。

### 4. 当前基线

当前后端已经具备：

* 页面定义、草稿保存、编辑锁、受控预览、发布、版本查询和回滚。
* 组件模板只读查询及模板引用诊断。
* 公开数据绑定、发布依赖记录和页面依赖诊断。
* Portal 路由渲染、可见性过滤、Redis 缓存、事务提交后失效和延迟二次删除。
* `Administrator` 鉴权、CSRF、审计记录和关键链路自动化测试。

当前不具备：响应式布局协议、可持久化草稿修订历史、运行指标/告警、性能压测、灰度流量切换和双应用实例部署演练。

## 二、阶段 P1：冻结 Schema 与布局协议

### P1-1 Schema 版本与兼容策略

**目的：** 防止前端编辑器、已保存草稿和已发布版本因字段演进失去兼容性。

- [ ] 在 `PageSchemaModel` 顶层增加必填 `schemaVersion`。
- [ ] 定义每个 `schemaVersion` 的兼容范围和废弃策略；读取历史草稿、版本和快照时必须可迁移或返回明确业务错误。
- [ ] 提供后端 Schema 升级器，仅允许从已知旧版本升级到当前版本；不得由前端自行篡改版本号绕过校验。
- [ ] 发布前校验草稿 Schema 版本，拒绝未知版本和不支持降级的版本。
- [ ] 为升级、未知版本、历史快照回滚各补充单元测试和集成测试。

**验收标准：**

* 当前版本编辑器能够读取历史发布版本并进行受控升级。
* 未知 `schemaVersion` 无法保存或发布，错误码和前端提示稳定。
* 升级后不改变既有组件的公开数据绑定语义。

### P1-2 布局与样式白名单

**目的：** 让前端的文本框拖拽、自定义摆放可以安全地保存和发布，同时避免非法布局或大 JSON 占用资源。

建议把可编辑配置分为 `layout`、`style`、`props` 三部分，而不是混入任意 JSON：

```json
{
  "layout": {
    "position": "absolute",
    "x": 120,
    "y": 48,
    "width": 360,
    "height": 72,
    "zIndex": 2
  },
  "style": {
    "textAlign": "left",
    "fontSize": 24,
    "fontWeight": 600,
    "color": "#111111"
  },
  "props": {
    "text": "示例标题"
  }
}
```

- [ ] 定义可用布局模式，例如 `flow`、`grid`、`absolute`；不支持的模式必须拒绝保存。
- [ ] 对 `x`、`y`、宽高、`zIndex`、边距、字号、透明度、颜色值等逐项做类型与范围校验。
- [ ] 限制区块数、组件数、嵌套深度、单个文本长度、图片/富文本引用数和整份 Schema 字节数。
- [ ] 限制 `zIndex` 范围，禁止通过无限层级干扰页面或浏览器性能。
- [ ] 颜色仅允许安全的预定义格式；URL、字体、动画名和 CSS 类名必须使用白名单，禁止写入任意 CSS、`javascript:` URL 或 HTML 事件属性。
- [ ] 在模板 Schema 中明确每个组件可编辑的 `layout`、`style`、`props` 字段、默认值和必填条件。
- [ ] 复用相同校验用于草稿保存、受控预览、发布和版本回滚，避免仅保存阶段校验。

**验收标准：**

* 前端可将拖拽后的文本框坐标、尺寸、层级、文本和样式写入草稿并完成预览、发布。
* 无效数值、超长 JSON、非法颜色、脚本 URL 和未声明属性不能进入草稿或发布快照。
* 已发布页面不会因单个组件的异常布局导致 Portal 整页接口失败。

### P1-3 响应式断点协议

**目的：** 避免自由拖拽只在某一个桌面分辨率可用。

- [ ] 定义首期断点：`desktop`、`tablet`、`mobile`；断点数、最小/最大宽度和优先级写入正式接口文档。
- [ ] 定义覆盖规则：基础 `layout/style` 为默认值，具体断点只覆盖声明的字段，缺失字段回退默认值。
- [ ] 明确是否允许不同断点切换布局模式；首期建议限制为同一组件内可控切换，防止组件树结构分叉。
- [ ] 对每个断点执行布局范围、组件数量和字段白名单校验。
- [ ] 在 Portal 渲染模型中保留经过校验的断点配置，前端按视口选择渲染，不返回后台绑定元数据。

**验收标准：**

* 一份页面配置可在三类视口下有确定的布局结果。
* 编辑器保存的断点覆盖不会删除未修改的基础配置。
* 草稿预览和已发布 Portal 返回的响应式配置语义一致。

### P1-4 组件模板与绑定参数扩展

**目的：** 让前端属性面板可由模板驱动生成，并将业务数据查询限制在明确白名单内。

- [ ] 盘点并确定首期组件集：容器/分栏、标题、正文、图片、按钮、富文本、轮播、产品列表、案例列表、表单。
- [ ] 对每个模板定义 `propsSchema`、`styleSchema`、`layoutSchema`、默认值、校验规则和兼容版本。
- [ ] 为绑定源定义允许的筛选参数，例如产品分类、指定 ID、展示数量、固定排序；不得允许前端提交任意 SQL、字段名或排序表达式。
- [ ] 对绑定参数的数量、ID 集合大小、排序枚举和最大返回条数做后端校验。
- [ ] 新增或修改模板继续采用代码与 Flyway 迁移托管，并通过模板引用诊断后再下线旧模板。

**验收标准：**

* 前端无需硬编码每个组件的属性字段，即可根据模板元数据生成属性面板。
* Portal 数据绑定只访问公开数据和声明允许的筛选范围。

## 三、阶段 P2：编辑体验与内容治理

### P2-1 草稿自动保存与修订历史

- [ ] 明确自动保存频率、节流策略、断网重试和保存冲突处理。
- [ ] 将当前“单一可编辑草稿”扩展为有限数量的草稿修订记录，或明确继续由前端本地承担撤销重做。
- [ ] 若增加服务端草稿历史，提供分页列表、单条详情、恢复指定修订的 Admin API，并保留乐观锁与编辑锁约束。
- [ ] 设置历史保留数量/保留期和清理任务，避免无限增长。

**验收标准：**

* 网络抖动和并发保存不会静默覆盖最新草稿。
* 若开启修订历史，管理员可审阅并恢复指定草稿修订，且恢复操作有审计记录。

### P2-2 版本差异与发布审阅

- [ ] 评估由前端基于两个版本详情计算 Diff，或新增后端受限的版本差异接口。
- [ ] 差异结果按组件路径、字段名、旧值、新值分类；敏感绑定配置不得返回给非管理员。
- [ ] 发布前审阅页至少展示版本号、Schema Hash、变更说明、组件数量、绑定源摘要和校验结果。

**验收标准：**

* 管理员在发布或回滚前可明确了解配置变化。
* 差异计算不会暴露 Token、内部查询条件或存储路径。

### P2-3 复用区块与模板建页

- [ ] 评估全局区块模型，例如页头、页尾、咨询区和 CTA 区；明确页面引用、版本冻结、级联发布和回滚语义。
- [ ] 提供从已发布页面或预设模板复制创建页面的能力。
- [ ] 复制操作必须重置页面路由、页面键等唯一字段，并记录来源页面/模板。

**验收标准：**

* 复制页面不会产生路由或页面键冲突。
* 共享区块更新的影响范围可诊断、可预览、可回滚。

### P2-4 审批、定时与灰度发布

**前置确认：** 当前需求尚未引入多角色审批流、定时发布或灰度状态。实施前必须先确定角色、状态机、审批记录、失败处理与回退权限。

- [ ] 如确认需要审批，设计草稿、待审、已拒绝、已发布等明确状态，避免复用 `visible/status` 承载审批语义。
- [ ] 如确认需要定时发布，设计时区、任务幂等、取消、失败重试、审计和缓存失效机制。
- [ ] 如确认需要灰度，后端仅提供可识别的页面版本或发布渠道；实际流量分配由网关/CDN/部署平台实现。

## 四、阶段 P3：生产可观测性与容量验证

### P3-1 指标、日志与告警

- [ ] 接入标准指标端点和监控采集，至少覆盖发布、回滚、预览创建、预览访问、绑定解析、缓存读写和缓存失效。
- [ ] 记录成功数、失败数、耗时分布、缓存失效失败数、绑定源类型和页面规模分布。
- [ ] 绑定解析异常日志必须包含页面 ID、区块 ID、组件编码、绑定源和 `traceId`；禁止记录完整 Schema、Token、隐私数据或内部查询条件。
- [ ] 根据指标在监控平台配置告警阈值，如连续发布失败、绑定异常激增、缓存失效失败、渲染耗时异常。

**验收标准：**

* 发生绑定或缓存故障时，可以使用 `traceId` 定位到页面和组件。
* 监控数据不泄露草稿配置、客户信息或预览 Token。

### P3-2 性能压测与查询治理

- [ ] 构造长页面、多区块、多绑定源和热点页面的压测数据。
- [ ] 测量草稿保存、预览渲染、首次 Portal 渲染、缓存命中渲染、发布和实体变更后的缓存重建。
- [ ] 检查绑定解析中的 N+1 查询；需要时按绑定源批量查询或预聚合。
- [ ] 验证 Redis 失效后高并发回源时的缓存击穿保护与数据库连接池容量。
- [ ] 将目标并发、P95/P99 延迟、错误率和资源上限写入部署验收标准。

### P3-3 多实例部署演练

- [ ] 以两个以上 Spring Boot 应用实例连接同一个 MySQL 和 Redis。
- [ ] 验证任一实例完成发布、回滚、页面状态调整或绑定实体变更后，其他实例不会继续命中旧缓存。
- [ ] 验证编辑锁、预览 Token、会话策略和异步缓存失效在多实例下的行为；会话如不共享，必须由负载均衡粘性会话或统一会话存储解决。
- [ ] 记录部署参数、Redis Key 前缀、健康检查、扩缩容和故障恢复步骤。

## 五、阶段 P4：首页灰度与一键回退

该阶段依赖网关、Nginx、CDN 或部署平台，不应由页面编辑 API 直接承担流量调度。

- [ ] 保留传统首页读取链路，并明确低代码首页对应的路由/版本标识。
- [ ] 在网关或边缘层设计按用户、Cookie、请求头或百分比的灰度策略。
- [ ] 提供运营可执行的切换清单：预检查、灰度比例调整、观测窗口、全量切换和故障回退。
- [ ] 一键回退应恢复传统首页流量或稳定的已发布页面版本；回退权限仅限管理员，操作必须审计。
- [ ] 灰度期间持续观测 Portal 错误率、渲染耗时、缓存命中率和用户表单转化等业务指标。

**验收标准：**

* 灰度流量可逐步扩大且不影响传统链路。
* 低代码首页出现故障时，可在既定恢复目标时间内切回稳定版本。

## 六、推荐实施顺序

1. P1-1、P1-2、P1-3：先冻结 Schema、布局和响应式协议，再开始前端画布研发。
2. P1-4：补齐首期组件模板及受限绑定参数，让属性面板和数据源联调有明确边界。
3. 前端实现画布、组件面板、属性面板和 Portal 渲染器，直接对接现有草稿、预览、发布和版本接口。
4. P2-1、P2-2：根据编辑器实际使用情况补草稿修订历史和版本审阅。
5. P3-1、P3-2、P3-3：上线前完成可观测性、压测和共享 Redis 多实例验证。
6. P4：首页试点灰度，稳定后再迁移更多页面。

## 七、每项后端变更的交付门禁

每个新增或修改的低代码后端功能必须同时满足：

* 更新 [接口文档](../docs/接口文档.md)，包含路径、方法、鉴权、CSRF、请求字段、响应字段、错误码、Controller 路径与方法名。
* 管理端写接口要求 `Administrator` 角色，关键内容操作记录审计日志。
* Portal 读取不返回草稿、隐藏、逻辑删除或非公开绑定数据。
* Schema、模板、绑定参数和布局配置有服务端白名单校验，禁止把客户端输入拼接为 SQL、脚本或任意查询。
* 缓存失效在事务提交后执行，并验证 TTL 和延迟二次删除行为。
* 新增业务规则覆盖成功、失败、边界、权限和并发场景测试；涉及 MySQL/Redis 的关键链路使用 Docker 验证。
