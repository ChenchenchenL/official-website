# 官网可视化编辑模式：后端接口补充执行顺序

> 文档状态：待实施  
> 版本：v1.0  
> 更新日期：2026-07-14  
> 适用范围：页面构建器、产品、案例、行业方案的 Admin 编辑接口与 Portal 公开读取接口。  
> 上位契约：[官网可视化编辑模式后端接口缺口清单](./官网可视化编辑模式后端接口缺口清单.md)。

## 1. 文档定位

本文件只定义**后端接口补充工作的实施顺序**、阶段依赖、每阶段交付物、测试和退出条件；不替代接口契约。

以下文档的优先级保持不变：

1. 已实现接口以 `docs/接口文档.md` 与 `docs/重构接口文档.md` 为准。
2. 新增与收紧接口以 `plan/官网可视化编辑模式后端接口缺口清单.md` 为准。
3. 本文用于安排实现，不得自行改变路径、请求字段、Header、响应结构或错误语义。

所有新增、变更或移除的接口，必须在完成同一个阶段时同步更新 `docs/接口文档.md`；页面构建器相关接口还必须同步更新 `docs/重构接口文档.md`。错误码、数据库变更和缓存行为分别同步至 `docs/错误码说明.md`、`docs/数据库设计说明.md` 与对应迁移脚本。

## 2. 现状与补充范围

截至本文日期，页面构建器已具备组件模板、页面定义、页面草稿、受控预览、发布、回滚、版本历史、Portal 页面渲染与 SEO 元数据读取的基础接口。产品和案例已有传统 Admin CRUD 与 Portal 详情读取；行业方案已有传统 Admin CRUD 与 Portal 列表读取。

本次接口补充不以传统 CRUD 作为编辑器后端。编辑器必须使用下列完整生命周期：

```text
创建详情草稿 -> 获取编辑锁 -> 保存草稿 -> 创建预览 -> 访问/撤销预览
-> 发布 -> 查询版本 -> 回滚 -> 下线
```

补充范围如下：

| 范围 | 必须补充或收紧的能力 |
| --- | --- |
| 共性基础设施 | 编辑锁、锁 Token 校验、锁过期、强制解锁审计、统一错误码与测试支撑。 |
| 页面构建器 | 为现有草稿、预览、发布、回滚接口接入锁与版本安全约束，并补齐页面锁接口。 |
| 产品/案例/行业方案 | 草稿、预览、发布快照、版本、回滚、下线、编辑锁及相应 Admin 接口。 |
| Portal | 行业方案详情接口；三类详情均只读取已发布、可见、未逻辑删除的快照。 |
| 治理联动 | 引用关系校验、媒体引用保护、审计、提交后缓存失效与延迟二次删除。 |

## 3. 不可跨越的实施原则

1. **先数据与安全边界，后写接口。** 未落地草稿、快照、版本、锁或引用数据模型前，不得暴露对应写接口。
2. **先锁能力，后编辑器写操作。** 保存、预览创建和撤销、发布、回滚、下线、删除等受保护操作，必须在锁校验可用后再接入。
3. **Portal 永远最后接入新详情数据。** Portal 接口只能读取已发布快照，绝不能为联调读取草稿或传统 Admin 数据。
4. **每阶段闭环。** 每个阶段都必须完成迁移、DTO/VO、Service、Controller、授权、审计、缓存、自动化测试和文档同步；不得将安全、审计、缓存或测试留到后续阶段。
5. **旧 CRUD 不承载新流程。** 不改变传统 CRUD 的既有语义；新编辑器只调用缺口清单规定的专用路径。
6. **后续阶段不得绕过前置退出条件。** 任一前置阶段未通过测试或文档未同步时，后续阶段不得开始联调或发布。

## 4. 总体执行顺序

```text
S0 契约冻结与基线回归
  -> S1 数据模型、错误码与共性支撑
  -> S2 独占编辑锁与受保护写操作门禁
  -> S3 页面构建器安全收紧与锁接入
  -> S4 三类详情的草稿/预览/发布通用能力
  -> S5 产品详情编辑接口闭环
  -> S6 案例详情编辑接口闭环
  -> S7 行业方案详情编辑与 Portal 详情闭环
  -> S8 引用校验、媒体保护与缓存联动收口
  -> S9 全链路回归、文档核验与上线准入
```

`S5`、`S6` 与 `S7` 在 `S4` 通过后可由不同开发人员并行实施，但必须使用 `S4` 冻结的 DTO/VO、锁、预览、版本、审计和缓存扩展点；三者均完成后才能进入 `S8`。

## 5. 阶段 S0：契约冻结与基线回归

### 5.1 目的

建立实现前的真实接口基线，避免以计划文档误覆盖已实现接口，或让前端继续使用不符合新流程的传统 CRUD。

### 5.2 工作项

1. 对照缺口清单逐项标记：已实现、需收紧、待新增、明确不属于编辑器调用范围。
2. 核对现有 Controller 路径、方法和返回 VO；对于已实现页面构建器接口，保留现有路径并只追加契约规定的安全约束。
3. 明确传统产品、案例、行业方案 CRUD 的兼容策略：保留给既有后台页面，不作为可视化编辑器接口。
4. 固定本期 `X-CSRF-Token`、`X-Editor-Lock-Token`、`version`、错误响应与 `traceId` 的统一行为。
5. 记录当前相关自动化测试的基线结果，防止后续安全收紧造成未识别的兼容性回归。

### 5.3 交付物

- 接口差异核对记录（可作为本文件的实施勾选附录或单独 Review 记录）。
- 冻结后的错误码清单和数据模型变更清单。
- 受影响前端调用方清单，明确其应切换到的专用编辑接口。

### 5.4 退出条件

- 已确认页面构建器现有接口的实际 Controller 与方法位置。
- 已确认三类详情的传统 CRUD 不被新编辑器调用。
- 已确认没有 API 路径、Header 或请求体字段仍存在歧义。

## 6. 阶段 S1：数据模型、错误码与共性支撑

### 6.1 目标

先建立所有后续接口共同依赖的持久化模型和稳定错误语义，确保业务接口不以临时字段、内存状态或非事务数据完成核心流程。

### 6.2 实施顺序

1. 定义资源类型枚举：`PAGE`、`PRODUCT`、`CASE`、`INDUSTRY_SOLUTION`。
2. 新增编辑锁表或等价可靠持久化模型，最少包含资源类型、资源 ID、持有人、持有人脱敏展示名、获锁时间、最后心跳时间、到期时间、锁 Token 摘要/安全校验材料、版本及逻辑删除字段。
3. 新增产品、案例、行业方案的详情草稿、发布快照和版本表；表结构必须能保存 `draft`、哈希、版本号、发布人、发布时间、变更摘要与回滚来源。
4. 扩展或统一内容引用模型，使其可表达页面区块、详情草稿/已发布快照与媒体、产品、案例、行业方案之间的正反向引用。
5. 为锁冲突、锁过期、锁归属不匹配、版本冲突、预览失效、引用冲突、发布校验失败和无强制解锁权限登记稳定错误码与 HTTP 状态。
6. 定义共性 DTO/VO 与 Service 扩展点：锁状态、草稿、预览创建结果、版本、下线请求和强制解锁请求；API 不得返回 Entity。
7. 编写迁移脚本的空库验证与历史数据兼容说明，涉及唯一约束时使用 `deleted_marker` 规则。

### 6.3 必须验证

- MySQL 环境可从空库执行所有新增迁移。
- 锁记录对同一 `resourceType + resourceId` 的有效锁具有数据库层最终约束或等价并发保护。
- 草稿、快照、版本与引用记录均有清晰的逻辑删除、版本和审计字段策略。
- 新增错误码的数值、名称、HTTP 状态和前端处理语义已文档化。

### 6.4 退出条件

- 所有后续接口依赖的表、枚举、DTO/VO 和错误码已合入并通过数据访问测试。
- 无接口依赖未迁移的临时表、自由 JSON 约定或不受控内存锁。

## 7. 阶段 S2：独占编辑锁与受保护写操作门禁

### 7.1 目标

实现页面、产品、案例、行业方案共用的独占编辑锁能力，并让所有后续写接口可复用同一套锁校验与审计机制。

### 7.2 接口交付顺序

按资源嵌套路径实现，页面接口先完成并作为其他资源的参考实现：

1. `GET /admin/api/page-builder/pages/{pageId}/lock`
2. `POST /admin/api/page-builder/pages/{pageId}/lock`
3. `POST /admin/api/page-builder/pages/{pageId}/lock/heartbeat`
4. `DELETE /admin/api/page-builder/pages/{pageId}/lock`
5. `POST /admin/api/page-builder/pages/{pageId}/lock/force-release`
6. 按相同语义实现产品、案例、行业方案的五组锁接口。

### 7.3 固定行为

- 所有接口要求有效管理员 Session；创建、心跳、主动释放和强制释放均按 Cookie Session 的 CSRF 规则处理。
- 强制释放只校验管理员、CSRF、强制解锁权限和原因，不接收 `X-Editor-Lock-Token`；必须写审计日志及变更原因。
- 成功创建锁时才返回明文 `lockToken`；查询锁状态、冲突响应和日志不得返回或记录完整 Token。
- 第二名管理员获取有效锁时返回 `409`，携带脱敏持有人显示名、到期时间与是否允许强制解锁。
- 锁已过期、锁归属不匹配、无强制解锁权限分别返回缺口清单规定的 `409` 或 `403`。
- Service 层在事务中进行锁占用、续期、释放和过期抢占的并发控制；Controller 不得直接访问 Mapper。

### 7.4 必须测试

- 未登录、非管理员、CSRF 缺失/错误、正常加锁、重复加锁、两管理员抢锁、过期后重新加锁、心跳续期、主动释放、强制释放、无权强制释放。
- Token 不出现在响应日志、异常日志、审计快照或 URL 中。
- 并发请求不会让同一资源产生两把有效锁。

### 7.5 退出条件

- 四类资源的锁接口路径、状态码和响应字段完全一致。
- 锁 Service 可被后续草稿、预览、发布、回滚、下线和删除操作直接复用。
- 锁相关错误码、审计字段、API 文档与自动化测试齐全。

## 8. 阶段 S3：页面构建器安全收紧与锁接入

### 8.1 目标

在不变更现有页面构建器公开路径的前提下，为其草稿、预览与发布链路接入锁、版本、Schema 校验、预览最小权限和缓存治理。

### 8.2 实施顺序

1. 调整保存页面草稿接口 `PUT /admin/api/page-builder/drafts/{pageId}`：要求 `X-CSRF-Token`、`X-Editor-Lock-Token` 与请求体 `version`，校验受控组件、合法插槽、媒体、链接协议、富文本和页面依赖。
2. 调整创建页面预览接口 `POST /admin/api/page-builder/drafts/{pageId}/previews`：要求 CSRF、锁 Token 与 `schemaHash`，仅允许当前锁持有人创建。
3. 调整撤销页面预览接口 `DELETE /admin/api/page-builder/previews/{previewToken}`：要求 CSRF、锁 Token，仅允许创建管理员撤销。
4. 调整 Portal 预览读取接口：校验有效管理员 Session、Token 创建管理员、页面资源与草稿哈希；响应固定添加 `Cache-Control: no-store` 与 `Referrer-Policy: no-referrer`。
5. 调整发布与回滚接口：要求 CSRF、锁 Token、`version` 和变更说明；回滚必须生成新的草稿与发布版本，禁止覆盖历史版本。
6. 接入事务提交后的页面渲染、SEO、关联列表与引用页面缓存失效，以及延迟二次删除。

### 8.3 必须测试

- 无锁、错误锁、过期锁、版本冲突、非法组件、非法插槽、无效媒体、危险链接、未净化富文本。
- 普通 Portal 用户、未登录用户、非创建管理员、过期 Token 和已撤销 Token 无法读取预览。
- 发布和回滚成功后只在事务提交后失效缓存；事务回滚时不得误删或回填脏缓存。
- Portal 页面渲染和 SEO 接口不返回草稿、锁、预览 Token、版本或审计数据。

### 8.4 退出条件

- 页面编辑完整链路只能由持锁管理员完成。
- 旧的页面构建器公开渲染和 SEO 接口仍保持已发布快照读取语义。
- `docs/重构接口文档.md`、`docs/接口文档.md` 和错误码说明已同步。

## 9. 阶段 S4：三类详情的通用草稿、预览与发布能力

### 9.1 目标

抽取产品、案例、行业方案共享的详情编辑生命周期能力，避免三个模块分别实现不一致的锁、版本、预览和发布语义。

### 9.2 工作项

1. 定义详情草稿和发布快照的抽象契约：`draft`、`draftHash`、`version`、`editorSessionRemark`、`changeSummary`、`updatedAt`。
2. 定义详情预览 Token 的资源绑定、创建管理员绑定、草稿哈希绑定、有效期、撤销和脱敏日志策略。
3. 定义详情版本列表和回滚语义：仅以已发布版本为来源，回滚生成新的草稿与发布版本。
4. 定义详情下线语义：校验乐观锁、引用关系、审计和缓存失效；若已发布页面存在强引用则返回 `409`。
5. 定义共享发布校验扩展点：必填字段、富文本白名单、媒体可用性、关联实体可用性、SEO 与排序字段校验。
6. 定义共享缓存失效事件，事件仅在事务提交后由监听器处理；缓存失败必须记录可重试任务、审计/监控事件，并由 TTL 最终收敛。

### 9.3 退出条件

- 三个模块不复制安全校验、预览 Token、版本、审计和缓存失效的核心逻辑。
- 共享抽象不通过跨模块 Mapper 访问其他模块数据；引用、媒体和缓存能力通过 Service 或 Application Service 协作。
- 共享能力具备单元测试和服务集成测试，可供 S5 至 S7 并行复用。

## 10. 阶段 S5：产品详情编辑接口闭环

### 10.1 按顺序交付接口

1. `POST /admin/api/products/drafts`
2. `GET /admin/api/products/{id}/draft`
3. `PUT /admin/api/products/{id}/draft`
4. `POST /admin/api/products/{id}/draft/previews`
5. `GET /admin/api/products/{id}/previews/{previewToken}`
6. `DELETE /admin/api/products/{id}/previews/{previewToken}`
7. `POST /admin/api/products/{id}/publish`
8. `GET /admin/api/products/{id}/versions`
9. `POST /admin/api/products/{id}/rollback/{versionId}`
10. `POST /admin/api/products/{id}/offline`

产品锁接口必须已在 S2 完成；除创建草稿、读取草稿、读取版本和访问预览外，接口 Header 与 `version` 规则严格遵循缺口清单。

### 10.2 业务验证

- 发布快照仅在产品草稿校验、媒体绑定、关联案例/行业方案和 SEO 校验通过时生成。
- Portal 产品详情只读取已发布、可见、未逻辑删除的发布快照。
- 下线、删除、回滚前检查已发布页面强引用；冲突时不改变产品状态。
- 发布、回滚、下线写入审计日志，并在提交后失效产品详情、产品列表、关联页面与 SEO 缓存。

### 10.3 退出条件

- 产品编辑器全生命周期接口、权限、锁、版本、预览和缓存测试通过。
- 传统产品 CRUD 未被修改为草稿发布语义，编辑器前端已有明确的专用接口调用约束。

## 11. 阶段 S6：案例详情编辑接口闭环

按 S5 的接口顺序和通用约束，实现案例资源对应路径：

```text
/admin/api/cases/drafts
/admin/api/cases/{id}/draft
/admin/api/cases/{id}/draft/previews
/admin/api/cases/{id}/previews/{previewToken}
/admin/api/cases/{id}/publish
/admin/api/cases/{id}/versions
/admin/api/cases/{id}/rollback/{versionId}
/admin/api/cases/{id}/offline
```

额外验证案例富文本、封面/图集、关联产品、推荐案例、行业信息和 SEO 的草稿校验、媒体引用维护及发布快照隔离。

### 11.1 退出条件

- 案例的锁、草稿、预览、发布、版本、回滚、下线与产品接口具有一致的安全与错误语义。
- Portal 案例详情仍只输出已发布、可见、未逻辑删除数据。
- 引用冲突、媒体不可用、版本冲突、锁冲突和越权场景均有自动化覆盖。

## 12. 阶段 S7：行业方案详情编辑与 Portal 详情闭环

### 12.1 按顺序交付 Admin 接口

1. `POST /admin/api/industry-solutions/drafts`
2. `GET /admin/api/industry-solutions/{id}/draft`
3. `PUT /admin/api/industry-solutions/{id}/draft`
4. `POST /admin/api/industry-solutions/{id}/draft/previews`
5. `GET /admin/api/industry-solutions/{id}/previews/{previewToken}`
6. `DELETE /admin/api/industry-solutions/{id}/previews/{previewToken}`
7. `POST /admin/api/industry-solutions/{id}/publish`
8. `GET /admin/api/industry-solutions/{id}/versions`
9. `POST /admin/api/industry-solutions/{id}/rollback/{versionId}`
10. `POST /admin/api/industry-solutions/{id}/offline`

### 12.2 新增 Portal 接口

在 Admin 发布链路和发布快照查询完成后，再新增：

```text
GET /portal/api/industry-solutions/{id}
```

该接口只读取已发布、可见、未逻辑删除的行业方案详情；不得返回草稿、预览、锁、版本、审计、内部存储路径或 Admin 字段。不存在、隐藏、下线或逻辑删除的资源按 Portal 资源不可见的统一错误语义处理，不能泄露后台状态。

### 12.3 退出条件

- 行业方案完成与产品、案例对等的独立详情编辑生命周期。
- Portal 行业方案列表与详情都不会读取传统编辑草稿或未发布内容。
- 公开详情、预览详情、Admin 草稿详情三种读取边界有明确且独立的 API 测试。

## 13. 阶段 S8：引用校验、媒体保护与缓存联动收口

### 13.1 目标

在所有资源接口已可独立运行后，按真实跨资源依赖收口引用完整性和缓存一致性，避免页面或详情发布后出现悬挂引用、误删媒体或多实例缓存不一致。

### 13.2 工作项

1. 发布页面时校验所引用的详情实体已发布且可见；保存草稿可保留待修复引用，但发布必须拒绝无效强引用。
2. 发布详情时校验关联实体和媒体可用，并更新页面、详情与媒体引用记录。
3. 下线、删除或回滚详情时，若已发布页面存在强引用，返回 `409` 且不改变原状态。
4. 删除媒体时，若存在已发布快照引用，返回 `409` 且不执行删除；富文本引用以提交的媒体引用清单为准，不依赖 HTML 扫描。
5. 为页面、页面 SEO、首页聚合、产品/案例/行业方案列表与详情、导航和站点配置建立明确的失效依赖表。
6. 对所有发布、回滚、下线、删除路径核对：数据库事务提交后立即删除缓存、安排延迟二次删除、缓存失败记录审计/监控/重试任务、TTL 作为最终兜底。

### 13.3 必须测试

- 页面发布引用未发布详情被拒绝。
- 已发布页面强引用的详情无法下线或删除。
- 破坏强引用的详情回滚被拒绝。
- 已发布快照引用的媒体无法删除。
- 事务回滚、缓存删除失败、延迟二次删除和 TTL 兜底均具有可重复验证。
- 多实例场景使用共享 Redis 或等价共享缓存，不允许使用本地进程缓存作为生产一致性方案。

### 13.4 退出条件

- 所有引用冲突均有明确错误码、审计记录和前端可处理的响应。
- 所有影响已发布数据的操作均已接入提交后缓存失效，且不存在事务内直接删缓存的实现。

## 14. 阶段 S9：全链路回归、文档核验与上线准入

### 14.1 回归顺序

1. 运行锁、草稿、预览、发布、回滚、下线、引用和缓存的 Service 单元测试与集成测试。
2. 在 Docker MySQL 环境验证所有迁移可从空库执行，并验证包含历史产品、案例和行业方案数据的升级路径。
3. 运行 Admin API 回归：未登录、非管理员、CSRF 失败、锁冲突、锁过期、错误 Token、版本冲突、参数校验、引用冲突和成功路径。
4. 运行 Portal API 回归：只返回已发布可见内容；不可读取草稿、预览、锁、版本、审计或敏感内部字段。
5. 在至少两个应用实例配合共享缓存的环境验证发布、回滚、下线后的缓存收敛。
6. 逐项核对正式接口文档、重构接口文档、错误码文档、数据库设计、迁移脚本与 Controller 实现位置。

### 14.2 上线阻断项

以下任一项未完成，不得开放可视化编辑接口：

- 任意 Admin 写接口缺少管理员授权、CSRF、锁或版本校验。
- 任意 Portal 接口能读取草稿、预览、隐藏、下线或逻辑删除内容。
- 发布、回滚、下线或删除缺少事务、审计、引用校验或提交后缓存失效。
- 预览 Token 可被普通用户、非创建管理员、过期 Token 或日志读取复用。
- 迁移未在 Docker MySQL 环境验证，或无法说明历史数据兼容与回滚风险。
- 新接口、错误码、数据表、缓存影响或 Controller 方法未同步至正式文档。
