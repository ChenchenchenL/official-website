# 功能点 15：核心价值观卡片管理详细设计说明书

## 1. 文档目标

本文档用于落实“功能点 15：核心价值观卡片管理”的后端详细设计，覆盖数据模型、接口契约、缓存策略、审计要求、权限边界、测试范围与落地实现建议，供研发、测试、评审统一执行。

该功能对应官网“关于我们”页面中的“核心价值观”卡片区域，目标是将原有可能固定数量的展示内容改造为可后台动态维护的内容集合。

---

## 2. 设计范围

### 2.1 本期包含

本期仅包含“核心价值观卡片”管理，单张卡片字段如下：

- `icon`：图标资源
- `title`：大字标题
- `subtitle`：副标语
- `description`：详情描述
- `visible`：前台显示状态
- `sortOrder`：前台排序值

### 2.2 本期不包含

以下内容不属于本功能点，需独立规划：

- “我们的承诺”主体长文案
- 承诺标签管理
- 多语言字段
- 卡片分组、分类
- 卡片样式模板配置

---

## 3. 业务目标与设计原则

### 3.1 业务目标

- 解除固定数量限制，支持动态 N 张价值观卡片
- 后台支持分页查询、新增、编辑、删除、显示/隐藏、重排
- 前台 Portal 提供只读公开接口，稳定返回全部可见卡片
- 接入 Redis 缓存，保证高频访问性能
- 关键写操作具备审计日志与并发保护

### 3.2 设计原则

- 与现有 `TimelineEvent`、`AiCard`、`StrengthMetric` 模块风格保持一致
- 管理端写操作全部在 Service 层承接事务、审计和缓存失效
- Portal 只返回可展示字段，不暴露实体字段
- 媒体资源必须通过媒体模块统一校验
- 缓存删除必须在事务提交后执行，并带延迟二次删除

## 5. 业务规则

### 5.1 字段规则

#### 5.1.1 图标 `iconMediaId`

- 必填
- 类型：`Long`
- 必须引用媒体库中存在、可用、未逻辑删除的图片类资源
- 不允许直接传 URL 落库

#### 5.1.2 大字标题 `title`

- 必填
- 去除首尾空格
- 最大长度：`32`
- 活跃数据中全局唯一

说明：
当前页面中的大字标题本质上代表一条独立文化关键词，例如“同事”“同仁”“同享”，允许重复没有业务意义，反而会增加后台维护歧义，因此按活跃数据全局唯一处理。

#### 5.1.3 副标语 `subtitle`

- 必填
- 去除首尾空格
- 最大长度：`128`

#### 5.1.4 详情描述 `description`

- 必填
- 去除首尾空格
- 最大长度：`512`

#### 5.1.5 显示状态 `visible`

- 必填
- 布尔值
- 默认值：`true`

#### 5.1.6 排序值 `sortOrder`

- 系统维护
- 同一模块内越小越靠前
- 默认新增到末尾
- 排序步长建议使用 `10`

#### 5.1.7 版本号 `version`

- 编辑和删除必须传入
- 默认值：`0`
- 用于乐观锁校验

### 5.2 生命周期规则

- 新增后默认进入活跃数据集
- 删除采用逻辑删除
- 删除后允许重建同名卡片
- 隐藏卡片保留在后台列表中，但 Portal 不返回

### 5.3 排序规则

- Admin 默认排序：`sort_order ASC, id ASC`
- Portal 固定排序：`sort_order ASC, id ASC`
- 重排接口必须提交全部活跃卡片 ID
- 排序重写后按 `10, 20, 30...` 重新分配

### 5.4 并发规则

以下操作必须校验版本号：

- 编辑
- 删除

以下操作必须校验列表完整性：

- 重排

### 5.5 媒体绑定规则

- 卡片依赖媒体资源，但不持有媒体二进制内容
- 删除卡片时仅解除业务引用，不删除媒体文件
- 如后续媒体模块支持“被引用计数/引用清单”，本模块应补充绑定与解绑

### 5.6 权限规则

- 所有 `/admin/api/value-cards/**` 接口必须登录
- 所有 Admin 接口必须校验 `Administrator` 角色
- Portal 接口匿名开放

### 5.7 审计规则

以下操作必须写入 `sys_audit_log`：

- `CREATE_VALUE_CARD`
- `UPDATE_VALUE_CARD`
- `DELETE_VALUE_CARD`
- `REORDER_VALUE_CARD`

审计字段至少包括：

- 操作人 ID
- 操作动作
- 模块：`SITE`
- 对象类型：`VALUE_CARD`
- 对象 ID
- 请求来源 IP
- User-Agent
- 操作时间
- `traceId`
- 变更前快照
- 变更后快照

## 6. 数据库设计

### 6.1 表名

`cms_value_card`

### 6.2 表结构

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | BIGINT | PK | - | 主键 |
| icon_media_id | BIGINT | NOT NULL | - | 图标媒体 ID |
| title | VARCHAR(32) | NOT NULL | - | 大字标题 |
| subtitle | VARCHAR(128) | NOT NULL | - | 副标语 |
| description | VARCHAR(512) | NOT NULL | - | 详情描述 |
| visible | TINYINT(1) | NOT NULL | 1 | 是否显示 |
| sort_order | INT | NOT NULL | 0 | 排序值 |
| version | INT | NOT NULL | 0 | 乐观锁版本 |
| created_at | DATETIME | NOT NULL | - | 创建时间 |
| updated_at | DATETIME | NOT NULL | - | 更新时间 |
| created_by | BIGINT | NOT NULL | - | 创建人 |
| updated_by | BIGINT | NOT NULL | - | 更新人 |
| deleted_marker | BIGINT | NOT NULL | 0 | 逻辑删除标记 |

### 6.3 索引设计

唯一索引：

- `uk_cms_value_card_title_del(title, deleted_marker)`

普通索引：

- `idx_cms_value_card_visible_del_sort(visible, deleted_marker, sort_order, id)`

说明：

- 唯一索引保证活跃标题唯一，删除后可重建
- Portal 查询高频走 `visible + deleted_marker + sort_order` 组合索引

### 6.4 建表 SQL 草案

```sql
CREATE TABLE cms_value_card (
    id BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键 ID',
    icon_media_id BIGINT NOT NULL COMMENT '图标媒体 ID',
    title VARCHAR(32) NOT NULL COMMENT '大字标题',
    subtitle VARCHAR(128) NOT NULL COMMENT '副标语',
    description VARCHAR(512) NOT NULL COMMENT '详情描述',
    visible TINYINT(1) NOT NULL DEFAULT 1 COMMENT '是否显示：1-显示，0-隐藏',
    sort_order INT NOT NULL DEFAULT 0 COMMENT '排序值，越小越靠前',
    version INT NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
    created_at DATETIME NOT NULL COMMENT '创建时间',
    updated_at DATETIME NOT NULL COMMENT '更新时间',
    created_by BIGINT NOT NULL COMMENT '创建人 ID',
    updated_by BIGINT NOT NULL COMMENT '更新人 ID',
    deleted_marker BIGINT NOT NULL DEFAULT 0 COMMENT '逻辑删除标记：0-未删除，非0-已删除',
    PRIMARY KEY (id),
    UNIQUE KEY uk_cms_value_card_title_del (title, deleted_marker),
    KEY idx_cms_value_card_visible_del_sort (visible, deleted_marker, sort_order, id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='核心价值观卡片表';
```

## 9. Service 关键流程设计

### 9.1 新增流程

1. 校验 DTO 基础字段
2. 清洗字符串字段
3. 校验图标媒体资源
4. 校验标题唯一
5. 查询当前最大排序值并分配新 `sortOrder`
6. 执行插入
7. 写审计日志
8. 事务提交后失效 Portal 缓存

### 9.2 编辑流程

1. 查询活跃数据
2. 校验 `version`
3. 保存变更前快照
4. 校验图标媒体资源
5. 校验标题唯一
6. 执行更新
7. 写审计日志
8. 事务提交后失效 Portal 缓存

### 9.3 删除流程

1. 查询活跃数据
2. 校验 `version`
3. 保存删除前快照
4. 执行逻辑删除
5. 写审计日志
6. 事务提交后失效 Portal 缓存

### 9.4 重排流程

1. 去重并校验 ID 列表非空
2. 查询请求涉及的活跃记录
3. 校验列表完整覆盖全部活跃卡片
4. 保存排序前快照
5. 重写 `sortOrder`
6. 写审计日志
7. 事务提交后失效 Portal 缓存

### 9.5 Portal 查询流程

1. 读取 Redis 缓存
2. 缓存命中则直接返回
3. 缓存未命中则查询数据库
4. 转换为 `PortalValueCardVO`
5. 写入 Redis，TTL 1 小时
6. 返回结果

## 16. 实现难点与处理策略

### 16.1 难点一：固定数量改为动态集合后，前端兼容性风险上升

处理策略：

- 后端不再输出固定槽位结构
- Portal 统一返回数组
- 前端按数组渲染，不依赖固定三张卡片

### 16.2 难点二：图标资源依赖媒体模块

处理策略：

- 通过媒体服务或媒体查询能力校验 `iconMediaId`
- 若媒体模块尚无公共查询服务，应先补一个只读校验能力

### 16.3 难点三：重排接口的一致性

处理策略：

- 强制请求完整覆盖全部活跃卡片
- Service 内校验请求集合与当前集合完全一致
- 不支持部分重排，避免隐藏数据、并发变更带来的顺序漂移

### 16.4 难点四：缓存与事务一致性

处理策略：

- 统一通过 `PortalCacheInvalidationSupport` 做提交后删除
- 不允许在事务中直接删缓存

---

## 17. 推荐落地方案

推荐采用“独立内容卡片表 + 标准 CRUD + Portal 缓存”的实现方式。

理由：

- 与现有站点内容模块结构一致，便于统一维护
- 支持未来从 3 张扩展为任意数量，不需要改表结构
- 便于挂接审计、权限、缓存、排序等站点通用能力
- 比将数据硬塞进站点配置表更清晰，也更适合分页、重排和乐观锁

## 19. 待确认项

以下事项不阻塞设计输出，但在正式开发前应确认：

- 媒体模块是否已提供统一的媒体资源可用性校验服务
- 前端是否需要返回 `iconAlt`、深色模式图标或 SVG 原文
- 默认种子数据是否沿用现有文案中的“同事 / 同仁 / 同享”
- 后台重排交互是拖拽即保存，还是拖拽后统一点击保存
