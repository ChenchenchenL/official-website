# 功能点 16：我们的承诺与标签管理详细设计说明书

## 1. 文档目标

本文档用于落实“功能点 16：我们的承诺与标签管理”的后端详细设计，覆盖业务边界、核心对象、数据库设计、接口契约、缓存策略、审计要求、错误码、测试场景与推荐实现方案，供产品、研发、测试和评审统一执行。

该功能归属于官网“关于我们”页面中的“核心价值观与承诺”区域，但在实现层面需要作为独立补充模块建设，避免与“核心价值观卡片管理”发生对象耦合。

---

## 2. 设计范围

### 2.1 本期包含

本期包含两类能力：

- “我们的承诺”主体宣导长文本配置
- “我们的承诺”底部标签集合的动态管理

### 2.2 本期不包含

以下内容不属于本功能点：

- 核心价值观卡片的图标、标题、副标语、描述管理
- 富文本编辑器能力
- 多语言内容
- 标签颜色、样式、图标、分组配置
- 承诺模块的上下架、定时发布、审批流

---

## 3. 页面业务场景

### 3.1 页面结构理解

“我们的承诺”区域由两部分组成：

1. 主体文案区  
   展示一段完整的品牌宣导文本，例如“云台数据全体员工将继续秉承……”

2. 底部标签区  
   展示多个短文本标签，例如“过硬的技术”“简便的操作”“实用的功能”

### 3.2 后端建模原则

该页面虽然在前台表现为一个区块，但其数据形态并不一致：

- 主体文案是单例配置
- 标签是可无限扩展的列表数据

因此推荐拆分为两个独立核心对象，再通过 Portal 聚合接口对前台一次性输出，避免把单例配置和列表项混到同一张表中。

---

## 4. 核心对象设计

### 4.1 单例对象：PromiseContent

对象名称：`PromiseContent`

业务含义：存储“我们的承诺”模块的主体宣导长文本。

设计特点：

- 单例配置
- 后台仅允许查看和更新
- 不提供新增多条和删除能力
- 使用乐观锁控制并发更新

### 4.2 列表对象：PromiseTag

对象名称：`PromiseTag`

业务含义：存储“我们的承诺”模块底部的单个承诺特性标签。

设计特点：

- 动态 N 项
- 支持新增、编辑、删除、批量重排
- 支持逻辑删除
- 使用 `sort_order` 控制前台顺序
- 编辑与删除使用乐观锁

### 4.3 聚合对象：PromiseModule

对象名称：`PromiseModule`

业务含义：Portal 只读聚合输出对象，对前台一次性返回主体文案与标签列表。

说明：

- `PromiseModule` 不是数据库实体
- 仅作为 Portal VO 或 application 层聚合结果存在

## 6. 业务规则

### 6.1 主体文案 `content`

- 必填
- 按纯文本处理，不提供富文本能力
- 去除首尾空白字符
- 保留中间换行
- 最大长度建议：`2000` 字符

说明：

- 该字段用于宣导文案展示，不需要 HTML 结构能力
- 为降低 XSS 风险和渲染复杂度，本期明确不引入富文本编辑器
- Portal 前端应按纯文本渲染，不应将其当作 HTML 片段插入

### 6.2 主体文案版本号 `version`

- 更新时必填
- 默认值建议：`0`
- 采用乐观锁，避免后台多人编辑时发生静默覆盖

### 6.3 标签文本 `tagText`

- 必填
- 去除首尾空格
- 最大长度建议：`32` 字符
- 活跃数据中全局唯一

说明：

承诺标签属于短语型展示文案，重复录入没有业务意义，因此建议在活跃数据中全局唯一。

### 6.4 标签排序值 `sortOrder`

- 系统维护
- 默认新增到末尾
- 数值越小越靠前
- 建议按 `10, 20, 30...` 分配

### 6.5 标签版本号 `version`

- 编辑时必填
- 删除时必填
- 默认值建议：`0`

### 6.6 删除规则

- 标签删除采用逻辑删除
- 删除后允许重新创建相同文本标签
- 主体文案不提供删除能力

### 6.7 显示规则

本期不为主体文案和标签增加 `visible` 字段，规则如下：

- 主体文案默认始终存在且可被 Portal 读取
- Portal 标签列表返回全部活跃标签
- 如后续存在“临时隐藏但不删除”的业务诉求，再新增 `visible` 字段并补充接口

### 6.8 排序规则

- Admin 标签列表默认排序：`sort_order ASC, id ASC`
- Portal 标签列表固定排序：`sort_order ASC, id ASC`
- 批量重排必须完整覆盖全部活跃标签
- 重排后统一重写为 `10, 20, 30...`

### 6.9 权限规则

- 所有 `/admin/api/**` 接口必须登录
- 所有 Admin 接口必须校验 `Administrator` 角色
- Portal 接口允许匿名访问

### 6.10 审计规则

以下操作必须写入 `sys_audit_log`：

- `UPDATE_PROMISE_CONTENT`
- `CREATE_PROMISE_TAG`
- `UPDATE_PROMISE_TAG`
- `DELETE_PROMISE_TAG`
- `REORDER_PROMISE_TAG`

审计字段至少包括：

- 操作人 ID
- 操作动作
- 模块：`SITE`
- 对象类型：`PROMISE_CONTENT` 或 `PROMISE_TAG`
- 对象 ID
- 请求 IP
- User-Agent
- 操作时间
- `traceId`
- 变更前快照
- 变更后快照

## 7. 数据库设计

### 7.1 表一：主体文案表 `cms_promise_content`

推荐采用单例表承载主体文案。

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | BIGINT | PK | - | 主键 |
| config_key | VARCHAR(64) | NOT NULL | `default` | 单例配置键，固定为 `default` |
| content | VARCHAR(2000) | NOT NULL | - | 主体宣导长文本 |
| version | INT | NOT NULL | 0 | 乐观锁版本号 |
| created_at | DATETIME | NOT NULL | - | 创建时间 |
| updated_at | DATETIME | NOT NULL | - | 更新时间 |
| created_by | BIGINT | NOT NULL | - | 创建人 ID |
| updated_by | BIGINT | NOT NULL | - | 更新人 ID |
| deleted_marker | BIGINT | NOT NULL | 0 | 逻辑删除标记，单例配置固定为 0 |

设计说明：

- 本表仅允许存在 1 条 `config_key = default` 的活跃记录
- 建议初始化脚本预置一条默认数据
- 后台只提供查询和更新，不提供删除接口

### 7.2 表二：承诺标签表 `cms_promise_tag`

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | BIGINT | PK | - | 主键 |
| tag_text | VARCHAR(32) | NOT NULL | - | 标签文本 |
| sort_order | INT | NOT NULL | 0 | 排序值，越小越靠前 |
| version | INT | NOT NULL | 0 | 乐观锁版本号 |
| created_at | DATETIME | NOT NULL | - | 创建时间 |
| updated_at | DATETIME | NOT NULL | - | 更新时间 |
| created_by | BIGINT | NOT NULL | - | 创建人 ID |
| updated_by | BIGINT | NOT NULL | - | 更新人 ID |
| deleted_marker | BIGINT | NOT NULL | 0 | 逻辑删除标记 |

### 7.3 索引设计

`cms_promise_content`

- 唯一索引：`uk_cms_promise_content_key_del(config_key, deleted_marker)`

`cms_promise_tag`

- 唯一索引：`uk_cms_promise_tag_text_del(tag_text, deleted_marker)`
- 普通索引：`idx_cms_promise_tag_del_sort(deleted_marker, sort_order, id)`

## 8. 接口契约设计

## 8.1 Admin：主体文案接口

### 8.1.1 获取主体文案详情

- 路径：`GET /admin/api/promise-content`
- 说明：返回后台当前可编辑的主体文案配置

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": {
    "id": 1,
    "content": "云台数据全体员工将继续秉承专业、务实、创新、共赢的理念，为客户持续创造真实价值。",
    "version": 0,
    "updatedAt": "2026-06-28T12:00:00"
  }
}
```

### 8.1.2 更新主体文案

- 路径：`PUT /admin/api/promise-content`

请求示例：

```json
{
  "version": 0,
  "content": "云台数据全体员工将继续秉承专业、务实、创新、共赢的理念，为客户持续创造真实价值。"
}
```

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": null
}
```

## 8.2 Admin：标签管理接口

### 8.2.1 分页获取标签列表

- 路径：`GET /admin/api/promise-tags?pageNo=1&pageSize=20`
- 默认排序：`sort_order ASC, id ASC`

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": {
    "list": [
      {
        "id": 11,
        "tagText": "过硬的技术",
        "sortOrder": 10,
        "version": 0,
        "updatedAt": "2026-06-28T12:10:00"
      }
    ],
    "total": 3,
    "pageNo": 1,
    "pageSize": 20
  }
}
```

### 8.2.2 新增标签

- 路径：`POST /admin/api/promise-tags`

请求示例：

```json
{
  "tagText": "务实的交付"
}
```

说明：

- 新增标签自动追加到末尾
- `sortOrder` 由系统计算，不允许前端直接传入

### 8.2.3 编辑标签

- 路径：`PUT /admin/api/promise-tags/{id}`

请求示例：

```json
{
  "version": 0,
  "tagText": "稳定的交付"
}
```

### 8.2.4 删除标签

- 路径：`DELETE /admin/api/promise-tags/{id}`

请求参数：

- `version=0`

说明：

- 采用逻辑删除
- 删除后允许重新创建同名标签

### 8.2.5 批量重排标签

- 路径：`POST /admin/api/promise-tags/reorder`

请求示例：

```json
{
  "orderedPromiseTagIds": [13, 11, 12]
}
```

规则：

- 必须完整覆盖全部活跃标签
- 不允许遗漏、重复或混入已删除标签
- 按请求顺序重新分配 `sort_order`

## 9. 缓存策略

### 9.1 缓存对象

- Portal 聚合接口 `GET /portal/api/our-promises` 的完整返回值

### 9.2 缓存 Key

- `official:portal:our_promises`

### 9.3 TTL

- 默认 `1 小时`

### 9.4 失效时机

以下后台操作成功提交后，必须清除上述缓存：

- 更新主体文案
- 新增标签
- 编辑标签
- 删除标签
- 重排标签

### 9.5 失效要求

- 必须在数据库事务提交后执行
- 必须带延迟二次删除兜底
- Redis 异常时不得影响主流程提交

## 10. 错误码建议

建议在 `ErrorCode.java` 的 `SITE` 模块段新增以下错误码：

| 错误码 | 标识符 | 默认提示 |
| --- | --- | --- |
| 31401 | `SITE_PROMISE_CONTENT_NOT_FOUND` | 我们的承诺主体文案配置不存在 |
| 31402 | `SITE_PROMISE_TAG_NOT_FOUND` | 我们的承诺标签不存在或已被逻辑删除 |
| 31403 | `SITE_PROMISE_TAG_TEXT_DUPLICATE` | 我们的承诺标签文本重复 |

通用错误复用规则：

- 参数校验失败：`10001 COMMON_PARAM_INVALID`
- 版本冲突：`10003 COMMON_STATE_CONFLICT`
- 未登录：`20001 AUTH_UNAUTHORIZED`
- 无权限：`20002 AUTH_FORBIDDEN`

## 11. 初始化与种子数据建议

### 11.1 主体文案初始化

系统首次部署时建议插入 1 条默认主体文案记录。

### 11.2 标签初始化

系统首次部署时建议插入若干默认标签，例如：

- `过硬的技术`
- `简便的操作`
- `实用的功能`

### 11.3 初始化要求

- 初始化数据必须可重复执行且不会造成重复活跃记录
- 主体文案初始化记录必须固定 `config_key = default`
- 单例主体文案必须通过数据库唯一索引和 Service 查询规则共同保证只有 1 条活跃记录

## 12. 关键难点与解决建议

### 12.1 难点一：单例文案与列表标签的对象边界不同

问题：

- 主体文案是单例配置
- 标签是动态集合
- 如果强行塞入同一个表或同一个 CRUD 入口，会导致模型扭曲

解决建议：

- 单独设计 `PromiseContent` 表和 `PromiseTag` 表
- Portal 通过聚合 VO 对前台屏蔽内部拆分

### 12.2 难点二：标签重排的一致性

问题：

- 前端拖拽排序容易只传局部 ID
- 一旦后端接受不完整列表，排序结果可能漂移

解决建议：

- 重排接口必须完整覆盖全部活跃标签
- 后端对请求 ID 集合与数据库当前活跃集合做严格一致性校验
- 不一致时返回 `10003`

### 12.3 难点三：Portal 聚合缓存一致性

问题：

- 一个 Portal 接口依赖两个表
- 任一写操作都可能导致缓存过期

解决建议：

- 使用单一聚合缓存 Key
- 所有相关写操作统一失效同一个 Key
- 失效动作放到事务提交后，并加延迟二次删除

### 12.4 难点四：主体文案的安全边界

问题：

- 长文本最容易被误做成富文本
- 一旦引入 HTML，自带 XSS 和样式漂移风险

解决建议：

- 本期明确规定主体文案仅支持纯文本
- DTO 和 Service 都按普通字符串处理
- 前端按纯文本渲染，不使用原样 HTML 注入

## 13. 推荐实现方案

### 13.1 推荐方案

推荐采用“后台拆分维护、前台聚合读取”的实现方式：

1. 后台单独维护主体文案
2. 后台单独维护标签列表
3. 前台通过 1 个 Portal 接口获取完整模块数据

### 13.2 推荐原因

- 符合真实业务对象边界，后续扩展更清晰
- 后台接口职责单一，便于权限、审计和测试覆盖
- 前台只调一次接口，渲染成本低
- 缓存粒度稳定，失效逻辑简单
- 未来若新增“标签可隐藏”“多语言”“主题版本”等能力，扩展路径清晰

### 13.3 不推荐方案

不建议采用以下做法：

- 把主体文案和标签混在一张表
- 只提供一个大而全的 Admin 保存接口，把文案和标签一起全量覆盖
- 为主体文案引入富文本编辑器

原因：

- 容易引发数据结构混乱
- 并发冲突粒度过粗
- 审计和缓存难以精确定位
- 页面简单需求被过度工程化

---

## 14. 测试与验收标准

至少覆盖以下场景：

1. 主体文案更新成功后，Portal 接口应返回最新内容
2. 主体文案并发更新时，旧 `version` 请求应返回 `10003`
3. 新增两个相同标签文本的活跃记录时，应抛出 `31403`
4. 删除标签后，应允许重新创建同名标签
5. Portal 标签列表必须严格按 `sort_order ASC, id ASC` 返回
6. 标签重排请求若遗漏任意一个活跃标签，应返回 `10003`
7. 标签重排请求若混入不存在或已删除标签，应返回 `10001` 或对应业务异常
8. 任意写操作成功后，`official:portal:our_promises` 缓存应在秒级内失效并重建
9. 主体文案字段长度超过上限时，应由参数校验拦截
10. 标签文本为空、全空格或超长时，应由参数校验拦截

## 15. 与其他功能点的关系

### 15.1 与功能点 15 的边界

功能点 15 负责：

- 核心价值观卡片
- Icon、大字标题、副标语、详情描述

功能点 16 负责：

- 我们的承诺主体文案
- 承诺短标签列表

两者页面邻近，但不应共用实体、表或 CRUD 接口。

### 15.2 与 Portal 页面关系

前端“关于我们”页可分别请求：

- `value-cards`
- `timeline-events`
- `our-promises`

也可后续由更高层页面聚合接口统一编排，但本功能点本身不强制要求页面级大聚合。

---

## 16. 最终结论

“我们的承诺与标签管理”应被建模为一个由单例文案和动态标签集合组成的独立模块，而不是附着在“核心价值观卡片”对象上的附属字段。

最稳妥的实现路径是：

- `PromiseContent` 负责主体文案
- `PromiseTag` 负责标签集合
- `PortalPromiseModuleVO` 负责前台聚合输出
