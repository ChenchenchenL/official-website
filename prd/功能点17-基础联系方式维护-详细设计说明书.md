# 功能点 17：基础联系方式维护详细设计说明书

## 1. 文档目标

本文档用于落实“功能点 17：基础联系方式维护”的后端详细设计，覆盖业务边界、核心对象、数据库设计、接口契约、缓存策略、安全审计、错误码、测试场景与推荐实现方案，供产品、研发、测试和评审统一执行。

该功能属于“联系我们与线索留资模块”中的基础配置能力，目标是将官网“联系我们”区域中的基础联系方式从静态页面文案升级为可后台动态维护的单例配置。

---

## 2. 设计范围

### 2.1 本期包含

本期仅包含以下三项基础联系方式字段的后台维护与前台公开读取：

- 联系地址
- 商务咨询电话
- 联系邮箱

### 2.2 本期不包含

以下内容不属于本功能点：

- 合作方向标签管理
- 留资表单提交流程
- 线索列表、跟进、导出
- 多地址、多电话、多邮箱
- 多语言内容
- 富文本编辑器能力
- 上下架、定时发布、审批流
- 历史版本回溯

### 2.3 与功能点 18 的边界

本功能点只负责单例基础联系方式配置，不承载“合作方向标签”的数据模型与接口。

边界约束如下：

- 功能点 17 只管理 `ContactInfo`
- 功能点 18 将独立管理“合作方向标签”列表对象
- Portal 本期只提供 `GET /portal/api/contact-info`
- 后续如前台页面需要一次性返回“联系方式 + 合作方向标签”，应通过 application 层新增聚合接口或聚合 VO 解决，不直接修改本接口字段结构，也不将两类数据混入同一张表

## 3. 页面业务场景

### 3.1 页面结构理解

官网“联系我们”页面左侧区域通常包含：

1. 联系地址
2. 商务咨询电话
3. 联系邮箱
4. 合作方向标签区

其中第 1 至第 3 项属于基础联系方式，第 4 项属于另一组动态标签能力，两者在前台页面上相邻展示，但后端数据形态不同。

### 3.2 后端建模原则

基础联系方式具备以下特征：

- 固定字段集合
- 永远只有一套当前生效配置
- 后台只需要查看与更新
- 前台只需要读取

因此推荐建模为单例配置对象，而不是列表对象，也不建议并入 `SiteConfig`。

---

## 4. 核心对象设计

### 4.1 核心业务对象

对象名称：`ContactInfo`

业务含义：存储官网“联系我们”区域中的基础联系方式配置。

### 4.2 对象定位

`ContactInfo` 是一个单例配置对象，不是列表对象，不支持多条并行生效记录。

设计特点：

- 单例配置
- 后台仅允许查询与更新
- 不提供新增接口
- 不提供删除接口
- 通过 `config_key=default` 固定定位活跃记录
- 更新时使用乐观锁控制并发覆盖

### 4.3 不并入 `SiteConfig` 的原因

虽然该对象也是配置类数据，但不建议并入 `SiteConfig`，原因如下：

- 从业务边界看，它属于“联系与线索”域，而不是“全局站点配置”域
- 后续功能点 18、19、20 会围绕“联系我们与线索留资模块”继续扩展，独立对象更便于维护边界
- 若并入 `SiteConfig`，将导致站点 SEO/Logo 与联系信息混合演进，增加无关字段耦合
- 单例对象的实现方式可以复用现有 `SiteConfig` 与 `PromiseContent` 模式，无需牺牲模块清晰度

## 6. 业务规则

### 6.1 联系地址 `contactAddress`

- 必填
- 去除首尾空格
- 按纯文本处理
- 最大长度建议：`255` 字符

说明：

- 地址字段用于页面展示，不需要 HTML 能力
- Portal 前端应按纯文本渲染，不应当作 HTML 片段插入

### 6.2 商务咨询电话 `businessPhone`

- 必填
- 去除首尾空格
- 最大长度建议：`64` 字符
- 必须至少包含 `1` 个数字
- 仅允许字符：数字、空格、`+`、`-`、`(`、`)`、`/`、`#`
- 按展示文本建模，不强制绑定某一国家手机号或固定座机格式

说明：

- 该字段本质上是“官网展示用电话文本”，不是电信号段校验字段
- 这样可以兼容座机、分机、国际区号、热线等真实展示形态
- 但仍需通过字符白名单和“至少包含一个数字”的规则拦截纯符号垃圾值

### 6.3 联系邮箱 `contactEmail`

- 必填
- 去除首尾空格
- 最大长度建议：`128` 字符
- 必须满足邮箱格式校验
- 按纯文本存储

### 6.4 版本号 `version`

- 更新时必填
- 默认值建议：`0`
- 使用乐观锁控制并发更新

### 6.5 显示规则

本期不引入 `visible` 字段，规则如下：

- 当前活跃联系方式配置默认始终对 Portal 可读
- 不支持后台隐藏但保留
- 如后续存在“临时隐藏联系方式”的明确业务诉求，再新增 `visible` 字段并补充 Portal 过滤规则

### 6.6 生命周期规则

- 不提供新增能力
- 不提供删除能力
- 活跃记录固定只有一条
- 系统首次部署时必须初始化默认记录
- 若默认记录缺失，不做静默自动创建，也不返回空对象，直接报错暴露问题

### 6.7 权限规则

- 所有 `/admin/api/**` 接口必须登录
- 所有 Admin 接口必须校验 `Administrator` 角色
- 更新接口必须通过 CSRF 校验
- Portal 接口允许匿名访问

## 7. 数据库设计

### 7.1 表名

`cms_contact_info`

### 7.2 表结构

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | BIGINT | PRIMARY KEY | - | 主键 ID |
| config_key | VARCHAR(32) | NOT NULL | - | 单例配置键，固定为 `default` |
| contact_address | VARCHAR(255) | NOT NULL | - | 联系地址 |
| business_phone | VARCHAR(64) | NOT NULL | - | 商务咨询电话 |
| contact_email | VARCHAR(128) | NOT NULL | - | 联系邮箱 |
| version | INT | NOT NULL | 0 | 乐观锁版本号 |
| created_at | DATETIME | NOT NULL | - | 创建时间 |
| updated_at | DATETIME | NOT NULL | - | 更新时间 |
| created_by | BIGINT | NOT NULL | - | 创建人 ID |
| updated_by | BIGINT | NOT NULL | - | 更新人 ID |
| deleted_marker | BIGINT | NOT NULL | 0 | 逻辑删除标记 |

### 7.3 索引设计

唯一索引：

- `uk_cms_contact_info_key_del(config_key, deleted_marker)`

### 7.4 单例约束规则

活跃记录固定使用：

- `config_key = default`
- `deleted_marker = 0`

这样可以在数据库层保证单例配置的唯一性，同时保留与项目现有逻辑删除规范一致的结构。

### 7.5 初始化要求

系统初始化 SQL 必须预置一条默认记录，否则：

- Admin 编辑页无法初始化
- Portal 前台接口无法返回联系方式
- 缺失问题会被误判为业务空值，而不是部署缺陷

因此本期明确要求：默认记录缺失时直接抛出业务错误，不做自动兜底创建。

## 8. API 接口契约

### 8.1 接口基地址

- 后台管理：`/admin/api/contact-info`
- 前台公开：`/portal/api/contact-info`

### 8.2 Admin：获取基础联系方式详情

- 请求方法：`GET`
- 路径：`/admin/api/contact-info`

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": {
    "id": 1,
    "contactAddress": "武汉市东湖新技术开发区光谷大道 77 号",
    "businessPhone": "+86 027-88886666",
    "contactEmail": "business@example.com",
    "version": 0,
    "updatedAt": "2026-06-28T12:00:00"
  }
}
```

说明：

- 用于初始化后台编辑表单
- 只返回当前活跃单例记录

### 8.3 Admin：更新基础联系方式

- 请求方法：`PUT`
- 路径：`/admin/api/contact-info`

请求示例：

```json
{
  "version": 0,
  "contactAddress": "武汉市东湖新技术开发区光谷大道 77 号",
  "businessPhone": "+86 027-88886666",
  "contactEmail": "business@example.com"
}
```

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": null
}
```

接口约束：

- 更新接口返回 `ApiResponse<Void>`
- 前端如需最新版本号，应在保存成功后重新调用 `GET /admin/api/contact-info`
- 不在写接口中返回最新 VO，避免单例写接口返回风格继续分裂

### 8.4 Portal：获取公开基础联系方式

- 请求方法：`GET`
- 路径：`/portal/api/contact-info`

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": {
    "contactAddress": "武汉市东湖新技术开发区光谷大道 77 号",
    "businessPhone": "+86 027-88886666",
    "contactEmail": "business@example.com"
  }
}
```

接口规则：

- 只返回当前活跃配置
- 不返回 `id`、`version`、`createdBy`、`updatedBy`、`deletedMarker` 等后台字段
- 所有字段按纯文本输出

## 9. 缓存与基础设施

### 9.1 缓存对象

缓存对象为：

- `GET /portal/api/contact-info` 的完整返回值

### 9.2 缓存 Key

固定缓存键：

- `official:portal:contact_info`

### 9.3 TTL

- 默认 `1` 小时

### 9.4 缓存失效时机

后台更新成功后：

- 在数据库事务提交后立即删除缓存
- 同时调度延迟二次删除作为兜底

### 9.5 一致性要求

- 不允许在事务内直接删除缓存
- 缓存删除失败不回滚已提交业务数据
- 删除失败必须记录错误日志
- 下一次 Portal 请求应允许回源数据库并重建缓存

## 10. 安全与审计

### 10.1 输入安全

- 地址、电话、邮箱全部按纯文本处理
- 不允许提交 HTML 片段作为富文本
- 前端输出必须进行正常文本渲染，不得拼接进原始 HTML

### 10.2 审计要求

基础联系方式更新属于后台关键配置变更，必须写入 `sys_audit_log`。

推荐审计字段：

- 操作人 ID
- 操作人账号
- 模块标识
- 操作动作
- 对象类型
- 对象 ID
- 请求 IP
- User-Agent
- 操作时间
- `traceId`
- 变更前快照
- 变更后快照

### 10.3 审计动作建议

- `UPDATE_CONTACT_INFO`

### 10.4 模块与对象类型建议

- 模块：`LEAD`
- 对象类型：`CONTACT_INFO`

说明：

- 该对象虽然是官网公开内容，但从业务归属看仍属于“联系与线索”域
- 审计模块使用 `LEAD` 更符合后续与功能点 18、19、20 的统一治理方向

### 10.5 日志脱敏要求

- 普通运行日志不打印完整联系邮箱
- 普通运行日志不打印完整商务电话
- 审计快照允许完整存储，因为其属于受控后台审计数据

## 11. 错误码设计

### 11.1 推荐错误码

| 错误码 | 标识符 | 默认提示 |
| --- | --- | --- |
| 70001 | `LEAD_CONTACT_INFO_NOT_FOUND` | 基础联系方式配置不存在 |
| 10001 | `COMMON_PARAM_INVALID` | 请求参数不正确 |
| 10003 | `COMMON_STATE_CONFLICT` | 当前状态不允许执行该操作 |
| 20001 | `AUTH_UNAUTHORIZED` | 请先登录 |
| 20002 | `AUTH_FORBIDDEN` | 无权执行该操作 |
| 20005 | `AUTH_CSRF_INVALID` | 请求校验失败，请刷新页面后重试 |

### 11.2 错误码触发说明

- `70001`
  - 默认单例记录不存在
  - 记录已被逻辑删除但未补种子数据

- `10001`
  - 地址为空
  - 电话为空
  - 电话包含非法字符
  - 电话不包含任何数字
  - 邮箱为空
  - 邮箱格式不合法
  - 字段长度超限

- `10003`
  - 更新时 `version` 不匹配

### 11.3 分段说明

该错误码建议归入 `70000-79999` 段。

后续如同步更新全局文档，建议将该分段语义从“客户线索”扩展为“联系与线索”，以覆盖 `ContactInfo` 与 `Lead`。

## 12. 测试与验收场景

1. 默认记录读取验证
   - 初始化 SQL 已存在默认记录时，Admin 和 Portal 都能正确返回联系方式

2. 地址必填校验
   - 地址为空或仅空白字符时，更新接口应返回 `10001`

3. 电话合法性校验
   - 电话为空、只包含符号、不包含任何数字、存在非法字符时，应返回 `10001`

4. 邮箱合法性校验
   - 邮箱为空、邮箱格式不合法时，应返回 `10001`

5. 长度边界校验
   - 地址超过 `255` 字符、电话超过 `64` 字符、邮箱超过 `128` 字符时，应返回 `10001`

6. 并发锁冲突验证
   - 两个管理员同时编辑同一配置，后提交请求因 `version` 不匹配返回 `10003`

7. Portal 输出边界验证
   - Portal 接口不得输出 `id`、`version`、`createdBy`、`updatedBy`、`deletedMarker`

8. 缓存一致性验证
   - 管理端更新成功后，`official:portal:contact_info` 应在秒级内失效并可读取到新值

9. 缺失记录异常验证
   - 若默认记录不存在，Admin 与 Portal 均应返回 `70001`，而不是空对象或静默成功

10. 审计写入验证
   - 更新联系方式后，`sys_audit_log` 中必须存在对应 `UPDATE_CONTACT_INFO` 审计记录，且包含前后快照

## 13. 实现难点与解决方案

### 13.1 单例配置并发覆盖

难点：

后台虽然只有一条配置记录，但仍然存在多人同时编辑的风险，后保存者可能静默覆盖先保存者内容。

解决方案：

- 使用 `version` 乐观锁
- 更新 SQL 必须带 `id + version` 条件
- 受影响行数不为 `1` 时返回 `10003`

### 13.2 多实例缓存一致性

难点：

Portal 接口为高频公开读取，在多实例下若后台更新后缓存未及时失效，会继续返回旧数据。

解决方案：

- 通过统一缓存失效组件，在事务提交后删除缓存
- 同时调度延迟二次删除
- 配置 TTL 作为最终兜底

### 13.3 与功能点 18 的未来聚合边界

难点：

功能点 17 与功能点 18 在页面上同处“联系我们”左侧区域，容易在实现时被错误合并为一个对象或一个表。

解决方案：

- 保持 `ContactInfo` 与“合作方向标签”两个对象独立
- 单例字段与动态标签列表禁止混表
- 前台如需一次聚合展示，通过 application 层或聚合 VO 实现，不破坏单对象接口边界

## 14. 推荐实现方案与理由

### 14.1 推荐方案

采用“独立单例表 + `lead` 模块独立 Service + Portal 缓存”的实现方式。

### 14.2 方案理由

1. 业务边界清晰
   - 联系方式属于“联系与线索”域，不应继续膨胀 `SiteConfig`

2. 与后续功能兼容
   - 功能点 18、19、20 可继续围绕 `lead` 模块扩展，不需要在 `site` 模块中横向插入无关对象

3. 技术复用成本低
   - 单例模型可复用现有 `SiteConfig`、`PromiseContent` 的实现经验
   - 审计、乐观锁、事务提交后删缓存的机制都已有现成范式

4. 前后端契约稳定
   - Admin 与 Portal 接口职责单一
   - 本期不提前引入聚合复杂度
   - 后续新增“联系方式 + 标签”的聚合接口时，不会破坏本接口兼容性

### 14.3 本期默认假设

- 本期只维护一套主联系方式
- 电话字段按展示文本建模，不做强国家格式校验
- 默认单例记录由初始化 SQL 保证存在
- 暂不引入多语言、显示开关、富文本、历史版本
