# 功能点 18：合作方向标签管理详细设计说明书

## 1. 文档目标

本文档用于落实“功能点 18：合作方向标签管理”的后端详细设计，覆盖业务边界、核心对象、数据库设计、接口契约、缓存策略、安全审计、错误码、测试场景与推荐实现方案，供产品、研发、测试和评审统一执行。

该功能属于“联系我们与线索留资模块”中的页面内容配置能力，目标是将官网“联系我们”页面左侧蓝色背景区中的合作方向标签从静态文案升级为可后台动态维护的动态标签列表。

---

## 2. 设计范围

### 2.1 本期包含

本期包含以下能力：

- 合作方向标签列表查询
- 合作方向标签新增
- 合作方向标签编辑
- 合作方向标签删除
- 合作方向标签批量排序
- Portal 前台公开标签列表读取
- 标签列表缓存与后台写操作后的失效

### 2.2 本期不包含

以下内容不属于本功能点：

- 基础联系方式维护
- 留资表单提交流程
- 线索列表、状态跟进、导出
- 标签颜色、图标、跳转链接、分组
- 多语言标签
- 富文本能力
- 标签显示/隐藏开关
- 审批流、定时发布、历史版本回溯

### 2.3 与功能点 17 的边界

本功能点只负责“合作方向标签”列表，不负责联系地址、商务电话、联系邮箱。

边界约束如下：

- 功能点 17 负责单例对象 `ContactInfo`
- 功能点 18 负责列表对象 `CooperationDirectionTag`
- 本期 Portal 端保持两个独立公开接口
- 后续如前台页面需要一次性返回“联系方式 + 合作方向标签”，应通过 application 层新增聚合接口解决，不直接修改现有单对象接口结构，也不将两类数据混入同一张表

## 3. 页面业务场景

### 3.1 页面结构理解

官网“联系我们”页面左侧蓝色背景区用于展示企业可承接的合作方向，例如：

- 企业数字化建设
- 数据平台建设
- AI 应用开发

这些标签的作用是向外部访问者说明企业服务边界与合作方向，不承担筛选、跳转或交互逻辑。

### 3.2 后端建模原则

该区域的数据特征如下：

- 为短文本标签列表
- 标签数量不固定
- 标签顺序影响前台视觉呈现
- 标签之间没有父子层级
- 标签允许为空列表

因此推荐建模为独立列表对象 `CooperationDirectionTag`，而不是单例配置字段，也不应合并到 `ContactInfo` 表中。

## 4. 核心对象设计

### 4.1 核心业务对象

对象名称：`CooperationDirectionTag`

业务含义：存储“联系我们”页面左侧合作方向标签区中的单个文本标签。

### 4.2 对象定位

`CooperationDirectionTag` 是一个动态列表对象。

设计特点：

- 动态 N 项
- 支持新增、编辑、删除
- 支持批量排序
- 使用逻辑删除
- 使用 `sort_order` 控制前台展示顺序
- 编辑与删除使用乐观锁

### 4.3 为什么本期纳入排序能力

原始需求只明确写到“新增、编辑、删除”，但该类标签直接面向前台展示，顺序会影响页面表达。如果没有排序能力，只能依赖创建顺序，不利于后台内容运营。

因此本 PRD 推荐本期直接纳入批量排序能力，保持与现有标签类模块的一致性。

## 6. 业务规则

### 6.1 标签文本 `tagText`

- 必填
- 去除首尾空格
- 按纯文本处理
- 最大长度建议：`32` 字符
- 活跃数据中全局唯一

说明：

- 该标签用于短文本展示，重复录入没有业务意义
- 为避免录入“企业数字化建设”和“ 企业数字化建设 ”这类伪差异值，必须先去空白再校验唯一性
- Portal 前端应按纯文本渲染，不得作为 HTML 片段插入

### 6.2 排序值 `sortOrder`

- 系统维护
- 数值越小越靠前
- 默认新增到末尾
- 推荐按 `10, 20, 30...` 分配

### 6.3 版本号 `version`

- 编辑时必填
- 删除时必填
- 默认值建议：`0`
- 用于乐观锁校验

### 6.4 删除规则

- 删除采用逻辑删除
- 删除后允许重新创建相同文本标签
- 删除后 Portal 不返回该标签

### 6.5 显示规则

本期不引入 `visible` 字段，规则如下：

- Portal 返回全部活跃标签
- 不支持“隐藏但保留”
- 如后续有运营需求，再增加 `visible` 字段并补充筛选逻辑

### 6.6 排序规则

- Admin 列表默认排序：`sort_order ASC, id ASC`
- Portal 列表固定排序：`sort_order ASC, id ASC`
- 批量重排必须完整覆盖全部活跃标签
- 重排后统一重写为 `10, 20, 30...`

### 6.7 空数据规则

与功能点 17 不同，本对象允许空列表存在。

规则如下：

- 后台可在没有任何标签的情况下正常打开列表页
- Portal 在没有活跃标签时返回空数组
- 不要求初始化 SQL 必须预置标签数据
- 当当前无任何活跃标签时，重排接口返回 `10001`，表示当前请求不成立

### 6.8 权限规则

- 所有 `/admin/api/**` 接口必须登录
- 所有 Admin 接口必须校验 `Administrator` 角色
- 所有 Admin 写接口必须通过 CSRF 校验
- Portal 接口允许匿名访问

## 7. 数据库设计

### 7.1 表名

`cms_cooperation_direction_tag`

### 7.2 表结构

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | BIGINT | PRIMARY KEY | - | 主键 ID |
| tag_text | VARCHAR(32) | NOT NULL | - | 标签文本 |
| sort_order | INT | NOT NULL | 0 | 排序值 |
| version | INT | NOT NULL | 0 | 乐观锁版本号 |
| created_at | DATETIME | NOT NULL | - | 创建时间 |
| updated_at | DATETIME | NOT NULL | - | 更新时间 |
| created_by | BIGINT | NOT NULL | - | 创建人 ID |
| updated_by | BIGINT | NOT NULL | - | 更新人 ID |
| deleted_marker | BIGINT | NOT NULL | 0 | 逻辑删除标记 |

### 7.3 索引设计

唯一索引：

- `uk_cms_coop_direction_tag_text_del(tag_text, deleted_marker)`

普通索引：

- `idx_cms_coop_direction_tag_del_sort(deleted_marker, sort_order, id)`

说明：

- Portal 和 Admin 的核心查询路径都是先过滤活跃数据，再按排序输出
- 因此普通索引应优先以 `deleted_marker` 作为前导列，而不是把它放在末尾

## 8. API 接口契约

### 8.1 接口基地址

- 后台管理：`/admin/api/cooperation-direction-tags`
- 前台公开：`/portal/api/cooperation-direction-tags`

### 8.2 Admin：获取标签全量列表

- 请求方法：`GET`
- 路径：`/admin/api/cooperation-direction-tags`

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": [
    {
      "id": 1,
      "tagText": "企业数字化建设",
      "sortOrder": 10,
      "version": 0,
      "updatedAt": "2026-06-28T13:00:00"
    }
  ]
}
```

说明：

- Admin 列表不分页，直接返回全部活跃标签
- 这样可直接支撑后台拖拽排序，避免“分页列表”和“全集重排”之间的协议冲突
- 若后续标签数量显著增长，再单独补充分页能力与重排专用全量接口

### 8.3 Admin：新增标签

- 请求方法：`POST`
- 路径：`/admin/api/cooperation-direction-tags`

请求示例：

```json
{
  "tagText": "数据平台建设"
}
```

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": null
}
```

规则：

- 新增标签自动追加到末尾
- 新增后写审计并失效 Portal 缓存

### 8.4 Admin：编辑标签

- 请求方法：`PUT`
- 路径：`/admin/api/cooperation-direction-tags/{id}`

请求示例：

```json
{
  "version": 0,
  "tagText": "AI 应用开发"
}
```

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": null
}
```

规则：

- 必须校验 `version`
- 编辑不改变当前 `sort_order`

### 8.5 Admin：删除标签

- 请求方法：`DELETE`
- 路径：`/admin/api/cooperation-direction-tags/{id}?version=0`

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": null
}
```

规则：

- 删除采用逻辑删除
- 删除后允许同名重新创建
- 删除后写审计并失效 Portal 缓存

### 8.6 Admin：批量重排标签

- 请求方法：`POST`
- 路径：`/admin/api/cooperation-direction-tags/reorder`

请求示例：

```json
{
  "orderedCooperationDirectionTagIds": [3, 1, 2]
}
```

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": null
}
```

规则：

- 必须完整覆盖全部活跃标签
- 不允许重复 ID
- 不允许混入已删除标签
- 当当前无活跃标签时，返回 `10001`
- 保存后按请求顺序重写 `sort_order = 10, 20, 30...`

### 8.7 Portal：获取公开标签列表

- 请求方法：`GET`
- 路径：`/portal/api/cooperation-direction-tags`

响应示例：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": [
    {
      "tagText": "企业数字化建设"
    },
    {
      "tagText": "数据平台建设"
    }
  ]
}
```

规则：

- 仅返回活跃标签
- 固定排序：`sort_order ASC, id ASC`
- 不返回 `id`、`version`、`createdBy`、`updatedBy`、`deletedMarker`

## 9. 缓存与基础设施

### 9.1 缓存对象

缓存对象为：

- `GET /portal/api/cooperation-direction-tags` 的完整返回值

### 9.2 缓存 Key

固定缓存键：

- `official:portal:cooperation_direction_tags`

### 9.3 TTL

- 默认 `1` 小时

### 9.4 缓存失效时机

后台以下操作成功后，在事务提交后清理缓存，并执行延迟二次删除：

- 新增
- 编辑
- 删除
- 重排

### 9.5 一致性要求

- 缓存删除必须在事务提交后执行
- 删除失败不回滚业务数据
- 删除失败必须记录日志
- Portal 请求应允许回源数据库并重建缓存

## 11. 错误码设计

### 11.1 推荐错误码

| 错误码 | 标识符 | 默认提示 |
| --- | --- | --- |
| 70002 | `LEAD_COOPERATION_DIRECTION_TAG_NOT_FOUND` | 合作方向标签不存在或已被逻辑删除 |
| 70003 | `LEAD_COOPERATION_DIRECTION_TAG_TEXT_DUPLICATE` | 合作方向标签文本重复 |
| 10001 | `COMMON_PARAM_INVALID` | 请求参数不正确 |
| 10003 | `COMMON_STATE_CONFLICT` | 当前状态不允许执行该操作 |
| 20001 | `AUTH_UNAUTHORIZED` | 请先登录 |
| 20002 | `AUTH_FORBIDDEN` | 无权执行该操作 |
| 20005 | `AUTH_CSRF_INVALID` | 请求校验失败，请刷新页面后重试 |

### 11.2 错误码触发说明

- `70002`
  - 编辑目标不存在
  - 删除目标不存在
  - 重排请求中包含不存在或已删除标签

- `70003`
  - 活跃数据中已存在同名标签

- `10001`
  - 标签文本为空
  - 标签文本超长
  - 排序列表为空
  - 排序列表包含空 ID
  - 排序列表包含重复 ID
  - 当前无活跃标签却发起重排

- `10003`
  - 排序列表未完整覆盖当前全部活跃标签
  - 更新或删除时版本冲突

## 12. 测试与验收场景

1. 新增标签验证  
   - 新增合法标签成功，自动追加到末尾

2. 重名校验验证  
   - 活跃数据中新增同名标签时返回 `70003`
   - 删除后的同名标签允许重新创建

3. 编辑校验验证  
   - 编辑为重复文本时返回 `70003`
   - 编辑时 `version` 不匹配返回 `10003`

4. 删除校验验证  
   - 删除不存在标签时返回 `70002`
   - 删除后 Portal 不再返回该标签

5. Portal 排序验证  
   - Portal 返回结果必须按 `sort_order ASC, id ASC` 输出

6. 重排完整性验证  
   - 排序列表遗漏任一活跃标签时返回 `10003`
   - 排序列表重复 ID 时返回 `10001`

7. 空列表验证  
   - 无任何活跃标签时，Portal 返回空数组，Admin 返回空列表

8. 空列表重排验证  
   - 当前无活跃标签时调用重排接口，返回 `10001`

9. 缓存一致性验证  
   - 任意写操作成功后，`official:portal:cooperation_direction_tags` 应在秒级内失效

10. 审计写入验证  
   - 新增、编辑、删除、重排后都必须存在对应审计记录

## 13. 实现难点与解决方案

### 13.1 与功能点 17 的边界耦合

难点：

“联系方式”和“合作方向标签”在前台处于同一块区域，容易被误设计成一个对象或一张表。

解决方案：

- `ContactInfo` 保持单例
- `CooperationDirectionTag` 保持列表
- 禁止单例字段和标签列表混表
- 如需前台一次性读取，使用 application 层聚合接口解决

### 13.2 标签排序完整性

难点：

如果只提交局部排序，可能导致数据库中的旧排序值与新顺序混杂，前台展示不稳定。

解决方案：

- 批量重排必须完整覆盖全部活跃标签
- 服务端统一按固定步长重写排序值
- 不接受“局部挪动但不带全集”的排序协议
- Admin 列表本期直接采用全量接口，避免分页视图与全集重排之间的契约冲突

### 13.3 多实例缓存一致性

难点：

后台写操作后，Portal 高并发场景下可能继续返回旧标签列表。

解决方案：

- 事务提交后失效缓存
- 增加延迟二次删除
- 保留 TTL 兜底

## 14. 推荐实现方案与理由

### 14.1 推荐方案

采用“独立列表表 + `lead` 模块 Service + Portal 公开列表缓存”的实现方案。

### 14.2 方案理由

1. 对象边界清晰  
   - 与功能点 17 的单例配置职责分离

2. 适合后续扩展  
   - 后续可继续加入聚合接口，而不破坏现有对象模型

3. 与现有标签模块风格一致  
   - 可直接复用“标签 CRUD + 排序 + 缓存 + 审计”的实现模式

4. Portal 契约稳定  
   - 前台只取短文本数组，不暴露后台字段，便于前端长期稳定接入

### 14.3 本期默认假设

- 合作方向标签本期按纯文本短标签处理
- 本期不提供颜色、图标、链接、分组
- 本期直接支持批量排序
- 标签允许为空列表存在，不强制初始化默认数据
