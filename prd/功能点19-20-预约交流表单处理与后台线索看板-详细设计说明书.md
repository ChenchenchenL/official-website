# 功能点 19-20：预约交流表单处理与后台线索看板详细设计说明书

## 1. 文档目标

本文档用于落实“功能点 19：C端留资数据接收接口”和“功能点 20：后台线索看板中心”的后端详细设计，覆盖业务边界、核心对象、数据库设计、接口契约、限流防刷、安全审计、错误码、测试场景与推荐实现方案，供产品、研发、测试和评审统一执行。

该功能属于“联系我们与线索留资模块”，目标是形成“前台提交线索 -> 后台查看与跟进 -> 数据导出”的完整业务闭环。

---

## 2. 设计范围

### 2.1 本期包含

本期包含以下能力：

- 前台匿名预约交流表单提交
- 同一 IP 的提交频率限制
- 后台线索分页查询
- 后台单条线索详情查看
- 后台单条线索状态流转
- 后台线索 Excel 导出
- 线索数据审计与操作留痕

### 2.2 本期不包含

以下内容不属于本功能点：

- 线索删除
- 线索分配给具体销售人员
- CRM、企业微信、短信、邮件自动同步
- 多级审批、回访记录、跟进备注富文本
- 前台用户登录态绑定
- 线索公开展示

### 2.3 与其它模块的边界

本功能点只负责留资数据接收和后台看板，不负责“基础联系方式维护”和“合作方向标签管理”。

边界约束如下：

- 基础联系方式属于功能点 17，保持独立单例配置
- 合作方向标签属于功能点 18，保持独立列表对象
- 留资表单仅负责接收用户提交的线索数据
- 后台线索看板仅负责查询、状态流转和导出
- 如后续前台页面需要组合“联系方式 + 合作方向标签 + 留资表单配置”，应通过 application 层新增聚合接口解决，不直接混表

## 3. 业务场景与目标

### 3.1 业务场景

官网“预约交流”入口承接来自首页、导航栏、联系我们页面等场景的访客提交。用户填写姓名、公司、邮箱、电话和需求描述后，系统保存为一条线索记录，供销售团队在后台持续跟进。

### 3.2 功能目标

- 前台提交要足够轻量，降低访客流失
- 后台列表要支持按时间筛选和分页查看
- 管理员能够修改线索状态，形成跟进闭环
- 导出能力要满足销售交接、外部汇总和离线分析
- 整个过程必须具备基础防刷、审计和敏感信息保护

## 4. 核心对象设计

### 4.1 核心业务对象

对象名称：`Lead`

业务含义：一条访客提交的预约交流线索记录。

### 4.2 对象定位

`Lead` 是一个标准的留资记录对象，不是配置对象，也不是一次性消息对象。

设计特点：

- 支持匿名提交
- 支持后台查看与状态流转
- 支持导出
- 使用逻辑删除字段作为底座兼容，但本期不提供后台删除接口
- 使用乐观锁控制状态更新并发

### 4.3 状态枚举

建议状态如下：

| 状态码 | 状态值 | 说明 |
| --- | --- | --- |
| `0` | `UNHANDLED` | 未处理 |
| `1` | `PROCESSING` | 处理中 |
| `2` | `ARCHIVED` | 已归档 |
| `3` | `INVALID` | 无效线索 |

说明：

- 新建记录默认进入 `UNHANDLED`
- 后台可直接切换到任意合法状态
- 重复提交相同状态按幂等请求处理，不产生额外数据变更

## 6. 业务规则

### 6.1 前台提交字段规则

#### 姓名 `name`

- 必填
- 去除首尾空格
- 最大长度建议：`64` 字符
- 纯文本存储

#### 公司 `company`

- 必填
- 去除首尾空格
- 最大长度建议：`128` 字符
- 纯文本存储

#### 邮箱 `email`

- 必填
- 去除首尾空格
- 最大长度建议：`128` 字符
- 必须满足邮箱格式
- 纯文本存储

#### 电话 `phone`

- 选填
- 去除首尾空格
- 最大长度建议：`64` 字符
- 仅允许数字、空格、`+`、`-`、`(`、`)`、`/`、`#`
- 若填写，必须至少包含一个数字

#### 需求描述 `demandDescription`

- 选填
- 去除首尾空格
- 最大长度建议：`1000` 字符
- 按纯文本处理，禁止 HTML 渲染

### 6.2 前台提交规则

- 接口允许匿名访问
- 提交成功后返回统一成功响应
- 前台不返回内部主键、版本号、IP 限流细节或后台状态值
- 表单提交不提供“立即成功分配”或“回访结果”类信息，避免暴露内部流程

### 6.3 防刷与限流规则

- 同一 IP 在 1 小时内最多允许提交 5 次
- 限流以真实客户端 IP 为准，必须从可信代理链路解析，不直接信任任意请求头
- 限流计数建议在 Redis 中实现，采用原子计数加 TTL 的固定窗口方案
- 计数对象建议覆盖进入提交接口的请求，避免恶意请求通过反复打空参数绕过限制
- 达到上限后返回统一的频控错误，不暴露具体阈值实现细节

### 6.4 后台列表规则

- 支持按提交时间范围筛选
- 支持分页查询
- 默认按 `提交时间 DESC, id DESC` 排序
- 后台列表默认展示脱敏结果
- 后台列表不返回逻辑删除数据
- 后台列表不缓存，保证敏感数据实时性和操作一致性

### 6.5 后台状态流转规则

- 线索状态仅允许在枚举值内流转
- 状态更新必须校验版本号，防止并发覆盖
- 状态更新成功后必须记录审计
- 重复提交同一状态视为幂等请求，不应产生异常

### 6.6 后台导出规则

- 导出仅限管理员
- 支持按筛选条件导出当前列表结果
- 支持按选中 ID 导出指定数据
- 筛选导出与选中导出必须二选一，避免语义冲突
- 导出结果使用 Excel 文件流输出
- 导出必须限制最大数据量，避免大文件拖垮服务
- 导出必须记录审计日志
- 筛选导出必须至少包含 1 个有效筛选条件，禁止空条件退化为全表导出
- 选中导出必须限制 `selectedIds` 数量上限，避免前端一次性提交过多 ID

### 6.7 敏感数据规则

线索数据属于敏感业务数据，处理原则如下：

- 列表默认脱敏显示电话和邮箱
- 列表可展示姓名、公司、状态、提交时间和需求描述摘要
- 单条详情仅管理员可查看完整信息
- 导出属于显式高风险操作，必须审计
- 日志中不得输出完整电话、完整邮箱和完整需求描述

### 6.8 权限规则

- 所有 `/admin/api/**` 接口必须登录
- 所有 Admin 接口必须校验 `Administrator` 角色
- Portal 提交接口允许匿名访问
- Portal 提交接口不返回后台敏感字段

## 7. 数据库设计

### 7.1 表名

`cms_lead_record`

### 7.2 表结构

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | BIGINT | PRIMARY KEY | - | 主键 ID |
| name | VARCHAR(64) | NOT NULL | - | 姓名 |
| company | VARCHAR(128) | NOT NULL | - | 公司 |
| email | VARCHAR(128) | NOT NULL | - | 邮箱 |
| phone | VARCHAR(64) | NULL | NULL | 电话 |
| demand_description | VARCHAR(1000) | NULL | NULL | 需求描述 |
| status | TINYINT | NOT NULL | 0 | 跟进状态 |
| submit_ip | VARCHAR(45) | NOT NULL | - | 提交 IP |
| user_agent | VARCHAR(255) | NULL | NULL | 提交 UA |
| status_updated_at | DATETIME | NULL | NULL | 状态更新时间 |
| status_updated_by | BIGINT | NULL | NULL | 最近一次状态更新人 |
| version | INT | NOT NULL | 0 | 乐观锁版本号 |
| created_at | DATETIME | NOT NULL | - | 提交时间 |
| updated_at | DATETIME | NOT NULL | - | 更新时间 |
| created_by | BIGINT | NOT NULL | - | 创建人 ID，匿名提交固定写系统占位值 |
| updated_by | BIGINT | NOT NULL | - | 更新人 ID，创建时写系统占位值，状态更新时写管理员 ID |
| deleted_marker | BIGINT | NOT NULL | 0 | 逻辑删除标记 |

### 7.3 索引设计

建议索引：

- `idx_cms_lead_record_del_submit_at(deleted_marker, created_at, id)`
- `idx_cms_lead_record_del_status_submit_at(deleted_marker, status, created_at, id)`
- `idx_cms_lead_record_submit_ip_created_at(submit_ip, created_at)`

说明：

- 后台列表主路径是按时间筛选和倒序分页
- 状态筛选是看板操作的高频条件
- IP 索引用于排障和后续反刷分析

### 7.4 单例与唯一性说明

本功能不依赖自然唯一键，不设置姓名、邮箱、电话组合唯一约束。

原因：

- 同一访客可能多次提交
- 不同企业可能共享相同邮箱或电话格式
- 唯一约束会误伤正常业务

## 8. API 接口契约设计

### 8.1 前台公开接口

#### 8.1.1 预约交流表单提交

### 创建预约交流线索

- **接口类型**：Portal
- **请求方法**：POST
- **接口路径**：`/portal/api/leads`
- **实现位置**：`src/main/java/com/company/officialwebsite/modules/lead/controller/PortalLeadController#createLead`
- **鉴权要求**：无需登录，可匿名访问
- **功能说明**：接收前台“预约交流”表单提交，并保存为一条线索记录
- **请求参数**：
  - `name`：string，必填，最长 64 字符
  - `company`：string，必填，最长 128 字符
  - `email`：string，必填，最长 128 字符，需满足邮箱格式
  - `phone`：string，选填，最长 64 字符，仅允许电话白名单字符
  - `demandDescription`：string，选填，最长 1000 字符
- **响应字段**：无业务数据，统一返回成功即可
- **错误码**：
  - `10001`：请求参数缺失、格式不合法或长度超限
  - `10005`：提交过于频繁
  - `70007`：同一 IP 触发线索提交限流
- **备注**：提交成功后写入线索记录，并记录 IP、UA 等基础追踪信息；接口必须接入 Redis 限流，超过阈值后直接拒绝；不返回内部状态与后台字段；匿名创建时 `created_by` 与 `updated_by` 固定写入系统占位值 `0L`

### 8.2 后台管理接口

#### 8.2.1 线索分页列表

### 获取线索分页列表

- **接口类型**：Admin
- **请求方法**：GET
- **接口路径**：`/admin/api/leads`
- **实现位置**：`src/main/java/com/company/officialwebsite/modules/lead/controller/AdminLeadController#getLeadPage`
- **鉴权要求**：需登录；需角色：管理员
- **功能说明**：按提交时间范围分页查询客户线索列表
- **请求参数**：
  - `pageNo`：int，选填，默认 `1`
  - `pageSize`：int，选填，默认 `20`，最大 `100`
  - `submitAtStart`：datetime，选填，提交时间起始值
  - `submitAtEnd`：datetime，选填，提交时间结束值
  - `status`：int，选填，线索状态过滤
- **响应字段**：
  - `list`：array，当前页数据，每项包含：
    - `id`：long，线索 ID
    - `name`：string，姓名
    - `company`：string，公司
    - `maskedEmail`：string，脱敏邮箱
    - `maskedPhone`：string，脱敏电话
    - `demandDescriptionPreview`：string，需求描述摘要
    - `status`：int，状态码
    - `statusLabel`：string，状态名称
    - `submittedAt`：datetime，提交时间
    - `updatedAt`：datetime，最近更新时间
  - `total`：long，总记录数
  - `pageNo`：int，当前页码
  - `pageSize`：int，页大小
- **错误码**：
  - `10001`：请求参数缺失、时间范围不合法或页大小超限
  - `20001`：未登录访问
  - `20002`：非管理员访问
- **备注**：默认按 `created_at DESC, id DESC` 排序；列表默认脱敏，不返回完整邮箱、完整电话和完整需求描述

#### 8.2.3 状态流转

### 更新线索状态

- **接口类型**：Admin
- **请求方法**：PUT
- **接口路径**：`/admin/api/leads/{id}/status`
- **实现位置**：`src/main/java/com/company/officialwebsite/modules/lead/controller/AdminLeadController#updateLeadStatus`

- **功能说明**：修改单条线索的跟进状态
- **请求参数**：
  - `version`：int，必填，当前版本号

  - `10001`：请求参数缺失、状态值不合法

  - `70004`：线索记录不存在或已删除
  - `70005`：线索状态不合法
- **备注**：状态更新必须写审计日志，并在事务提交后触发必要的后续动作

#### 8.2.4 数据导出

### 导出线索 Excel

- **接口类型**：Admin
- **请求方法**：POST
- **接口路径**：`/admin/api/leads/export`
- **实现位置**：`src/main/java/com/company/officialwebsite/modules/lead/controller/AdminLeadController#exportLeads`
- **鉴权要求**：需登录；需角色：管理员；必须携带有效 CSRF Token
- **功能说明**：将筛选结果或选中记录导出为 Excel 文件
- **请求参数**：
  - `exportMode`：string，必填，取值 `FILTERED` / `SELECTED`
  - `submitAtStart`：datetime，选填，筛选导出时生效
  - `submitAtEnd`：datetime，选填，筛选导出时生效
  - `status`：int，选填，筛选导出时生效
  - `selectedIds`：array[long]，选填，选中导出时生效
- **响应字段**：Excel 文件流，无 JSON 业务数据
- **错误码**：
  - `10001`：请求参数缺失、导出模式不合法或选中 ID 为空
  - `10005`：导出请求过于频繁
  - `20001`：未登录访问
  - `20002`：非管理员访问
  - `20005`：缺少 CSRF Token 或 Token 无效
  - `70004`：线索记录不存在或已删除
  - `70006`：导出数据量超过上限
- **备注**：筛选导出与选中导出互斥；导出必须限制最大行数，建议采用流式写出，避免一次性加载过多数据到内存；导出内容可包含完整联系方式与需求描述，但必须审计

## 9. 限流与反刷设计

### 9.1 Portal 提交限流

推荐方案：

- 以真实客户端 IP 为维度建立 Redis 计数器
- 使用固定窗口：1 小时 TTL
- 计数上限：5 次
- 计数实现建议使用原子自增，必要时通过 Lua 保证并发准确性

### 9.2 反刷辅助建议

为增强防刷能力，建议增加以下辅助措施：

- 记录提交 IP 和 UA
- 对异常高频请求写入安全日志
- 对明显异常的请求保留短期风控痕迹

说明：

- 本期不强制引入验证码
- 如果后续攻击流量升高，可在不破坏接口契约的前提下补充验证码或更细粒度风控

## 10. 安全审计与可观测性

### 10.1 审计范围

以下行为必须审计：

- 访客提交预约交流线索
- 管理员查看单条线索完整详情
- 管理员修改线索状态
- 管理员导出线索数据

### 10.2 审计字段

审计建议至少包含：

- 操作人 ID
- 操作人名称
- 操作 IP
- User-Agent
- 模块标识：`LEAD`
- 操作动作
- 目标 ID
- 操作结果
- 操作时间
- `traceId`

### 10.3 动作建议

建议动作名称如下：

- `CREATE_LEAD`
- `VIEW_LEAD_DETAIL`
- `UPDATE_LEAD_STATUS`
- `EXPORT_LEAD`

### 10.4 日志要求

- 业务日志不得打印完整邮箱、完整电话和完整需求描述
- 限流命中要输出 `warn`
- 系统异常要输出 `error`
- 关键日志必须带 `traceId`

## 11. 错误码设计

本期建议使用以下错误码：

| 错误码 | 标识符 | 默认提示 | 适用场景 |
| --- | --- | --- | --- |
| `10001` | `COMMON_PARAM_INVALID` | 请求参数不正确 | 表单字段缺失、格式错误、导出参数不合法 |
| `10003` | `COMMON_STATE_CONFLICT` | 当前状态不允许执行该操作 | 版本冲突、状态更新并发冲突 |
| `10005` | `COMMON_REQUEST_TOO_FREQUENT` | 请求过于频繁 | Portal 提交或导出触发频控 |
| `20001` | `AUTH_UNAUTHORIZED` | 请先登录 | Admin 未登录 |
| `20002` | `AUTH_FORBIDDEN` | 无权执行该操作 | 非管理员访问 Admin 接口 |
| `20005` | `AUTH_CSRF_INVALID` | 请求校验失败，请刷新页面后重试 | Admin 写接口 CSRF 校验失败 |
| `70004` | `LEAD_RECORD_NOT_FOUND` | 线索记录不存在或已删除 | 详情、状态更新、导出目标不存在 |
| `70005` | `LEAD_STATUS_INVALID` | 线索状态不合法 | 状态值不在枚举范围内 |
| `70006` | `LEAD_EXPORT_TOO_LARGE` | 导出数据量超过上限 | 导出行数或选中 ID 数超限 |
| `70007` | `LEAD_SUBMIT_RATE_LIMITED` | 同一 IP 提交过于频繁 | Portal 线索提交限流 |

说明：

- 如果实现阶段希望复用公共码，也可以仅保留 `10005`
- 但从模块可读性和接口契约稳定性看，建议保留 `70007`

## 12. 测试场景

### 12.1 Portal 提交

- 姓名、公司、邮箱必填校验

## 13. 推荐实现方案

### 13.1 总体方案

采用“独立线索表 + `lead` 模块 Service + Redis 限流 + Admin 侧审计 + 流式 Excel 导出”的实现方式。

### 13.2 关键实现点

- Portal 提交使用 Service 层统一做参数清洗、邮箱/电话校验和 Redis 计数
- Admin 列表使用分页查询，字段级脱敏由 VO 完成
- Admin 详情单独提供，避免列表直接暴露完整联系方式
- 状态更新使用乐观锁，防止后台并发覆盖
- 导出使用流式写出，避免大结果集内存膨胀
- 敏感数据不进入 Portal 缓存，后台也不做结果缓存

### 13.3 技术难点与解决方式

#### 难点 1：匿名提交的防刷

解决方式：

- 以真实客户端 IP 为粒度做 Redis 计数
- 采用原子自增和 TTL，保证并发条件下限流准确
- 记录异常频率日志，便于后续安全分析

#### 难点 2：敏感数据默认脱敏

解决方式：

- 列表 VO 只返回脱敏邮箱、脱敏电话和需求描述摘要
- 详情接口单独返回完整信息
- 导出属于显式高风险动作，写审计并限制频率

#### 难点 3：导出量与性能

解决方式：

- 导出前先做行数上限校验
- 使用流式 Excel 输出，不把全量数据一次性放入内存
- 对选中导出设置 ID 数量上限，避免前端误操作

#### 难点 4：状态并发更新

解决方式：

- 使用 `version` 乐观锁
- 状态更新前先查活跃记录
- 发生并发冲突时返回稳定错误码，引导前端刷新

## 14. 结论

本方案采用单表承载线索、匿名提交、后台分页看板、状态流转和 Excel 导出的实现模式，边界清晰、风险可控、便于后续扩展。

推荐优先落地的最小闭环为：

1. Portal 线索提交
2. Admin 线索分页与详情
3. Admin 状态流转
4. Admin 导出
5. Redis 限流与审计补齐

该方案能够在不引入 CRM、审批流和复杂跟进体系的前提下，满足当前官网“预约交流”线索收集与销售跟进的核心需求。
