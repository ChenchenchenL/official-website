# 功能点 21：全局资源上传服务详细设计说明书

## 1. 文档目标

本文档用于落实“功能点 21：全局资源上传服务”的后端详细设计，覆盖业务边界、核心对象、接口契约、文件安全校验、媒体生命周期、审计要求、数据库适配、测试场景与推荐实现方案，供产品、研发、测试和评审统一执行。

该功能属于“统一文件与媒体中心”模块，目标是为官网后台各业务模块提供统一、可复用、可审计的资源上传能力，避免各模块分散实现上传逻辑，降低安全风险和维护成本。

---

## 2. 设计范围

### 2.1 本期包含

本期包含以下能力：

- 后台管理员统一上传公开图片素材
- 后台管理员统一上传公开文档素材
- 服务端自动校验文件类型、大小、MIME 与内容签名
- 服务端生成安全存储路径并落盘
- 媒体元数据入库
- 返回统一上传结果
- 统一纳入媒体生命周期管理
- 上传行为审计留痕

### 2.2 本期不包含

以下内容不属于本功能点：

- Portal 前台匿名上传
- 视频上传
- 分片上传
- 断点续传
- 富文本内嵌资源批量上传协议
- 文档在线预览转换
- 图片压缩、缩略图、WebP 衍生文件的正式异步处理落地
- 媒体后台删除管理页面
- 私有附件、签名下载、权限化文档访问
- 对象存储接入

### 2.3 业务边界

本功能点只负责“统一上传”和“媒体元数据登记”，不负责具体业务对象的保存逻辑。

边界约束如下：

- 业务模块只上传并引用 `mediaId`，不得自行实现文件上传
- 业务模块保存成功后再维护媒体绑定关系
- 媒体模块不负责业务页面缓存失效，页面缓存由具体业务模块在事务提交后处理
- 本期上传对象限定为“官网公开素材”，不支持内部私有文档

---

## 3. 业务场景与目标

### 3.1 业务场景

管理员在后台维护站点 Logo、Banner、图标、产品资料、方案白皮书等内容时，需要先上传文件，再将返回的 `mediaId` 或路径写入具体业务模块。

典型场景包括：

- 上传站点 Logo、首页 Banner、图标、封面图
- 上传官网公开 PDF 资料
- 后续在产品、案例、关于我们等模块中引用该资源

### 3.2 功能目标

- 提供单一上传入口，统一上传规则
- 保证上传内容安全可控
- 统一记录媒体元数据，支撑后续绑定、解绑和清理
- 为后续扩展对象存储、异步处理、媒体管理页保留清晰底座

## 4. 核心对象设计

### 4.1 核心业务对象

对象名称：`MediaAsset`

业务含义：一条上传后的媒体资源元数据记录。

### 4.2 对象定位

`MediaAsset` 是统一媒体中心的基础资源对象，不依附于单一业务模块。

设计特点：

- 上传后先独立存在
- 初始状态为 `TEMPORARY`
- 被业务内容引用后可转为 `BOUND`
- 失去全部有效引用后可转为 `UNBOUND`
- 后续可被清理任务回收

### 4.3 关联对象

对象名称：`MediaReference`

业务含义：记录业务对象与媒体资源之间的绑定关系。

说明：

- 具体业务对象不直接维护媒体生命周期状态
- 绑定关系由媒体模块统一维护
- 生命周期状态以引用关系是否存在为准

### 4.4 核心枚举

#### 媒体类型 `mediaType`

建议使用枚举值：

- `IMAGE`
- `DOCUMENT`

#### 媒体状态 `status`

建议使用枚举值：

- `TEMPORARY`
- `BOUND`
- `UNBOUND`
- `DELETED`

## 6. 当前基线与改造范围

### 6.1 当前已存在能力

仓库当前已具备以下基础能力：

- 后台图片上传接口：`POST /admin/api/media/assets`
- 图片扩展名、MIME 与签名校验
- 本地文件落盘
- 媒体元数据入库
- 基础媒体绑定/解绑能力

### 6.2 当前缺失能力

当前实现尚未完整覆盖本 PRD 的目标，至少缺少以下能力：

- 文档上传支持
- 文档签名校验
- 上传审计日志
- 更完整的返回字段
- 对“公开 URL”和“绝对 URL”的统一契约
- 更细粒度的媒体错误码

### 6.3 本期改造原则

- 优先复用现有 `media` 模块，不新开平行上传模块
- 优先在现有接口上扩展“统一上传”能力
- 严格控制文档类型范围，降低安全和兼容风险

## 7. 文件类型与安全规则

### 7.1 图片类型

本期支持以下图片格式：

- `png`
- `jpg`
- `jpeg`
- `webp`

### 7.2 文档类型

为控制首期实现复杂度与安全风险，本期仅支持：

- `pdf`

说明：

- `doc`、`xls`、`ppt` 等旧版 OLE2 文档格式暂不纳入首期
- `docx`、`xlsx`、`pptx` 等基于 ZIP 容器的格式可作为后续迭代项
- 若后续扩展文档类型，必须先补充签名校验策略和测试用例

### 7.3 明确禁止类型

必须拒绝以下文件类型：

- `html`
- `js`
- `jsp`
- `php`
- `exe`
- `sh`
- `bat`
- `jar`
- `class`
- `svg`
- 其他脚本型、可执行型或可注入型文件

### 7.4 大小限制

建议默认限制如下：

- 图片：单文件最大 `2MB`
- 文档：单文件最大 `10MB`

大小限制必须支持配置化，不应硬编码在业务逻辑中。

### 7.5 双重与三重校验原则

上传文件不得只依赖扩展名判断合法性，必须同时校验：

- 扩展名白名单
- MIME 白名单
- 文件内容签名或文件头

### 7.6 图片校验规则

图片上传至少校验：

- 文件非空
- 扩展名合法
- `Content-Type` 以 `image/` 开头
- 对应签名头合法

### 7.7 PDF 校验规则

PDF 上传至少校验：

- 文件非空
- 扩展名为 `pdf`
- `Content-Type` 为 `application/pdf` 或兼容白名单值
- 文件头包含 PDF 标识签名

### 7.8 安全存储规则

- 最终存储文件名必须由后端生成
- 不允许使用原始文件名作为最终磁盘文件名
- 不允许使用用户传入目录
- 不允许存储到应用可执行目录、模板目录或静态源码目录

## 8. 返回契约设计

### 8.1 字段定义

统一上传返回对象建议包含：

- `mediaId`：媒体主键 ID
- `mediaType`：媒体类型，`IMAGE` / `DOCUMENT`
- `originalFilename`：原始文件名
- `contentType`：文件 MIME 类型
- `size`：文件大小，单位字节
- `path`：相对存储路径，例如 `2026/06/28/abcd1234.pdf`
- `url`：公开访问相对 URL，例如 `/media/public/2026/06/28/abcd1234.pdf`
- `absoluteUrl`：绝对访问 URL；仅在系统配置了公开站点域名时返回

### 8.2 路径与 URL 语义约束

必须明确区分以下概念：

- `path`：服务端内部相对存储路径，不带公开前缀
- `url`：对外公开访问路径，可供前端直接拼接当前站点域名访问
- `absoluteUrl`：完整带域名的访问地址

### 8.3 当前实现兼容说明

当前仓库已存在的返回 VO 仅包含：

- `mediaId`
- `url`
- `contentType`
- `size`

本期若实施该功能，应同步扩展 VO 与接口文档，避免前后端字段预期不一致。

## 9. 生命周期与状态流转

### 9.1 初始状态

上传成功后，媒体状态为：

- `TEMPORARY`

### 9.2 绑定规则

业务模块保存成功后，通过媒体绑定关系将资源状态更新为：

- `BOUND`

### 9.3 解绑规则

业务内容更新、替换或逻辑删除后，如果该媒体不再被任何活跃业务对象引用，则状态更新为：

- `UNBOUND`

### 9.4 删除规则

本功能点不提供后台媒体删除 API，但数据模型和状态枚举必须保留：

- `DELETED`

以兼容后续媒体管理与清理能力。

### 9.5 清理边界

本期不要求落地完整清理任务，但必须在设计中预留规则：

- 超过一定时限且未绑定的 `TEMPORARY` 资源可被回收
- 超过一定时限且无引用的 `UNBOUND` 资源可被回收

建议默认时限：

- `TEMPORARY`：24 小时后可回收
- `UNBOUND`：7 天后可回收

以上时限建议配置化，后续由定时任务执行。

## 10. 权限、安全与审计规则

### 10.1 权限规则

- 所有 `/admin/api/**` 接口必须登录
- 上传接口必须校验 `Administrator` 角色
- Portal 不提供统一上传接口

### 10.2 CSRF 规则

若后台采用 Cookie Session 模式，则所有上传接口作为 Admin 写接口，必须通过 CSRF 校验。

接口文档中必须注明该要求，避免实现与安全规则不一致。

### 10.3 审计规则

上传行为属于关键后台操作，必须写入 `sys_audit_log`。

建议审计字段包括：

- 操作人 ID
- 操作人名称
- IP
- User-Agent
- 模块：`MEDIA`
- 动作：`UPLOAD_MEDIA`
- 目标对象 ID：`mediaId`
- 结果：成功 / 失败
- 变更快照：原始文件名、媒体类型、大小、返回路径

### 10.4 日志脱敏规则

日志中不得打印：

- 本地绝对磁盘路径
- 配置密钥
- 敏感请求头
- 未处理的完整异常上下文中的敏感配置

## 12. API 契约设计

### 12.1 接口定义

#### 上传统一公开资源

- 接口类型：Admin
- 请求方法：`POST`
- 接口路径：`/admin/api/media/assets`
- 鉴权要求：需登录；需角色 `Administrator`；必须通过有效 CSRF 校验

### 12.2 请求格式

`multipart/form-data`

请求参数：

- `file`：必填，单文件上传

### 12.3 响应示例

```json
{
  "code": 0,
  "message": "操作成功",
  "data": {
    "mediaId": 1001,
    "mediaType": "IMAGE",
    "originalFilename": "brand-logo.png",
    "contentType": "image/png",
    "size": 182344,
    "path": "2026/06/28/8f3a1d2c9b4e.png",
    "url": "/media/public/2026/06/28/8f3a1d2c9b4e.png",
    "absoluteUrl": "https://www.example.com/media/public/2026/06/28/8f3a1d2c9b4e.png"
  }
}
```

### 12.4 PDF 响应示例

```json
{
  "code": 0,
  "message": "操作成功",
  "data": {
    "mediaId": 1002,
    "mediaType": "DOCUMENT",
    "originalFilename": "solution-whitepaper.pdf",
    "contentType": "application/pdf",
    "size": 2048576,
    "path": "2026/06/28/7a8c9e1f2d3b.pdf",
    "url": "/media/public/2026/06/28/7a8c9e1f2d3b.pdf",
    "absoluteUrl": "https://www.example.com/media/public/2026/06/28/7a8c9e1f2d3b.pdf"
  }
}
```

### 12.5 错误码建议

本期建议至少支持以下错误码语义：

- `60001 MEDIA_FILE_INVALID`：上传文件不符合要求
- `60002 MEDIA_FILE_SIZE_EXCEEDED`：文件大小超出限制
- `60003 MEDIA_FILE_TYPE_UNSUPPORTED`：文件类型不支持
- `60004 MEDIA_FILE_SIGNATURE_INVALID`：文件内容签名不合法
- `60005 MEDIA_UPLOAD_FAILED`：文件上传失败
- `60006 MEDIA_STORAGE_WRITE_FAILED`：文件存储失败

若当前项目阶段暂不细分，也可先兼容使用 `60001`，但接口文档中应标注后续会细化。

## 13. 技术拆分建议

### 13.1 Controller 层

`AdminMediaAssetController`

职责：

- 接收上传请求
- 做基础参数适配
- 调用 Service
- 返回统一响应

### 13.2 Service 层

`MediaAssetService`
`MediaAssetServiceImpl`

职责：

- 识别上传媒体类型
- 分发图片或 PDF 校验逻辑
- 调用存储服务落盘
- 写入媒体元数据
- 写入审计日志
- 组装统一返回 VO

建议将当前仅支持图片的上传方法演进为统一入口，例如：

- `upload(MultipartFile file)`

### 13.3 Validator 支撑层

建议抽出独立校验支撑类，例如：

- `MediaValidationSupport`
- `ImageSignatureValidator`
- `PdfSignatureValidator`

理由：

- 避免单一 Service 方法过长
- 方便后续扩展文档类型
- 便于补充单元测试

### 13.4 Storage 层

`LocalMediaStorageService`

职责：

- 生成安全路径
- 文件落盘
- 写入失败回滚时删除孤儿文件

建议将当前偏“图片语义”的方法抽象为更通用的方法，避免命名限制后续演进。

### 13.5 绑定能力

继续复用现有：

- `bindMedia(Long mediaId, String bizModule, Long bizObjectId, String bizField)`

由具体业务模块在保存成功后调用，不在上传接口中完成业务绑定。

## 14. 难点与解决方案

### 14.1 难点一：不能只依赖扩展名判断文件安全

问题：

- 用户可能将恶意文件伪装为图片或 PDF 上传

解决方案：

- 同时校验扩展名、MIME 与文件头签名

### 14.2 难点二：文档格式兼容与安全识别复杂

问题：

- Office 文档格式兼容性高、签名识别复杂，首期贸然纳入会显著增加实现和测试成本

解决方案：

- 首期将文档类型收敛为 `pdf`
- 其他格式作为后续迭代项，待规则和测试补齐后再开放

### 14.3 难点三：上传成功但数据库写入失败会产生孤儿文件

问题：

- 文件落盘成功但元数据插入失败时，磁盘可能残留无主文件

解决方案：

- 先落盘，后入库
- 入库异常时立即删除已落盘文件
- 后续再通过清理任务做兜底回收

### 14.4 难点四：上传与业务绑定不是同一事务

问题：

- 用户可能上传完文件后不继续保存业务表单

解决方案：

- 上传后资源状态为 `TEMPORARY`
- 仅在业务保存成功后再绑定
- 后续由清理任务回收长期未绑定资源

### 14.5 难点五：公开文档与私有附件边界容易混淆

问题：

- 若未明确范围，后续可能误将内部资料也通过公开媒体中心上传

解决方案：

- 在 PRD、接口文档和实现层统一声明：本期仅支持官网公开素材
- 私有附件能力需独立建模，不复用本接口直接暴露

## 15. 测试与验收标准

### 15.1 上传成功场景

- 上传合法 `png/jpg/jpeg/webp` 文件，接口成功返回统一结果
- 上传合法 `pdf` 文件，接口成功返回统一结果

### 15.2 非法类型拦截

- 上传 `html/js/exe/sh/svg` 等文件必须失败

### 15.3 伪装文件拦截

- 扩展名为图片或 PDF，但实际内容签名不匹配，必须失败

### 15.4 超限文件拦截

- 文件超过对应大小上限时必须失败

### 15.5 权限校验

- 未登录不得调用
- 非管理员不得调用
- 缺少有效 CSRF Token 的 Admin 写请求必须失败

### 15.6 返回字段校验

- 返回中必须包含约定字段
- `path`、`url`、`absoluteUrl` 语义必须清晰一致

### 15.7 数据一致性校验

- 数据库写入失败时，磁盘上的落盘文件应被删除

### 15.8 生命周期校验

- 上传后状态应为 `TEMPORARY`
- 业务绑定后应转为 `BOUND`
- 全部解绑后应转为 `UNBOUND`

### 15.9 审计校验

- 上传成功应写入审计日志
- 上传失败也应有可追踪日志

## 16. 推荐实现方案

推荐采用“后台统一上传入口 + 本地存储 + 统一媒体元数据表 + 生命周期状态机”的实现方案。

### 16.1 方案说明

- 继续使用 `media` 模块作为统一上传中心
- 保留单一接口：`POST /admin/api/media/assets`
- 首期支持：
  - 图片：`png/jpg/jpeg/webp`
  - 文档：`pdf`
- 上传成功后统一写入 `media_asset`
- 返回标准化 `MediaUploadVO`
- 业务模块后续通过 `mediaId` 建立引用

### 16.2 方案理由

- 与当前仓库已有实现最兼容，改造成本最低
- 风险面清晰，不会因文档格式过宽导致安全与兼容问题失控
- 满足统一上传、安全校验、生命周期与后续扩展要求
- 便于未来平滑扩展到对象存储、异步处理和媒体管理页
