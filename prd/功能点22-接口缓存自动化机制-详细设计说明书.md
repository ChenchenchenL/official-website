# 功能点 22：接口缓存自动化机制详细设计说明书

## 1. 文档目标

本文档用于落实“功能点 22：接口缓存自动化机制”的后端详细设计，覆盖业务边界、缓存对象、关键规则、技术拆解、异常降级、测试验收与推荐实现方案，供产品、研发、测试和评审统一执行。

该功能属于官网后端的跨模块基础设施能力，不归属于单一业务模块，而是统一服务于 `site`、`product`、`casecenter`、`news`、`lead` 等所有面向 Portal 的公开查询能力。

---

## 2. 业务背景与目标

### 2.1 业务背景

官网前台以“高频读取、低频修改”为主，且生产环境默认按多实例部署设计。如果所有 Portal 查询都直接命中数据库，会带来如下问题：

- 公开页面访问量上升时数据库压力持续增大
- 同一份公开内容被重复读取，浪费后端资源
- 页面响应时延波动大，不利于官网稳定展示
- 后台修改内容后，前台无法稳定做到秒级更新

因此，需要建立统一的 Portal 缓存自动化机制。

### 2.2 功能目标

本功能点目标如下：

- Portal 公开只读查询接口默认优先读取 Redis 缓存
- Admin 任意 Create、Update、Delete、显示/隐藏、排序等写操作成功后，自动清除对应 Portal 缓存
- 缓存删除必须在数据库事务提交后执行
- 所有 Portal 缓存必须设置 TTL，不允许永久缓存
- 所有 Portal 缓存失效均采用“立即删除 + 延迟二次删除”策略
- Redis 异常时不能阻塞主业务，必须具备明确降级能力
- 为多模块缓存 Key 命名、失效映射和聚合接口联动提供统一底座

---

## 3. 设计范围

### 3.1 本期包含

本期包含以下能力：

- Portal 公开只读查询接口的 Redis 读缓存
- Portal 查询结果的 TTL 管理
- Admin 写操作成功后的自动缓存失效
- 事务提交后失效机制
- 延迟二次删除兜底机制
- 空结果缓存策略
- 模块级缓存 Key 命名规范
- 聚合接口缓存依赖失效规则
- 缓存异常日志、指标与降级约束

### 3.2 本期不包含

以下内容不属于本期：

- Admin 后台列表缓存
- 登录态、权限态、验证码等安全缓存
- Redis 集群架构设计和运维治理方案
- CDN 缓存刷新
- MQ 广播式缓存通知
- 手工缓存刷新页面
- 搜索索引重建
- 文件上传链路缓存

### 3.3 缓存适用边界

本机制仅适用于：

- `Portal` 匿名可访问的只读 `GET` 查询接口
- 公开列表接口
- 公开详情接口
- 公开单例配置接口
- 公开聚合配置接口

以下接口默认不纳入本机制：

- Portal 表单提交接口
- Portal 上传、下载、回调接口
- Admin 任意查询接口
- 敏感数据查询接口
- 登录、鉴权、权限类接口
- 限流计数、验证码等安全控制接口

说明：

- “门户端所有查询接口默认读取 Redis 缓存”在实现层面应解释为“所有 Portal 公开只读查询接口默认纳入 Redis 缓存机制”
- 不能机械扩展到所有 HTTP 查询动作，否则会误伤敏感数据和高实时性场景

## 4. 核心对象设计

本功能点是基础设施能力，不新增业务实体表，核心对象为缓存定义、缓存键和缓存失效请求。

### 4.1 核心对象一：PortalCacheKey

业务含义：Portal 查询缓存的唯一标识。

统一命名规则：

```text
official:portal:{module}:{identifier}
```

示例：

```text
official:portal:site-config
official:portal:navigation
official:portal:timeline-events
official:portal:value-cards
official:portal:contact-info
official:portal:cooperation-direction-tags
official:portal:products:list
official:portal:product:detail:1001
official:portal:about:page
```

说明：

- `official` 为系统统一前缀
- `portal` 为公开缓存命名空间
- `module` 为模块段
- `identifier` 为列表、详情、聚合页或特定子场景段

### 4.2 核心对象二：PortalCacheDefinition

业务含义：某个 Portal 接口对应的一组缓存配置定义。

建议至少包含以下属性：

- `module`
- `segments`
- `ttl`
- `cacheEmptyResult`
- `emptyResultTtl`
- `description`

### 4.3 核心对象三：PortalCacheInvalidationRequest

业务含义：某次后台写操作成功后需要清除的一组 Portal 缓存。

建议至少包含以下属性：

- `keys`
- `sourceModule`
- `sourceAction`
- `traceId`

### 4.4 核心对象四：PortalCacheModuleMapping

业务含义：定义“某个后台写动作影响哪些 Portal Key”。

示例：

- 更新时间轴节点 -> 清除 `official:portal:timeline-events`
- 更新联系方式 -> 清除 `official:portal:contact-info`
- 更新合作方向标签 -> 清除 `official:portal:cooperation-direction-tags`
- 更新产品信息 -> 清除产品列表 Key 与对应详情 Key
- 更新首页聚合数据子模块 -> 清除子模块 Key 与首页聚合 Key

## 5. 包结构与模块归属

该功能归属于 `infrastructure.cache` 基础设施层，并由各业务模块 Service 接入。

### 5.1 基础设施层

当前仓库已存在以下基础类，应继续复用：

- `com.company.officialwebsite.infrastructure.cache.PortalCacheKeyBuilder`
- `com.company.officialwebsite.infrastructure.cache.PortalCacheInvalidationSupport`

如需增强，建议新增：

- `PortalCacheDefinition`
- `PortalCacheTtlResolver`
- `PortalCacheNames` 或 `PortalCacheModuleRegistry`

### 5.2 业务模块接入位置

缓存读取和缓存失效只能在 Service 或 Application Service 层接入，禁止 Controller 直接操作 Redis。

适用模块包括：

- `modules.site`
- `modules.product`
- `modules.casecenter`
- `modules.news`
- `modules.lead`
- `application.portal`

### 5.3 配置归属

建议配置归属于：

- `com.company.officialwebsite.common.config.properties.OfficialProperties`

建议配置项包括：

- `official.cache.key-prefix`
- `official.cache.default-ttl`
- `official.cache.empty-result-ttl`
- `official.cache.second-delete-delay`
- `official.cache.portal-ttl-overrides.*`

## 6. 核心业务规则

### 6.1 Portal 查询缓存规则

- Portal 公开只读查询接口默认先查 Redis
- 缓存命中时直接返回，不再访问数据库
- 缓存未命中时查询数据库并写入缓存
- Redis 读取失败时降级数据库查询，不直接向前台报错
- Redis 写入失败时仅记录日志，不影响接口成功返回
- Portal 查询响应结构不得因缓存接入而变化

### 6.2 Admin 写操作缓存失效规则

以下操作均应触发缓存失效：

- Create
- Update
- Delete
- 显示/隐藏切换
- 排序调整
- 批量内容变更
- 任意影响 Portal 输出结果的数据写操作

规则要求如下：

- 失效动作必须在数据库事务成功提交后执行
- 回滚场景不得提前删缓存
- 一次写操作若影响多个 Portal Key，必须全部清除
- 缓存删除失败不得回滚业务事务，但必须记录错误日志并保留 TTL 兜底

### 6.3 延迟二次删除规则

为降低并发下“读旧值再回填缓存”的风险，所有 Portal 缓存失效都必须采用双删机制。

规则如下：

- 第一次删除：事务提交后立即执行
- 第二次删除：在固定延迟后再次执行
- 默认延迟建议值：`3s`
- 延迟值必须支持配置化

### 6.4 TTL 规则

- 所有 Portal 缓存必须设置 TTL
- 默认 TTL 建议为 `1h`
- 不允许永久缓存
- 可按模块或接口细分覆盖 TTL
- 空结果缓存 TTL 必须短于正常结果 TTL

### 6.5 空结果缓存规则

为防止缓存穿透，可对部分公开查询启用空结果缓存。

规则如下：

- 仅适用于允许返回“空列表”或“空单例”语义的公开接口
- 空结果 TTL 建议 `5min`
- 是否允许缓存空结果必须由模块显式声明
- 敏感接口、强实时接口和易误导接口不应缓存空结果

### 6.6 缓存粒度规则

缓存粒度必须明确，不能只停留在“模块级”概念。

建议按以下粒度分别定义：

- 列表缓存
- 详情缓存
- 单例配置缓存
- 聚合页缓存

说明：

- 产品列表和产品详情必须用不同 Key
- 页面聚合接口与其子模块接口必须用不同 Key
- 不允许把全部缓存粗暴收敛成一个模块总 Key，避免未来失效过大或遗漏细粒度刷新

### 6.7 Key 管理规则

- 不允许业务代码手写完整 Redis Key
- 必须统一通过 `PortalCacheKeyBuilder` 构造 Key
- 不允许在生产逻辑中使用 `KEYS official:portal:*` 进行模糊删除
- 必须显式列出需要删除的 Key

### 6.8 失效责任归属规则

- 缓存失效责任归属于触发写操作的业务 Service
- 基础设施层只负责“执行删除”，不负责“猜测哪些 Key 应删除”
- 聚合接口依赖关系必须由业务模块或 application 层显式维护

### 6.9 多实例一致性规则

- 生产环境必须使用 Redis 或等价共享缓存
- 不允许将生产 Portal 主缓存建立在单机 JVM 本地缓存上
- 任一实例执行后台写操作后，其他实例必须共享同一份失效结果

### 6.10 敏感性边界规则

以下数据默认不纳入 Portal 缓存自动化机制：

- 后台敏感查询数据
- 线索后台看板数据
- 登录态、权限态、用户态
- 上传状态
- 限流与防刷计数器

## 7. 读取与写入标准链路

### 7.1 Portal 查询标准链路

推荐执行链路如下：

1. 根据模块常量构建缓存 Key
2. 从 Redis 读取缓存
3. 若命中则反序列化并返回
4. 若未命中则查询数据库
5. 组装 Portal VO
6. 写入 Redis 并设置 TTL
7. 返回结果

### 7.2 Admin 写操作标准链路

推荐执行链路如下：

1. Service 内完成参数校验与业务校验
2. 执行数据库写入
3. 写入审计日志
4. 组装需失效的 Portal Key 列表
5. 在事务中注册 after-commit 失效动作
6. 事务提交后立即删除缓存
7. 调度延迟二次删除

### 7.3 聚合接口链路要求

如果 Portal 存在页面级聚合接口，例如首页聚合、关于我们聚合、联系我们聚合，则必须遵循：

- 聚合接口缓存独立成 Key
- 子模块更新时除清除子模块 Key 外，还要同步清除聚合 Key
- 聚合接口不能依赖隐式联动或人工约定

## 8. 现有底座与推荐技术拆分

### 8.1 当前已具备底座

仓库当前已存在以下可复用能力：

- `PortalCacheKeyBuilder`：统一构造 Portal Redis Key
- `PortalCacheInvalidationSupport`：统一承接 after-commit 删除与延迟二次删除
- 多个模块 Service 已采用 RedisTemplate 实现 Portal read-through cache

### 8.2 本期推荐拆分

建议按以下层次推进：

#### 基础设施层

- 继续复用 `PortalCacheKeyBuilder`
- 继续复用 `PortalCacheInvalidationSupport`
- 如需要，补齐 TTL 解析与缓存定义类

#### 业务模块层

- 每个模块定义自己的 `CACHE_SEGMENT`
- 每个 Portal 查询 Service 显式实现 read-through cache
- 每个 Admin 写 Service 显式声明需要失效的 Portal Key 集合

#### 应用层

- 聚合 Portal 接口在 `application.portal` 维护自身缓存
- 聚合依赖关系在 application 层统一编排

### 8.3 推荐常量组织方式

建议每个模块维护独立缓存常量，例如：

- `SitePortalCacheConstants`
- `ProductPortalCacheConstants`
- `LeadPortalCacheConstants`
- `AboutPortalCacheConstants`

建议常量至少包含：

- `CACHE_SEGMENT`
- `LIST_SEGMENT`
- `DETAIL_SEGMENT`
- `AGGREGATE_SEGMENT`
- 依赖联动涉及的附加 Key 描述

## 9. 序列化与兼容性规则

### 9.1 序列化规则

- Portal 缓存值统一采用项目统一的 Redis 序列化方案
- 缓存结构必须与 Portal VO 语义一致
- 不允许直接缓存 Entity 作为 Portal 对外结构

### 9.2 反序列化异常处理

- 反序列化失败时应视为缓存损坏
- 应删除坏缓存并重新回源数据库
- 不允许因为坏缓存导致 Portal 接口直接返回 500

### 9.3 结构兼容性规则

当 Portal VO 结构发生破坏性变化时，建议采用以下任一方案：

- 升级缓存 key segment 版本号
- 部署时清理历史相关缓存

说明：

- 不能默认假设旧缓存一定与新 VO 兼容
- 该规则尤其适用于聚合接口和复杂详情接口

## 10. 配置设计

建议配置如下：

```yaml
official:
  cache:
    key-prefix: official
    default-ttl: PT1H
    empty-result-ttl: PT5M
    second-delete-delay: PT3S
```

如需按模块定制 TTL，可扩展：

```yaml
official:
  cache:
    portal-ttl-overrides:
      timeline-events: PT1H
      contact-info: PT30M
      home: PT10M
```

约束如下：

- 所有 TTL 必须配置化
- 配置缺省时走默认值
- 不允许硬编码在业务 Service 中

## 11. 异常、超时与降级策略

### 11.1 Portal 查询异常策略

- Redis 不可用时降级查数据库
- 数据库成功则接口仍正常返回
- 前台不感知缓存层异常细节

### 11.2 Admin 写操作异常策略

- 数据库事务失败时不得删除缓存
- 缓存删除失败时不回滚已提交业务数据
- 由延迟二次删除与 TTL 提供兜底

### 11.3 Redis 超时策略

- Redis 访问超时时间必须较短
- 不允许缓存层阻塞主请求过久
- 读写超时后应尽快降级，不应拖垮 Portal 主链路

### 11.4 反序列化异常策略

- 视为缓存失效
- 清理坏缓存
- 重新回源数据库

### 11.5 异常日志规则

- Redis 读取失败记录 `warn`
- Redis 写入失败记录 `warn`
- 立即删除失败记录 `error`
- 延迟删除调度失败记录 `error`

日志中应包含：

- `traceId`
- cache key
- module
- action
- result

日志中不得输出：

- 敏感数据明文
- 完整业务实体快照

## 12. 监控与可观测性要求

本功能点应具备最基本的缓存观测能力，便于排障和容量评估。

建议至少统计：

- 缓存命中次数
- 缓存未命中次数
- 数据库回源次数
- 缓存写入失败次数
- 立即删除失败次数
- 延迟二次删除失败次数
- 反序列化失败次数

说明：

- 本期不强制要求接入完整监控平台，但 PRD 层面应明确需要保留扩展点
- 关键异常至少要有日志和可检索依据

## 13. 难点分析与解决方案

### 13.1 难点一：事务与缓存时序一致性

问题：

- 若在事务提交前清理缓存，事务回滚后会导致前台重新回源并读到错误状态

解决方案：

- 强制使用 `afterCommit` 机制执行缓存失效
- 禁止 Controller、Mapper 或事务中间流程直接删除 Redis

### 13.2 难点二：并发读旧值回填

问题：

- 第一次删除后，其他线程可能在更新传播窗口内读到旧数据并重新写回缓存

解决方案：

- 采用“立即删除 + 延迟二次删除”兜底
- 对未来极热点接口可扩展短锁或单飞重建机制，但本期不强制引入复杂方案

### 13.3 难点三：多模块依赖导致漏删

问题：

- 一次后台写操作可能影响多个 Portal 接口，若依赖人工记忆容易漏删

解决方案：

- 显式建立模块失效映射关系
- 聚合接口依赖关系由 application 层或常量注册表统一维护
- 测试中增加“一次写操作触发多个 Key 失效”场景

### 13.4 难点四：Redis 异常影响主流程

问题：

- Redis 宕机、超时或序列化异常不能把官网主流程一起拖垮

解决方案：

- 所有缓存异常都要支持降级
- 读失败降级查库
- 写失败只记日志，不影响主业务返回
- 删除失败不回滚业务事务

### 13.5 难点五：缓存结构演进兼容

问题：

- Portal VO 升级后，旧缓存结构可能无法兼容新代码

解决方案：

- 对破坏性变更使用缓存 key 版本化或部署期清理策略
- 将“缓存结构兼容性”纳入接口升级评审项

## 14. 数据库与持久化影响

本功能点本身不新增数据库表结构。

但要求如下：

- 所有影响 Portal 内容的写操作都必须明确事务边界
- 缓存失效动作必须建立在事务提交后
- 审计仍复用既有审计体系
- 若未来新增“手工缓存刷新”能力，应纳入审计范围

---

## 15. API 契约影响

### 15.1 对外 API 变化

本功能点原则上不新增外部业务 API，属于基础设施增强。

对 Portal 接口的影响：

- 响应结构不变
- 查询性能增强
- 后台写操作后前台数据秒级更新

对 Admin 接口的影响：

- 请求与响应结构不变
- 写操作成功后自动触发对应 Portal 缓存失效

### 15.2 接口文档要求

后续在 `docs/接口文档.md` 中，建议对每个 Portal 接口补充：

- 是否使用 Redis 缓存
- 缓存 Key 规则
- 默认 TTL
- 是否支持空结果缓存
- 哪些 Admin 写操作会触发该接口缓存失效

## 16. 测试与验收标准

必须覆盖以下场景：

1. Portal 查询命中缓存时，不访问数据库且返回结果正确。
2. Portal 查询未命中缓存时，查库成功并写入 Redis。
3. Admin 写操作成功提交后，对应 Portal Key 立即失效。
4. Admin 写操作事务回滚时，缓存不得被删除。
5. 延迟二次删除应被正确调度。
6. 一次后台写操作影响多个 Portal Key 时，应全部失效。
7. Redis 读取失败时，Portal 查询应降级成功。
8. Redis 写入失败时，Portal 查询仍应成功返回。
9. 反序列化异常时，应清理坏缓存并重新回源。
10. 空结果缓存命中与过期行为应符合预期。
11. 聚合接口依赖的子模块更新后，聚合缓存也应被清除。
12. 所有 Portal 缓存 Key 都应符合统一命名规则。
13. TTL 与延迟二次删除配置应可生效。
14. 多实例共享 Redis 场景下，一个实例写入后，其他实例读取应获得新数据。

## 17. 推荐实现方案

推荐采用：

**“统一 Portal Key Builder + 统一失效支持组件 + 各模块 Service 显式接入 + Redis read-through cache + 提交后删除 + 延迟二次删除 + 聚合依赖显式映射”** 的实现方案。

### 17.1 推荐理由

- 与当前仓库已存在的 `PortalCacheKeyBuilder`、`PortalCacheInvalidationSupport` 保持一致
- 不引入额外中间件，改造成本低
- 技术路径清晰，适合多模块平滑接入
- 能满足当前官网“高频读、低频写、秒级更新”的核心目标
- 具备较好的可测试性、可维护性和扩展性

### 17.2 当前阶段不推荐方案

当前不建议采用以下做法：

- Controller 直接读写 Redis
- 业务代码硬编码完整缓存 Key
- 使用 `KEYS` 做生产模糊删除
- 事务未提交就删缓存
- 以单机 JVM 本地缓存作为生产主缓存
- 过早引入 MQ 广播、复杂一致性框架或专用缓存中台
